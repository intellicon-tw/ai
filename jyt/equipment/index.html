共 3 款 Agent POC部門 設備工程／維修部

設備工程／維修部是這次盤點裡資料含金量最高、也最值得優先做的部門。MTTR 偏高、排障高度依賴幾位資深師傅，正是 AI 人機協作最能立刻見效的場景；只要把故障碼對照、歷史維修紀錄與停機統計整理出來，三隻 Agent 都能很快落地。目前比較缺的是結構化的維修紀錄與停機分類碼，建議優先把這兩塊資料補齊。以下每一頁對應一個需求，點進去可看到：需求與痛點 → 開發思路 → 完整提示詞 → 知識庫設計 → 建置流程 → 驗測情境。

#1.1

設備故障排除知識問答

知識問答型（RAG 文件查詢）

產線設備一旦跳警報停機，年輕技師多半看不懂故障碼，得打電話找兩三位資深師傅，師傅不在或在大陸廠時，動輒等上一兩小時才開始排查。同樣的故障每個月重複發生，但排障經驗只在師傅腦袋裡，沒有累積下來，整體 MTTR 一直壓不下去。

 #1.2

維修工單自動整理

標準化作業型（表單生成）

技師排完故障多半口頭交代或潦草手寫，工單常缺故障碼、根因或停機時數，回去補登又拖一兩天，導致歷史資料殘缺、無法分析。每張工單欄位格式各寫各的，後續要統計停機原因得人工再清洗，每月光整理工單就耗掉設備工程師不少時間。

 #1.3

設備稼動停機數據解讀

數據解讀型（Text-to-SQL）

每月稼動率與停機統計散落在各機台的 Excel，設備課長要人工樞紐分析才能看出哪台機、哪類停機最吃時間，往往月底才看到問題、已經來不及改善。停機分類沒有統一拆解，主管想問『這個月哪台貼合機停最久、主要卡在哪』常常得等工程師整理半天才有答案。
