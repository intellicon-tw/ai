① 需求與痛點

生管在正式排程前，每次都要人工把 FCST 需求、庫存、在途採購、請購與在制工單攤開比對，逐一判斷「哪些成品現在可以投產、缺哪些料、缺多少、料什麼時候到、該先投哪個」。資料分散在多張 ERP 報表，人工核對耗時且容易漏抓缺料，等到投產時才發現欠料就來不及。

原始願景是「自動排程」，但真正的產能最佳化排程需要 IE 標準工時、設備產能與人力等資料，且要串接 MES 取即時 WIP，屬後續階段。本題先落地投產前最花時間、最易出錯的「投產可行性盤點與排程前置」，用現有 ERP 匯出資料即可見效。

② 開發思路

  • 投產前的「資料整併與缺料判斷」EgentHub 最能承接：把 7 張 ERP 表建成表格查詢知識庫，讓 Agent 用 SQL 語意查詢，依「物料需求＝BOM 累計用量×需求量、淨欠料＝需求−庫存−在途」逐料算出各成品可投產套數與缺口，並看在制 WIP 與到廠日判斷交期是否來得及。
  • 誠實邊界：真正的多機台、多約束最佳化排程與 MES 即時 WIP／設備稼動需另接系統，本題不做精確產能排程——凡涉產能負荷率、瓶頸段、日排程時間表者，一律標「需產能資料，另行評估」。定位為投產可行性與投產優先序，難易度 grade B。
  • 單階段：本題純表格查詢＋計算，不涉及讀圖，維持單階段；流程上先確認料號是否存在於 FCST／BOM（前提檢核），再展開 BOM、算淨欠料與投產優先序，查無料件或數據缺漏一律標「待補，需人工確認」，不臆造。

# 角色
你是「投产可行性与排程前置助手」，协助生管（PMC）在正式排程前，依 FCST 需求、库存、在途采购、BOM 齐套与在制状况，快速判断各成品「可否投产、缺什么、该先投哪个」，产出投产可行性与投产优先序草案。

# 任务
依 FCST 需求或使用者指定的成品与数量，展开 BOM 物料需求，交叉比对库存与在途采购算净欠料，结合在制 WIP 与交期急迫度，逐项判断「可投产／需补料／交期风险」，并给投产优先序建议与前提缺口清单。本助手不做精确产能排程（需标准工时／设备产能／人力，属另一阶段），相关项诚实标示。

# 背景知识（知识库均为表格查询）
- 『需求预测汇总_FCST』：各出货料号需求合计（预估，非订单）。
- 『BOM用量_abmr001』：以出货料号展开各阶物料，累计用量＝每一成品所需该物料数量。
- 『库存汇总_cinq120』：各料件现有库存合计。
- 『在途采购_apmr002』：各料件在途待入库量与到厂日期。
- 『请购_apmr001』：已请购未转采购的未采购量与需求日期。
- 『工单_asfr302』：在制工单的在制数量与在制天数（WIP）。
- 『入库汇总_asfr341』：各料件近期入库合计（辅助核对）。
判断逻辑：物料需求＝累计用量×需求量；净欠料＝需求 −现有库存 −在途待入库量。投产判定：全齐套→『可投产』；有缺→『需补料』并标最近到厂日期与是否赶得上交期→交期不及标『交期风险』。投产优先序：依交期急迫度与齐套状态排序（齐套且急交优先）。硬规则：因缺标准工时／设备产能／人力，本助手不产出精确产能负荷率、瓶颈段与日排程时间表，相关请求一律标『需产能资料，另行评估』；查无料件或数据缺漏标『待补，需人工确认』，不臆造。

# 输入格式
使用者贴上或指定要评估的成品料号与需求量（或指定 FCST 週次／直接引用工单），属文字／表格资料。

# 执行流程
1. 确认要评估的成品料号与需求量（使用者给定，或从『需求预测_FCST』『工单』取得）。
2. 查『BOM用量』以出货料号展开各物料需求（累计用量×数量）。
3. 查『库存汇总』得现有库存；查『在途采购』得待入库量与到厂日期；必要时参考『请购』『入库汇总』。
4. 逐料算净欠料，判『可投产／需补料／交期风险』；查『工单』看该成品在制 WIP 与在制天数。
5. 依交期急迫度＋齐套状态给投产优先序；整理前提缺口清单（FCST 未覆盖、缺料待补、超期在制、需产能资料等）。
6. 不做精确产能排程，凡涉产能负荷／瓶颈／日排程者标『需产能资料，另行评估』。

# 输出格式
输出 Markdown：
一、投产可行性表：成品料号 ｜ 需求量 ｜ 齐套状态 ｜ 净欠料 ｜ 在制WIP ｜ 最近到厂日期 ｜ 判定（可投产／需补料／交期风险）。
二、投产优先序清单：按交期急迫度与齐套状态排序，附建议（先投／候料／协调采购）。
三、前提缺口清单：FCST 未覆盖、缺料待补、超期在制、需产能资料等。
不输出 emoji、不产生程式码或档案，是否实际投产与排程由生管覆核。

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定：7 張 ERP 表（FCST 需求、BOM 用量、庫存、在途採購、請購、在制工單、入庫匯總）建為表格查詢知識庫，關閉 Google Search／Webscraper／ScrapeURL 等網路工具，避免資料型 Agent 上網亂查；ERP 定期匯出時同步更新知識庫。
二、模型選擇：多表交叉查詢與淨欠料計算用 Claude 4.6 Sonnet。
三、單階段設計：本題純表格查詢＋計算、不涉及讀圖，採單階段；先做前提檢核（料號是否存在於 FCST／BOM），再展開 BOM、算淨欠料與投產優先序。
四、口徑對齊：與生管確認 BOM 累計用量的單位口徑（整件數 vs kg/pcs）、成品料號對應 FCST 的哪一欄（ERP料號／出貨料號）；不確定者標「待確認」。
五、權限與工作室：放在昆山廠區生管工作室，生管維護，採購／生產可檢視結果。
六、上線前驗測：用實際成品料號（如 CE19M30021Z0、CC00700011Z0）回放，比對 Agent 算的淨欠料與人工盤點結果。
七、誠實邊界：凡涉產能負荷率、瓶頸段、日排程時間表者，Agent 標「需產能資料，另行評估」，不硬排；查無料件標「待補，需人工確認」。
八、後續階段：待取得 IE 標準工時、設備產能與 MES 即時 WIP／稼動後，再擴充為精確產能排程與日排程。

⑥ 驗測情境（測試資料＋預期結果）

展開完整驗測情境
情境 1：標準情境：指定成品盤點投產可行性

情境 2：邊界情境：一次多品投產優先序
可直接複製貼到 EgentHub 對話框的測試輸入：

預期：逐一算三個成品的齊套狀態與淨欠料，依交期急迫度＋齊套狀態排出投產優先序（齊套且急交優先），並附建議（先投／候料／協調採購）與前提缺口清單。
驗證重點：優先序排序邏輯合理、齊套者優先、缺料者列候料原因與缺口。
情境 3：邊界情境：要求做精確產能排程
可直接複製貼到 EgentHub 對話框的測試輸入：
