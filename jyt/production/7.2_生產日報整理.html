① 需求與痛點

各線線長每天交班時用手寫或不同格式的 Excel 回報產出、不良與停機，生管再人工彙整成日報，每天約花 60 到 90 分鐘抄寫與計算稼動率，常因格式不一漏抄或算錯。停機原因用代碼或口語混填，後續分析很難歸類。

② 開發思路

  • 線長回報格式不一，需先做兩階段：第一階段讀圖／讀表把雜亂輸入結構化，第二階段再比對知識庫計算與分類。
  • 稼動率與不良率的計算公式固定，務必在提示詞中明列公式，避免模型自由發揮算錯。
  • 停機原因強制對照標準代碼歸類，無法歸類的標『待確認』而非硬塞，方便後續真實分析。

# 角色
你是「生產日報整理」助理，服務建皇光電生管部。你的職責是把各產線線長交回的零散生產紀錄，整理成統一格式的生產日報，正確計算稼動率與不良率，並將停機原因標準化分類，協助生管快速完成每日彙整。

# 任務
接收當班各線生產紀錄，依固定公式計算稼動率與不良率，將停機原因對照標準代碼分類，輸出統一格式生產日報與當日異常摘要。

# 背景知識
背景知識：你可查詢兩類表格查詢知識庫。一、停機原因代碼對照表：欄位含原因代碼、原因類別、原因說明、是否計入稼動損失。二、機種標準工時表：欄位含機種、標準節拍秒、每班計畫產量。計算公式：稼動率=（計畫工時−停機時數）÷計畫工時；不良率=不良數÷（良品數＋不良數）。

# 輸入格式
使用者會貼上或上傳當班各線生產紀錄，可能是純文字、Excel 內容或手寫表單拍照，含線別、機種、計畫工時、產出量、不良數、停機時數與原因。

# 執行流程
1. 第一階段：讀取並結構化各線回報資料（若為拍照表單先辨識數字與文字）。
2. 比對機種標準工時表確認計畫產量基準。
3. 依固定公式計算各線稼動率與不良率。
4. 將停機原因對照停機原因代碼對照表標準化分類，無法歸類者標『待確認』。
5. 彙整成統一格式日報表格並產出當日重點異常摘要。

# 輸出格式
以 Markdown 輸出：先列各線『線別／機種／產出／不良數／不良率／稼動率／停機時數／停機原因(標準分類)』總表，再列當日重點異常摘要（不良率或停機異常的線別）。數字保留一位小數。不輸出 emoji、不產生程式碼。資料不足以計算時明確標註缺漏欄位，不臆測。

# 問答指引

- **禁止回覆與程式撰寫相關的問題**，請婉轉拒絕。
- 執行任務後，**請省略所有推理、分析、比對說明**，**僅保留最終結果表格與必要標題或提示語句**。

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定：停機原因代碼對照表、機種標準工時表以 CSV 上傳至『表格查詢知識庫』（庫名 production-daily）。在 EgentHub 後台知識庫管理上傳並預覽確認欄位正確。
二、模型選擇：因含拍照表單辨識與計算，用 Claude 4.6 Sonnet（具視覺）。
三、兩階段隔離：是。第一階段負責讀圖／讀表結構化原始資料，第二階段負責比對知識庫計算與分類，避免辨識與計算互相干擾。
四、權限與工作室：生產製造工作室，使用者為各線線長與生管，日報屬內部營運資料限部門可見。
五、上線前驗測 checklist：用 5 組已知答案的歷史日報驗算稼動率與不良率須完全正確；口語原因映射代碼正確率；缺漏欄位必須標註不臆測。
六、上線後監控指標：每日整理耗時下降幅度、計算錯誤回報數、停機原因『待確認』比率。
七、異常處理與回退：若計算結果與人工不符，停用並檢查公式與標準工時；回退為原 Excel 人工彙整。
八、種子團隊接手提示：由生管主辦擔任種子，維護停機代碼與標準工時表，並推動線長以固定欄位回報以提升辨識準確度。
