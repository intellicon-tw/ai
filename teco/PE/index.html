五項設計原則 — 您觀察 Agent 行為時的依據

原則一：流程優先於模型

先定義 PE 業務該如何跑估算 SOP、如何判讀合約風險，AI 才介入輔助每個步驟。

原則二：文件完整性是硬閘門　PE 特有

合約少一頁、附件不齊、估算欄位缺失時，Agent 主動退回要求補資料，不對殘缺資訊做判讀。

原則三：判斷邏輯與資料分離

判斷規則、估算邏輯、條款風險判讀邏輯內建於 Agent 行為規範；條款庫、風險案例、估算項目、設備單價則存於企業知識庫。

原則四：跨角色視角支援　PE 特有

業務、PM、法務、技術問同一份合約，Agent 依角色提供不同視角的回應 — 法務看風險、技術看規格、業務看商業條件。

原則五：人機協作，不是全自動

AI 的責任是加速、結構化、指出不確定性。最終估算數字與合約簽核仍由業務、PM、法務、業務主管做決策。

2POC Agent 設計理念

理解這四個 Agent 為什麼這樣拆分、為什麼要有兩道閘門、為什麼支援跨角色視角 — 這些設計決策背後都有 PE 業務特性下的刻意考量。

為什麼是 4 個 Agent，而不是 1 個「萬能 Agent」

把工作拆成 4 個專責 Agent，是刻意的設計決策：

  • 失敗範圍可控 — 某個 Agent 表現不好不會影響其他 Agent
  • 業務有彈性的進入點 — 想做即時查詢用 1.1、想要完整估算引導用 1.2、想看合約結構用 1.3、想問合約問題用 1.4
  • 輔助型 + 核心型分工 — 輔助型（1.1、1.3）處理單點查詢與文件解析等高頻雜事，核心型（1.2、1.4）專心跑完整 SOP 給判斷
  • 培訓與交接更容易 — 種子團隊可以一個一個 Agent 學會，不必一次掌握整個複雜流程

為什麼 PE 案有「兩道閘門」（資料閘門 + 文件完整性閘門）

傳統的 LLM Agent 在資料不完整時會「強行給看似專業實則根據不足的答案」。對 PE 案而言，這種輸出特別危險 — 一個對殘缺合約做出的「沒看到紅旗條款」結論，可能讓業務以為合約安全，最終造成嚴重損失。所以 PE 案採用兩道閘門：

  • 資料閘門（用於估算 Agent 1.1 / 1.2）：每次跑 SOP 時先評估「資訊充分度（0-100 分）」，分數 < 60 不給數字答案，只列出補資料清單
  • 文件完整性閘門（用於合約 Agent 1.3 / 1.4，PE 特有）：合約上傳前先檢查章節齊全、附件到位、頁碼連續；若有缺漏，拒絕進行條款分析，要求補上後再判讀

這兩道閘門分別對應原則二「文件完整性是硬閘門」的兩個面向 — 一個保護估算數字、一個保護合約判讀。

為什麼支援「跨角色視角」（PE 特有）

PE 業務涉及五個角色：業務、PM、法務、技術、財務。每個角色看同一份合約、同一個案件的關注點完全不同：

  • 業務 看商業條件與客戶關係影響
  • PM 看工期義務與資源配置
  • 法務 看條款風險與責任歸屬
  • 技術 看規格符合度與履約難度

1.4 合約問答助手在回應使用者問題時，會依使用者輸入的角色（業務／PM／法務／技術）調整聚焦點與用語。同一句「這合約有什麼問題？」，業務拿到的是商業風險，法務拿到的是條款風險，視角完全不同。

為什麼採用 Phase 1-7 結構化 SOP

把資深 PM 與法務的判斷邏輯轉成 7 個明確的 Phase 步驟，是為了：

  • 讓判斷邏輯被組織保存 — 不再藏在資深 PM 與法務腦中
  • 每個 Phase 都有明確輸入輸出，方便驗測（例如：Phase 4 的輸出是否引用了 KB 案例編號）
  • 新人也能產出水準一致的初稿，再由資深 PM 與法務做最後校對
  • 1.2 與 1.4 的 Phase 結構刻意對稱（都是 Phase 1 需求定義 → Phase 2 閘門 → Phase 3 結構化 → Phase 4-6 核心戰場 → Phase 7 輸出），讓組織學習成本降到最低

為什麼用「條件式兩階段」而非「全部兩階段」

EgentHub 的雙階段隔離設計是為了避免「同時讀圖與調用工具」造成的判斷品質下降。但對 1.3 / 1.4 這類「有時讀 PDF、有時純文字」的 Agent，全部都走兩階段反而拖慢純文字輸入時的回應速度。所以採用條件式設計：

  • 上傳合約 PDF 時：先進步驟一（純讀 PDF、結構化、完整性檢查、不調 KB），再進步驟二（比對標準條款庫、風險案例庫）
  • 純文字輸入時：直接走步驟二，省掉不必要的階段切換（例如業務直接問「我的付款條件是什麼？」並貼上條款內容）

知識庫

本 Agent 整合三個知識庫：上傳的合約 PDF、PE 標準合約條款庫、過往合約風險案例庫。

PE 標準合約條款庫　（14 行）

用途：比對客戶版本合約與東元慣用版本的差異，識別紅旗條款。

欄位：條款類型、條款名稱、標準條款摘要、東元慣用版本要點、紅旗條款警示、修訂建議

內容範圍：商業條款（付款、罰則、履約保證、驗收）、技術條款（規格、保固、性能保證）、法務條款（智財、保密、不可抗力、終止）、風險條款（保險、爭議解決、稅務）共 14 條。

過往合約風險案例庫　（8 行）

用途：援引過往風險案例，提示「曾經因類似條款踩過坑」。

欄位：案例編號（RISK-XXXX-XX）、案例描述、風險類型、發生原因、損失金額區間、處理方式、後續改善

內容範圍：原物料風險、匯率風險、工期風險、客變風險、性能風險、供應鏈風險、驗收風險、法務風險共 8 類典型風險案例。

測試重點預期：1.4 收到「南亞 ESS 合約罰則無上限」這類問題時，應援引標準條款庫中的紅旗警示 + RISK-2023-02（工期風險）；視角依使用者角色調整。

5核心觀察點

這四個觀察點是本次 POC 最想驗證的設計理念，請您在測試時特別留意。

資料閘門啟動（估算 Agent）

當您給 1.1 / 1.2「不完整的案件特徵」（測試 #3、閘門測試），Agent 是否確實啟動「資訊充分度評分」並婉拒給數字？或是它仍然強行給出看似專業實則根據不足的估算？這個觀察點直接驗證原則二。

文件完整性閘門啟動（合約 Agent）

當您給 1.3 / 1.4「殘缺的合約」（缺附件、缺章節），Agent 是否拒絕進行條款判讀？對殘缺合約做出「沒看到紅旗條款」的結論是 PE 案最危險的錯誤 — 這個閘門直接保護組織不踩坑。

跨角色視角支援

對 1.4 合約問答助手，請務必嘗試以不同角色（業務／PM／法務／技術）提問同一份合約。Agent 的回應視角是否確實切換？業務看到的是商業條件影響，法務看到的是條款風險與修訂優先級。這是 PE 案的特有設計。

歷史案例與條款庫援引

1.2 的輸出是否能具體援引「歷史 PE 案件估算紀錄」中的案例編號（PE-XXXX-XXX）？1.4 的輸出是否能援引「過往合約風險案例」中的案例（RISK-XXXX-XX）？這代表 Agent 真的在查 KB 而不是憑空想像。
