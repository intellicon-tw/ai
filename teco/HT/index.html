五項設計原則 — 您觀察 Agent 行為時的依據

原則一：流程優先於模型

先定義 HT 區經理該如何看銷售資料、如何決定巡店優先級、如何評估展店，AI 才介入輔助每個步驟。

原則二：資料閘門 — POS 品質是硬條件

品類分類不一致、日期跳日、銷售件數異常時，Agent 主動退回要求補資料；不對殘缺資料做預測。

原則三：判斷邏輯與資料分離

預測模型、異常偵測規則、巡店優先級邏輯內建於 Agent；門市、商圈、銷售、巡店、展店資料則存於知識庫。

原則四：解釋優先於預測　HT 特有

預測數字本身不重要，重要的是「為什麼會這樣」與「該怎麼做」。沒有解釋的預測不是有用的預測。

原則五：人機協作，不是全自動

AI 的責任是加速、結構化、指出不確定性。最終巡店、展店、促銷決策仍由區經理、營業部主管、決策者做決策。

2POC Agent 設計理念

理解這兩個 Agent 為什麼這樣拆分、為什麼要強調「解釋」、為什麼有資料品質閘門 — 這些設計決策背後都有 HT 通路業務特性下的刻意考量。

為什麼是 2 個 Agent，而不是 1 個「萬能 Agent」

單一題目其實涵蓋兩個不同的工作面向 — 銷售資料分析（看資料）與行動建議產出（做決策）。把這兩個面向對應到兩個 Agent：

  • 失敗範圍可控 — 某個 Agent 表現不好不會影響另一個
  • 使用時機分離 — 1.1 早上看儀表板用、1.2 週一規劃巡店或評估展店時用，業務節奏不同
  • 各自獨立可用 — 不必每次都跑完整鏈路；1.1 的儀表板可單獨給營運高層看，1.2 的展店評估可不依賴 1.1 直接執行
  • 培訓與交接更容易 — 種子團隊可以一個一個 Agent 學會，不必一次掌握整個複雜流程

為什麼有「資料品質閘門」（HT 案的核心安全防線）

傳統的 LLM Agent 在資料不完整時會「強行給看似專業實則根據不足的答案」。對 HT 通路業務而言，這種輸出會誤導區經理採取錯誤對策 — 把員工問題誤判為商圈問題、用品類分類錯誤的資料做預測、把缺日的銷售資料當成「真的下滑」。所以本案採用「資料品質閘門」設計：

  • 每次跑 SOP 前，先評估「資料品質（0-100 分）」
  • 檢查：日期區間完整性、品類分類一致性、銷售件數合理性
  • 分數低於 60 時，Agent 不做預測，只列出資料缺口清單
  • 業務知道要重新匯出哪些資料才能繼續，而不是收到看似精準但實際根據不足的預測

為什麼強調「解釋優先於預測」（HT 特有）

HT 案的核心設計理念是「解釋優先於預測」 — 一個沒有解釋的預測（「下週 S007 會下滑 15%」）對區經理沒有用，他不知道該怎麼處理。一個有解釋但解釋錯誤的預測（「下滑是因為員工流動」但其實是商圈問題）甚至比沒有預測更危險，會誘導區經理採取錯誤對策。所以本案的 Agent 在給出預測時，必須同時給出：

  • 主因解釋 — 從 5 種候選原因（員工流動／商圈變化／競品衝擊／促銷檔期／季節影響）中找出最可能組合
  • 援引來源 — 引用 KB 中的具體紀錄（例如「巡店紀錄 2026-02-12 提到新店長報到 1 個月」）
  • 建議行動 — 應該採取什麼具體行動
  • 歷史可比案例 — 過去類似情況是怎麼處理的

為什麼採用 Phase 1-7 結構化 SOP

把資深區經理的判斷邏輯轉成 7 個明確的 Phase 步驟，是為了：

  • 讓判斷邏輯被組織保存 — 不再藏在資深區經理腦中
  • 每個 Phase 都有明確輸入輸出，方便驗測（例如：Phase 4 的下滑主因是否引用了巡店紀錄）
  • 新進區經理也能產出水準一致的分析，再由資深區經理做最後校對
  • 1.1 與 1.2 的 Phase 結構刻意對稱（都是 Phase 1 需求定義 → Phase 2 閘門 → Phase 3 結構化 → Phase 4-6 核心戰場 → Phase 7 輸出），讓組織學習成本降到最低

為什麼採用「鬆耦合」串接（Markdown 中介產物）

1.1 → 1.2 的串接，刻意採用 Markdown 文字作為中介產物，而不是 API 直連：

  • 業務直接複製貼上即可串接，不需特殊框架支援
  • 可中斷可重啟 — 早上跑 1.1 後，下午再用 1.2，輸出文字保留即可
  • 各自獨立可用 — 1.1 儀表板可單獨給營運高層看，1.2 展店評估可不依賴 1.1
  • 可審計 — Agent 的輸入輸出都是純文字，方便 Code Review 與稽核追蹤

5核心觀察點

這四個觀察點是本次 POC 最想驗證的設計理念，請您在測試時特別留意。

資料品質閘門啟動

當您給 1.1「品類不一致」或「日期跳日」的資料（測試 #3、#4），Agent 是否確實啟動「資料品質評分」並婉拒預測？這是 HT 案的核心安全防線 — 不對殘缺資料做誤導性預測。

解釋優先於預測

對 1.1 的下滑主因解釋，請特別觀察：(1) 是否援引巡店紀錄具體日期（例如 2026-02-12 提到員工流動）；(2) 是否歸因為單一原因還是多重組合；(3) 解釋是否真的反映業務 reality。HT 案最重視的就是「解釋」的品質。

巡店優先級的合理性

對 1.2 的巡店優先順序，請以資深區經理視角判斷：排序是否合理？S007 異常店是否確實排第 1？每店推薦巡店重點是否援引歷史巡店紀錄與後續成效？

展店歷史案例援引

1.2 的展店評估，是否援引「展店評估參考案例」中的具體案例編號（DSE-XXXX-XX）？同一候選地有歷史評估時是否引用？學習點是否被應用於新候選地的判斷？
