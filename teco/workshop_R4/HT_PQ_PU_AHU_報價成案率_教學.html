PQ／PU／AHU 報價與成案率分析助手

把「報價」和「成案率分析」拆成兩支 Agent，先用 POC 搭骨架、邊蒐資料邊長大。本頁含可直接複製的提示詞與資料蒐集清單。

這個助手要解決什麼

業務／工程在報 PQ／PU／AHU 案件時，選型與報價要查很多表、靠經驗；報完之後，哪些會成案、為什麼輸，也缺乏系統化回顧。這支助手要同時幫上這兩件事——但這兩件事本質不同，建議拆成兩支 Agent。

① 報價助手
文件生成／選型計算型
輸入案件需求 → 選機型與配件 → 產出報價草稿與選型理由。先做這支，看得到、見效快。
② 成案率分析
預測／分析型
吃歷史「報價 vs 成交」資料，找出成案的規律，並對新報價給成案研判與建議。第二期再做，它吃資料量。

三個關鍵建議

  1. 拆兩支、別綁一起。報價是「選型＋生成」，成案率是「分析＋研判」，資料需求與驗收方式都不同。先把報價助手做出來，成案率分析做好了再串。
  2. 別等資料到齊才開始。用你們已有的試作內容＋表單欄位定義，先把 prompt、知識庫結構、輸出格式定下來（POC）。資料到位再灌進去。先求能跑，再逐步加深。
  3. 成案率分析的成敗，90% 在「表單收了哪些欄位」。沒有「結果（成交／未成交）＋輸贏原因」，再強的 AI 也分析不出成案率。資料蒐集清單見第 ③ 分頁。
對應目前進度：你們已有①項目需求 ②功能架構與流程 ③蒐集表單 ④小範圍試作——這正好是 POC 的起手包。下一步不是等資料，而是先用試作內容把「報價助手」的提示詞與知識庫骨架做出來，同時用第 ③ 分頁的清單去校準表單欄位（確保收得到成案率分析要的資料）。

落地節奏建議

第 1 期：報價助手 POC→ 同步校準表單欄位→ 累積歷史報價–成交資料→ 第 2 期：成案率分析

① 報價助手

輸入案件需求 → 選型與配件 → 產出報價草稿與選型理由，讓業務快速確認、調整後對外報價。

開發前必答的 5 個問題（填好可貼進提示詞設計助手）

  1. 應用場景與業務流程：幫業務／工程在收到 PQ／PU／AHU 案件需求時，快速選型並產出報價草稿。
  2. 資料輸入與來源：案件需求（文字或上傳客戶規格書／圖面）——場域、面積／噸位、風量／靜壓、電源、數量、交期、特殊需求。
  3. 預期輸出內容：報價草稿（品項／型號／規格／數量／參考單價／小計／交期）＋選型理由＋需人工確認項目。
  4. 任務步驟：①理解需求 ②查選型規則與規格 ③選機型配件 ④帶參考單價 ⑤彙整報價草稿。
  5. 知識來源：選型規則表、產品規格表、報價範本與歷史單價（表格查詢為主）。

開發思路

若客戶會附規格書／圖面，採兩階段：階段一只讀檔、把需求整理成結構化文字（不查庫）；階段二再依整理結果查選型知識庫、算報價。避免「看圖」與「查表」互相干擾。

完整系統提示詞（可複製，貼進 Dify Agent 的提示詞欄位）

＃ 角色
你是東元電機 HT 事業處的「PQ／PU／AHU 報價工程師助手」，
協助業務／工程依客戶案件需求完成機型選配與報價草稿。

＃ 任務
依輸入的案件需求，選出合適機型與配件，產出結構清楚的報價草稿與選型理由，
讓業務能快速確認、調整後對外報價。

＃ 背景知識（知識庫）
- 選型規則表：依噸數／風量／靜壓／使用場域對應建議機型與配件（表格查詢）。
- 產品規格表：PQ／PU／AHU 各機型規格（表格查詢）。
- 報價範本與歷史單價：報價單格式與各品項參考單價（模板＋表格查詢）。

＃ 執行流程（若附規格書／圖面，採兩階段）
階段一（理解需求）：只讀客戶規格書／需求文件／圖面，整理成結構化需求
（場域、面積／噸位、風量／靜壓、電源、數量、交期、特殊要求）。此階段不查知識庫。
階段二（選型與報價）：依整理好的需求查選型規則與規格表，選機型與配件，
帶入參考單價，產出報價草稿。

＃ 輸入格式
案件需求（文字或上傳規格書）：場域用途、面積／噸位、風量／靜壓、電源、
數量、交期、特殊需求。

＃ 輸出格式（Markdown）
1. 需求摘要（我理解的需求，請業務確認）
2. 建議機型與配件：品項｜型號｜規格｜數量｜參考單價｜小計｜選型理由
3. 報價彙總：設備小計、配件小計、預估交期、報價有效期
4. 提醒：需人工確認的項目（特殊規格、現場條件、最終定價）

＃ 問答指引（固定，原文照貼不改）
- 查不到對應規格或單價時，明確標「資料庫無此項，請人工補」，絕不杜撰型號或價格。
- 參考單價僅供內部估算，最終對外報價由業務／主管核定。
- 事實（規格、單價）與建議（選型理由）分開呈現。
- 繁體中文，技術術語保留英文。

知識庫設計

  • 選型規則表（表格查詢）：場域／噸數／風量／靜壓 → 建議機型＋配件。
  • 產品規格表（表格查詢）：PQ／PU／AHU 各型號規格欄位。
  • 報價範本（模板）＋歷史單價表（表格查詢）：報價單格式與各品項參考單價。

建置流程（Dify）

  1. 建 Agent App，貼上上方系統提示詞。
  2. 上傳三類知識庫（先用試作的小樣本，能跑就好）。
  3. 需讀規格書 PDF 才開「兩階段」；純文字輸入可單階段。
  4. 用 2–3 個真實案件試跑，調到報價草稿可用。

驗測情境

  • 正常案：一個標準商辦 AHU 案，確認選型、單價、彙整正確。
  • 邊界案：規格庫查無的特殊噸位／特殊需求——確認它會說「資料庫無此項，請人工補」而不是硬湊。
  • 缺資料案：輸入缺靜壓／電源——確認它會回問或標註待補，不擅自假設。

② 成案率分析

根據歷史「報價 vs 成交」紀錄，找出影響成案的因子，並對新報價給成案研判與建議。

定位（很重要）：先做「描述性分析＋建議」，不要做黑箱機率預測。案件量通常不足以訓練嚴謹模型；定位成「哪類案子成案率高／低、這張報價相對歷史落在哪、風險與建議」最實用且可信。等資料累積到數十～上百筆且有輸贏標記，再談機率預測。

開發前必答的 5 個問題

  1. 應用場景與業務流程：回顧歷史報價成敗、找出成案規律；並對新報價給成案機會研判與建議。
  2. 資料輸入與來源：一張「歷史報價–成交對照表」（見第 ③ 分頁欄位）；單案研判時另輸入本案資訊。
  3. 預期輸出內容：整體分析（成案率高／低的族群、影響因子）＋單案研判（高／中／低＋理由＋建議）。
  4. 任務步驟：①釐清整體或單案 ②查歷史做描述性統計 ③本案落點比對 ④給研判與建議＋資料依據。
  5. 知識來源：歷史報價–成交對照表（表格查詢，必須含「結果＋輸贏原因」）。

完整系統提示詞（可複製）

＃ 角色
你是東元電機 HT 事業處的「報價成案率分析助手」，
根據歷史報價與成交紀錄，找出影響成案的因子，並對新報價給成案研判與建議。

＃ 任務（兩種模式）
（A）整體分析：從歷史資料找出「哪類案子成案率高／低」的規律。
（B）單案研判：對一張新報價，比對歷史，給成案機會 高／中／低 的研判與可行建議。

＃ 背景知識（知識庫）
- 歷史報價–成交對照表（表格查詢）：每筆含產品線、案件規模、報價金額、毛利、
  客戶類型／產業、案件來源、交期、競爭對手與其報價、業務員、
  結果（成交／未成交）、輸贏原因。這是分析的命脈。

＃ 執行流程
1. 先釐清是「整體分析」還是「單案研判」。
2. 查歷史對照表做描述性統計：依產品線／規模／價格帶／客戶類型／案件來源分群，比較成案率。
3. 單案研判：把本案放進對應分群，看相對歷史落點（價格偏高？交期偏長？該客戶類型成案率？），
   給 高／中／低 研判與 2–3 個可行建議。
4. 每個結論都附「依據哪些歷史資料」。

＃ 輸入格式
- 整體分析：直接問（可指定產品線／期間）。
- 單案研判：本案的產品線、案件規模、報價金額、客戶類型、案件來源、交期、競品報價（若知）。

＃ 輸出格式（Markdown）
1. 分析範圍與樣本數（明說用了幾筆歷史資料）
2. 關鍵發現（成案率高／低的族群、主要影響因子）
3. 本案研判（若為單案）：成案機會 高／中／低 ＋ 理由 ＋ 2–3 建議
4. 資料品質提醒（樣本不足或欄位缺漏要說明）

＃ 問答指引（固定，原文照貼不改）
- 不做黑箱機率預測；只做有資料依據的描述性分析與研判，每個結論說清楚來自哪些資料。
- 樣本數太少（例如該族群少於 10 筆）必須明講「樣本不足，僅供參考」。
- 絕不編造數字或趨勢。
- 繁體中文，技術術語保留英文。

知識庫設計

核心就一張歷史報價–成交對照表（表格查詢）。它的欄位完整度，直接決定這支 Agent 有沒有用——欄位清單與必填項見第 ③ 分頁。

驗測情境

  • 整體分析：問「哪個價格帶／客戶類型成案率最高」，確認它有引用樣本數與依據。
  • 單案研判：丟一張明顯偏貴的報價，確認它判低成案並指出原因、給建議。
  • 樣本不足：問一個歷史很少的冷門族群，確認它會說「樣本不足，僅供參考」而非硬給結論。

③ 資料蒐集清單

給對應單位蒐集資料時用。紅色為成案率分析的必填欄位——缺了這些，成案率就分析不出來。

欄位說明用途
案件編號／報價日期唯一識別與時間序兩者皆需
產品線PQ／PU／AHU（可複選）兩者皆需
案件規模噸數／台數／金額級距兩者皆需
客戶名稱／客戶類型・產業直客／經銷／系統商；製造／商辦／醫院…成案率必填
案件來源經銷、直客、公開標案、舊客回購…成案率必填
需求規格摘要場域、風量／靜壓、電源、特殊需求報價用
報價金額／估算成本・毛利對外報價與內部成本（毛利率）成案率必填
交期承諾／要求交期兩者皆需
競爭對手與其報價對手品牌、得標價（若知）成案率（加分）

結果成交／未成交／未定 — 最關鍵成案率必填

一句話提醒：表單一定要收到「結果＋輸贏原因」這兩欄。很多公司收了一堆報價資料卻沒記「最後有沒有成、為什麼輸」，結果成案率分析做不起來。這兩欄現在就要加進表單。

資料量門檻（誠實的期待管理）

  • 單案研判要有意義，對應族群至少要有約 10 筆以上有輸贏標記的歷史。
  • 整體規律要穩，最好累積 數十～上百筆。
  • 資料還不夠時，第 1 期先讓「報價助手」見效（價值在快與一致），成案率分析邊累積邊上。
