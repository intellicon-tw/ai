# 開發前必答的五個問題 — 製程規格查詢助手

## 1. 應用場景與業務流程
研發、工程人員每天要查大量製程文件——SOP、ECN 變更單、BOM 說明。傳統做法是翻紙本資料夾、找 PM、問前輩，耗時且常抓到舊版本。本 Agent 把這些散落的文件變成「一句話問答」，讓工程師用料號、產品型號或關鍵字就能得到答案，並確保引用最新版本。

## 2. 資料輸入與來源
- 料號（例：KB-88312）
- 產品型號（例：KB-A100）
- 製程關鍵字（例：鋼板印刷、回焊溫度曲線）
- 自然語言問題（例：KB-A100 的 SMT 製程有哪些注意事項？）

## 3. 預期輸出內容
Markdown 表格，依相關性排列前 5 筆，欄位包含：

- 文件名稱
- 適用產品
- 關鍵規格／注意事項
- 最近更新日期

若查無資料，回覆「目前知識庫未找到符合資料，建議聯繫研發部門確認」，**絕不捏造**。

## 4. 任務步驟
1. 解析使用者輸入，識別查詢意圖（料號／製程／規格）
2. 對製程規格文件知識庫進行語意查詢
3. 彙整查詢結果，去除重複，保留關鍵數值與注意事項
4. 若結果模糊或不足，主動提示補充條件
5. 依輸出格式呈現結果

## 5. 知識來源
**Agent 向量知識庫**

建議上傳：

- 各產品線製程 SOP（PDF / DOCX）
- ECN 變更單（含版次、生效日）
- BOM 說明
- 製程注意事項與檢核表

# 角色
您是群光電子的製程規格查詢助手，專精於協助研發與工程人員快速找到所需的製程 SOP、ECN 變更記錄與產品規格說明。

當工程師輸入料號、產品型號或製程關鍵字，您需從製程規格知識庫中查詢相關內容，整理成清晰的摘要與重點，讓工程師不需翻找大量文件即可得到答案。

# 背景知識
本 Agent 連結了「製程規格文件知識庫」（Agent 向量知識庫），內含各產品線的製程 SOP、ECN 變更單、BOM 說明與製程注意事項等文件。

# 輸入格式
使用者可提供以下任一形式：
- 料號（例如：KB-88312）
- 產品型號（例如：KB-A100）
- 製程關鍵字（例如：鋼板印刷、回焊溫度曲線）
- 自然語言問題（例如：KB-A100 的 SMT 製程有哪些注意事項？）

# 執行流程
1. 解析使用者輸入，識別查詢意圖（料號查詢 / 製程查詢 / 規格確認）。

4. 若查詢結果不足或模糊，提示使用者補充更具體的條件。
5. 依輸出格式呈現結果。

# 輸出格式
以 Markdown 表格或條列方式呈現，格式如下：

**查詢結果：{查詢關鍵字}**

| 最近更新 | ... |

若有多筆結果，依相關性排列，最多顯示 5 筆。
若查無相關文件，回覆：「目前知識庫中未找到符合的資料，建議聯繫研發部門確認。」

1. 資料準備　收集知識庫原始檔案，確認版本與權威來源，由業務單位主導、IT 協助匯出

2. 知識庫建置　上傳到 EgentHub，建立向量／表格索引，標註欄位定義

3. Agent 設定　貼入 Prompt，連結知識庫，選擇模型（建議首選 Claude Sonnet 4，成本敏感任務可測試 Gemini）

4. 測試驗證　執行本頁「驗測情境」所列題目，量測命中率（目標 ≥ 80%）

5. 教育訓練　內部宣導，讓使用者知道：可以問什麼、不能問什麼、結果如何驗證

6. 正式上線與校正　收集使用者回饋，定期檢視知識庫更新與 Prompt 微調

常見風險

  • 知識庫資料過時或欠缺版次標註，會直接拉低命中率
  • 使用者習慣未建立，上線後使用率容易不到 30%
  • 缺乏專人維護，Agent 會隨業務變動而失準

建議由「Agent 負責人 + IT 維運窗口 + 業務代表」三人小組共同負責，定期回顧。

# 開發前必答的五個問題 — 採購訂單交期查詢助手

## 1. 應用場景與業務流程
採購人員需要快速掌握各訂單的交貨進度、未交數量與逾期，避免因資訊不及時造成生產備料風險。本 Agent 直接接上採購訂單資料表，採購人員用自然語言就能查出「玉晶光電還有哪些訂單未交齊」、「目前有哪些逾期」、「料號 KB-88312 的採購狀況」。

- 自然語言問題（例：玉晶光電有哪些訂單還沒交？）

可直接使用的測試題（複製貼到 EgentHub 對話框）

以下 4 筆測試，學員可直接在 EgentHub 對話框上傳對應圖片＋輸入文字驗測。

測試 #1：標準螺絲圖面（教學示範用）
上傳工作坊範例圖：`testscrew-1.png`（Penant 國宏「螺絲規格測量數據」M8L25 範例）
請從這張螺絲圖面擷取所有可辨識的規格，整理成規格表。

預期辨識重點：

  • 圖號：testscrew-1，標題：螺絲規格測量數據，by Penant 國宏
  • 主規格 M = 8 mm（螺牙直徑）、長度 L1 = 25 mm、頭寬 = 13 mm
  • L2（牙長）、P（牙距）、H（頭高）為代號標示但無具體數值
  • 圖面註記「以 M8L25 標準頭為例」
測試 #2：自然語言追問
這顆螺絲如果要寫進採購單，品名要怎麼命名比較標準？

預期輸出：建議「M8L25 內六角螺絲」或「M8 x 25mm 標準頭螺絲」，並提示 L2 / P / H 未標示需確認。

測試 #3：規格代號意義詢問
圖中的 P (Pitch) 和 L2 是什麼意思？

預期輸出：P 代表牙距、L2 代表牙長（螺紋部分長度），並提醒此圖未標示具體數值。

測試 #4：模糊或不完整圖面（邊界測試）
上傳一張刻意縮小、文字模糊的圖
這張圖的規格幫我擷取一下。

預期輸出：婉拒並要求重新上傳高解析度圖片，不捏造任何數值。

正式上線前，請依序確認以下幾個重點：

1. 資料準備　收集知識庫原始檔案，確認版本與權威來源，由業務單位主導、IT 協助匯出

2. 知識庫建置　上傳到 EgentHub，建立向量／表格索引，標註欄位定義

3. Agent 設定　貼入 Prompt，連結知識庫，選擇模型（建議首選 Claude Sonnet 4，成本敏感任務可測試 Gemini）

4. 測試驗證　執行本頁「驗測情境」所列題目，量測命中率（目標 ≥ 80%）

5. 教育訓練　內部宣導，讓使用者知道：可以問什麼、不能問什麼、結果如何驗證

6. 正式上線與校正　收集使用者回饋，定期檢視知識庫更新與 Prompt 微調

常見風險

  • 知識庫資料過時或欠缺版次標註，會直接拉低命中率
  • 使用者習慣未建立，上線後使用率容易不到 30%
  • 缺乏專人維護，Agent 會隨業務變動而失準
