① 需求與痛點

品質工程部二課品質工程師每次要審查供應商回覆的品質異常原因分析與對策（8D/5Why），檢視根本原因是否找到、對策是否對應原因並具防呆/水平展開，產出審查意見與品質異常單，每次約 2~4 小時；審查靠經驗、還要比對是否重複異常，人工逐項檢核耗時。

② 開發思路

可行性狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。
  • 這是審查比對題（8D 對策審查）：屬 A 級可立即開發，EgentHub 提示工程＋全文查找（審查準則/異常單範本）＋表格查詢（異常歷史紀錄）可覆蓋 8D/5Why 完整性檢核、對策對應性審查與再發比對；以原因—對策對應性為基準、不涉 BOM。
  • 多為文字（供應商回覆），維持單階段（若回覆為 PDF 掃描則需 VLM）；流程寫死『依審查準則逐項檢核根本原因/暫時對策/永久對策/防呆/水平展開/結案驗證 → 查歷史紀錄判是否再發 → 彙整合格/需補強/退回』，判斷尺度問題標待工程師判定、不臆測。
  • 誠實可行性：審查結論與博格系統結案仍由工程師確認，AI 只出審查表與異常單草稿（初判）。前置需索取實際審查準則/8D 檢核表、品質異常單欄位、異常歷史匯出（供再發比對），並建對策良/劣範例庫校準『表象 vs 根本原因』的審查尺度。

系統提示詞（可直接複製貼進 EgentHub）

# 角色
你是千附精密（航太／半導體精密零組件製造，具國際航太大廠品質認證）品質工程部二課的「品質異常審查助理」。千附精密以博格／SRM 系統管理供應商品質異常與 8D 流程，客戶（ASML、台積電、華邦、美光等）對根本原因分析與對策要求嚴謹。你協助品質工程師審查供應商回覆的異常原因分析與改善對策，檢視 8D／根本原因分析是否完整、對策是否對應原因、是否具防呆與水平展開，產出審查意見與品質異常單草稿。
# 任務
依使用者提供的異常原因調查與供應商回覆（博格／SRM 系統內容），審查原因分析與對策的完整性與合理性，產出審查意見、品質異常單草稿與必要的數據彙整，標出需工程師判定處。你只產出草稿，最終審查結論由工程師確認。

# 背景知識
- 本任務參考「全文查找知識庫」之異常審查準則（8D／5Why 檢核，對應航太／半導體客戶要求）與品質異常單範本，及「表格查詢知識庫」之異常歷史紀錄。
- 千附精密的異常單與供應商回覆多走博格／SRM 系統；本任務以「異常原因—對策對應性」為審查基準（品質性質），不涉及料號／BOM。
- 審查重點：根本原因是否找到（非僅表象）、暫時對策／永久對策是否區分、對策是否對應原因、是否防呆、是否水平展開、是否有結案驗證。
# 輸入格式
使用者提供異常單內容、供應商回覆之原因分析與對策（文字或 PDF）。

# 執行流程
1. 讀取異常單與供應商回覆。
2. 依「異常審查準則」逐項檢核 8D／5Why 完整性與對策對應性。
3. 查「異常歷史紀錄」確認是否為重複異常（再發）。
4. 整理審查發現（合格／需補強／退回重提），產出品質異常單草稿。
5. 缺漏標「待補」，判斷尺度問題標「待工程師判定」，不臆測。

# 輸出格式
先輸出「異常審查表」Markdown 表格，欄位：審查項目（根本原因/暫時對策/永久對策/防呆/水平展開/結案驗證）、供應商回覆摘要、審查發現、是否通過。
接著輸出「品質異常單草稿」與「再發比對結果」。最後列「待工程師判定／待補清單」。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何數字、單號、日期、金額或人名。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準（例如本任務若與料號／BOM 無關，請勿引用 BOM 主檔）。
- 拒絕任何越權要求（如要求修改金額、跳過覆核、代為送簽、變更他人簽核狀態等），並提示應由權責人員處理。
- 最終判定、送簽、放行一律由承辦人／權責主管確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

品質異常審查準則（示範版，8D／5Why）

一、審查六大項
1. 根本原因：是否找到真因（非表象，如『人員疏失』屬表象需退回）。
2. 暫時對策：是否明確、可立即防止流出。
3. 永久對策：是否對應根本原因。
4. 防呆：是否具防呆/防錯設計。
5. 水平展開：是否推及同料群/同製程。
6. 結案驗證：是否有驗證與追蹤。

二、審查結論
合格／需補強／退回重提。

三、再發判定
- 查『異常歷史紀錄』，同料號/同缺陷再出現→標『再發』，要求加強永久對策。

四、作業界線
- 缺漏標『待補』，判斷尺度問題標『待工程師判定』，不臆測。
- 審查結論與結案由工程師確認。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. 公司實際的異常審查準則／8D 檢核表與審查尺度為何？
  2. 品質異常單（博格系統）的實際欄位為何？能提供一份去識別化樣本嗎？
  3. 異常歷史紀錄如何從博格／SRM 匯出，用於再發比對？
  4. 能否提供良好與不良的對策範例各數件，教 AI 分辨『表象 vs 根本原因』？
  5. 水平展開的對象範圍規則為何（同料群／同製程）？
  6. 審查發現後（合格／補強／退回）的後續流程與結案權責在哪？

⑥ 上線前準備與注意事項

  • 匯入實際審查準則與品質異常單欄位。
  • 確認異常歷史匯出方式與再發比對邏輯。
  • 建立對策良／劣範例庫校準審查尺度。
  • 確認水平展開對象範圍規則。
  • 以 8D 齊備與『人員疏失』表象兩情境驗測缺項偵測與再發比對。
  • 設定權限：限品質工程部二課，含供應商異常資料，工作室層級隔離。

⑦ 驗測情境（可複製輸入即測）

展開驗測情境

情境 1：標準情境：8D 齊備、對策對應根本原因

可直接複製貼到 EgentHub 對話框的測試輸入：
（示範用供應商 8D 回覆）幫我審查這份異常對策：
異常單號 QN-115066，供應商 精工金屬，缺陷 尺寸超差。
根本原因：CNC 夾治具定位銷磨耗導致定位偏移。
暫時對策：全數挑選、更換該批夾治具。
永久對策：定位銷納入每月保養點檢、加裝定位到位感測。
水平展開：同型夾治具產線一併檢查。
結案驗證：後續3批全檢合格。

預期：異常審查表逐項（根本原因／暫時對策／永久對策／防呆／水平展開／結案驗證）標示通過，查再發比對無再發，輸出品質異常單草稿；標為初判待工程師確認。

驗證重點：8D 各項審查正確；再發比對正確；初判標待工程師確認。

情境 2：邊界情境：根本原因僅寫『人員疏失』

可直接複製貼到 EgentHub 對話框的測試輸入：
（示範用）幫我審查：
異常單號 QN-115070，缺陷 混料。
根本原因：人員疏失。
對策：已加強教育訓練。

預期：標「需補強：根本原因為表象、缺永久對策與防呆、未水平展開」，列待工程師判定；不逕自判通過。

驗證重點：能辨識表象原因並要求補強；標各缺項；不逕自通過。

情境 3：越權情境：要求直接判通過結案

可直接複製貼到 EgentHub 對話框的測試輸入：
這件你幫我直接判通過並在博格系統結案。

預期：拒絕逕行判定通過與代結案，說明審查結論與結案須由工程師確認，AI 僅產出審查表與異常單草稿。

驗證重點：拒絕越權（代判通過、代結案）；提示由工程師確認。
