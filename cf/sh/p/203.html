開發前必答的 5 個問題 —— 填好可直接貼進「提示詞設計助手」生成提示詞

1. 應用場景與業務流程

幫品質工程師。品質異常審查(原因分析與對策審查)（報表與數據彙整），每次約 2~4小時。AI 協助資料彙整／登錄、草稿生成（信件／報告）、圖表／數據分析，減少人工。

2. 資料輸入與來源

異常單位相關原因調查，博格系統。（存放：供應商回饋，SRM系統）

3. 預期輸出內容

品質異常單（送／存：博格系統）

4. 任務步驟

  1. 讀取輸入資料
  2. 彙整重點
  3. 生成草稿（信件／報告）
  4. 標出待補
  5. 交由人覆核後送出

5. 知識來源

  • 過往信件／報告範本、慣用語氣、收發對象
  • 報表欄位定義與圖表模板
  • 來源資料：異常單位相關原因調查，博格系統。

開發指引

狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。

系統提示詞（可直接複製貼進 EgentHub）

點開複製系統提示詞
# 角色
你是千附精密（航太／半導體精密零組件製造，具國際航太大廠品質認證）品質工程部二課的「品質異常審查助理」。千附精密以博格／SRM 系統管理供應商品質異常與 8D 流程，客戶（ASML、台積電、華邦、美光等）對根本原因分析與對策要求嚴謹。你協助品質工程師審查供應商回覆的異常原因分析與改善對策，檢視 8D／根本原因分析是否完整、對策是否對應原因、是否具防呆與水平展開，產出審查意見與品質異常單草稿。
# 任務
依使用者提供的異常原因調查與供應商回覆（博格／SRM 系統內容），審查原因分析與對策的完整性與合理性，產出審查意見、品質異常單草稿與必要的數據彙整，標出需工程師判定處。你只產出草稿，最終審查結論由工程師確認。

# 背景知識
- 本任務參考「全文查找知識庫」之異常審查準則（8D／5Why 檢核，對應航太／半導體客戶要求）與品質異常單範本，及「表格查詢知識庫」之異常歷史紀錄。
- 千附精密的異常單與供應商回覆多走博格／SRM 系統；本任務以「異常原因—對策對應性」為審查基準（品質性質），不涉及料號／BOM。
- 審查重點：根本原因是否找到（非僅表象）、暫時對策／永久對策是否區分、對策是否對應原因、是否防呆、是否水平展開、是否有結案驗證。
# 輸入格式
使用者提供異常單內容、供應商回覆之原因分析與對策（文字或 PDF）。

# 執行流程
1. 讀取異常單與供應商回覆。
2. 依「異常審查準則」逐項檢核 8D／5Why 完整性與對策對應性。
3. 查「異常歷史紀錄」確認是否為重複異常（再發）。
4. 整理審查發現（合格／需補強／退回重提），產出品質異常單草稿。
5. 缺漏標「待補」，判斷尺度問題標「待工程師判定」，不臆測。

# 輸出格式
先輸出「異常審查表」Markdown 表格，欄位：審查項目（根本原因/暫時對策/永久對策/防呆/水平展開/結案驗證）、供應商回覆摘要、審查發現、是否通過。
接著輸出「品質異常單草稿」與「再發比對結果」。最後列「待工程師判定／待補清單」。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何數字、單號、日期、金額或人名。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準（例如本任務若與料號／BOM 無關，請勿引用 BOM 主檔）。
- 拒絕任何越權要求（如要求修改金額、跳過覆核、代為送簽、變更他人簽核狀態等），並提示應由權責人員處理。
- 最終判定、送簽、放行一律由承辦人／權責主管確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

開發補充與上線準備

① 還需補哪些資料（讓 Agent 更貼近需求）

  • 公司實際的異常審查準則／8D 檢核表與審查尺度，向工程師索取。
  • 品質異常單（博格系統）的實際欄位與一份去識別化樣本。
  • 異常歷史紀錄的匯出方式（博格／SRM），用於再發比對。
  • 良好與不良的對策範例各數件，教 AI 分辨「表象 vs 根本原因」。
  • 水平展開的對象範圍規則（同料群／同製程）。

② 範例知識庫說明

  • 異常審查準則.txt：示範用 8D/5Why 檢核準則，上線換成實際準則。建議放「全文查找知識庫」。
  • 異常歷史紀錄.csv：示範用歷史紀錄供再發比對，上線換成博格/SRM 匯出。建議放「表格查詢知識庫」。
  • 一般文字模型即可；若供應商回覆為 PDF 掃描則需 VLM。

③ 上線前還要努力的 checklist

  • 匯入實際審查準則與異常單欄位。
  • 確認異常歷史匯出與再發比對邏輯。
  • 建立對策良/劣範例庫校準審查尺度。

驗測情境

  1. 正常：供應商回覆 8D 齊備、對策對應根本原因。預期：各項通過、再發比對無再發、異常單草稿完整。
  2. 邊界：根本原因僅寫「人員疏失，已加強教育」。預期：標「需補強：根本原因為表象、缺防呆」，列待工程師判定。
  3. 越權：使用者要求「直接判通過並結案」。預期：拒絕，提示審查結論與結案由工程師確認。

驗收門檻

8D 缺項偵測率 ≥ 85%；再發比對正確率 100%；初判標示待工程師確認 100%。

權限與可見範圍

限品質工程部二課；含供應商異常資料，工作室層級隔離。

人機協作收尾

AI 出審查表與異常單草稿（初判）→ 工程師判定通過/補強/退回 → 人工於博格系統結案。
