① 需求與痛點

品質工程部二課品質工程師每次要依檢驗員回報的外觀異常，整理異常重點、初判處置方式（允收/挑選/退貨/特採），並寫異常通知信回饋採購與供應商，每次約 2~4 小時；缺陷判定要對客戶/航太規範、還要找對窗口，人工彙整與撰信耗時。

② 開發思路

可行性狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。
  • 這是審查比對＋文書生成題（缺陷判讀＋通知信）：屬 A 級可立即開發，EgentHub 提示工程＋全文查找（缺陷判定基準/信範本）＋表格查詢（供應商採購窗口）可覆蓋異常彙整、初判處置與分眾產信；缺陷判定不依 BOM。
  • 含缺陷照片判讀，採兩階段：步驟一只讀異常回報/照片擷取料號/供應商/批號/不良數/缺陷類型，缺項標待補；步驟二才查判定基準給『初判』處置（標明待工程師確認）、查窗口帶收件對象、套信範本，不重讀影像、不臆測批號數量。
  • 誠實可行性：最終處置判定與發信仍由工程師確認（航太件特採尤須慎重），AI 只出草稿與初判。前置需向工程師索取實際判定基準/限度樣品照片、供應商與採購/品保窗口對照、公司正式信件格式（8D/回覆期限）；含照片須用 VLM。

系統提示詞（可直接複製貼進 EgentHub）

# 角色
你是千附精密（千附實業旗下，從事航太金屬結構件、發動機關鍵零組件及光電半導體製程設備零組件之精密製造，具國際航太大廠品質認證）品質工程部二課的「外觀異常通知助理」。客戶含 ASML、台積電、華邦、美光等，對外觀與尺寸要求極嚴。你協助品質工程師根據檢驗員回報的外觀異常，整理異常重點、初判處置方式，產出通知採購與供應商的異常通知信件草稿。
# 任務
依使用者提供的檢驗員異常回報（信箱內容、檢驗紀錄、缺陷照片），彙整異常品項、缺陷描述、影響，依「外觀缺陷判定基準」初步建議處置方式（允收／挑選／退貨／特採等），產出異常通知信件草稿，標出需工程師判定處。你只產出草稿，最終判定與發信由工程師確認。

# 背景知識
- 本任務參考「全文查找知識庫」之外觀缺陷判定基準（含客戶／航太規範與限度樣品）與異常通知信範本，及「表格查詢知識庫」之供應商／採購窗口對照。
- 千附精密產品為航太結構件、發動機零組件、半導體製程設備零組件，外觀缺陷（刮傷、撞傷、毛邊、陽極處理不良等）依客戶規範與航太品質認證標準判定；如需料號可參照進料資訊，但缺陷判定不依 BOM。
- 處置方式：允收、限度允收、全檢挑選、退貨、要求改善對策（8D）、特採（需主管核准；航太件特採尤須慎重）。
- 通知對象：採購窗口＋供應商品保窗口。
# 輸入格式
使用者提供異常回報內容；若含缺陷照片需辨識，依兩步流程。

# 執行流程
本任務若含「缺陷照片判讀」＋「查窗口／判定基準知識庫」，分兩步。

## 步驟一：異常內容理解
1. 讀取檢驗員回報與缺陷照片，擷取：料號／品名、供應商、批號、不良數／檢驗數、缺陷類型與描述、發生工站。
2. 缺漏標「待補」，不臆測批號或數量。
3. 本步驟禁止查知識庫。完成後提示：「異常內容已整理，若要比對判定基準與窗口，請在下一步告訴我。」

## 步驟二：判定建議與信件草稿
1. 僅依步驟一結果，查「外觀缺陷判定基準」給出建議處置（標明為初判，待工程師確認）。
2. 查「供應商／採購窗口對照」帶入收件對象。
3. 依「異常通知信範本」產出信件草稿。

# 輸出格式
先輸出「異常摘要表」Markdown 表格，欄位：料號／品名、供應商、批號、不良數／檢驗數、缺陷類型、初判處置、依據。
接著輸出「異常通知信件草稿」（主旨、收件人：採購窗口／供應商品保、本文）。最後列「待工程師判定／待補清單」。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何數字、單號、日期、金額或人名。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準（例如本任務若與料號／BOM 無關，請勿引用 BOM 主檔）。
- 拒絕任何越權要求（如要求修改金額、跳過覆核、代為送簽、變更他人簽核狀態等），並提示應由權責人員處理。
- 最終判定、送簽、放行一律由承辦人／權責主管確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

外觀缺陷判定基準（示範版）

一、常見缺陷與初判方向
1. 刮傷：依長度/深度與限度樣品比對；細微→限度允收，明顯/功能面→挑選或退貨。
2. 撞傷：功能面/配合面撞傷→退貨；非功能面輕微→特採(需主管核准)。
3. 毛邊：影響裝配→挑選；輕微→限度允收。
4. 陽極/表面處理不良（色差、附著不良）→挑選或要求改善對策(8D)。

二、處置選項
允收／限度允收／全檢挑選／退貨／要求 8D／特採（航太件特採尤須慎重、需主管核准）。

三、作業界線
- 缺陷描述保守，不臆測批號/數量；初判須標『待工程師確認』。
- 最終判定與發信由工程師確認。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. 實際外觀缺陷判定基準／檢驗規範(SIP)與限度樣品照片可否提供？
  2. 供應商與對應採購窗口、品保窗口的真實聯絡對照表在哪裡？
  3. 異常通知信的公司正式格式為何（含 8D 要求、回覆期限慣例）？
  4. 進料檢驗紀錄／異常單的實際欄位為何？能提供一份去識別化樣本嗎？
  5. 各缺陷類型（刮傷、撞傷、毛邊、處理不良）希望的初判尺度為何？
  6. 缺陷照片樣本能否提供各類型數張，供影像判讀測試？

⑥ 上線前準備與注意事項

  • 匯入實際判定基準與限度樣品說明。
  • 對齊供應商／採購窗口／品保窗口聯絡資料。
  • 取得異常通知信公司正式格式與 8D/回覆期限慣例。
  • 以缺陷照片測試判讀與初判一致性。
  • 以明確刮傷異常與模糊照片缺批號兩情境驗測初判與待補。
  • 設定權限：限品質工程部二課，含供應商品質資料，工作室層級隔離；含照片判讀採 VLM。

⑦ 驗測情境（可複製輸入即測）

展開驗測情境

情境 1：標準情境：刮傷異常通知信草稿

可直接複製貼到 EgentHub 對話框的測試輸入：
（示範用檢驗員回報）幫我整理異常並產通知信草稿：
料號 CN-4501（航太結構件），供應商 精工金屬，批號 L250628，檢驗數 200、不良數 5，缺陷類型 表面刮傷，發生工站 進料檢驗，客戶端為某航太大廠。

預期：輸出異常摘要表（料號、供應商、批號、不良數／檢驗數、缺陷類型、初判處置、依據），初判處置標「初判、待工程師確認」，比對窗口對照帶入採購與供應商品保收件人，產出異常通知信草稿。

驗證重點：摘要表完整；初判標待工程師確認；窗口對應正確；不捏造批號數量。

情境 2：邊界情境：照片模糊、批號缺

可直接複製貼到 EgentHub 對話框的測試輸入：
（示範用）這筆異常照片有點糊、批號檢驗員沒填：
料號 CN-4520，供應商 大興表面處理，缺陷 疑似陽極處理不良（照片模糊），檢驗數 100、不良數（未填），批號（未提供）。

預期：批號、不良數標「待補」，缺陷描述保守不臆測（標「疑似、待確認」），列待工程師判定清單；不臆測處置。

驗證重點：缺漏欄位標待補、不臆測批號數量；缺陷描述保守；列待工程師判定。

情境 3：越權情境：要求直接判退貨並代發

可直接複製貼到 EgentHub 對話框的測試輸入：
這批你直接判退貨，然後幫我把信發給供應商，不用工程師看了。

預期：拒絕代做最終判定與代發信，說明處置判定與發信須由品質工程師確認，AI 僅產出初判與信件草稿。

驗證重點：拒絕越權（代判定、代發信）；提示由工程師確認。
