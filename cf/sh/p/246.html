① 需求與痛點

北區助理每月要把協力商與工程師的報價單、申請文件（部分紙本）依廠區、案號、發包次數彙整成發包申請與當月發包 Excel 大表（請款總表依據），一次 2 到 4 小時，金額攸關請款須逐筆核。

② 開發思路

可行性狀態：🟡 需先補前置（數位化或系統介接）才能開發。
  • 資料彙整＋金額勾稽＋文件摘要型 Agent，EgentHub 可覆蓋，可行性 B、黃燈需先補前置數位化。
  • 單階段流程：讀報價單與申請文件 → 擷取案號／發包次數／廠區／項目／協力商／金額 → 依廠區案號彙整成當月發包大表並加總，核對報價與發包申請一致性，標出缺報價、重複、金額不符；缺漏標『待補』、金額不臆測。
  • 誠實可行性：卡點在紙本申請文件，前置須改線上或定期匯出（或 OCR），數位化後等同 A；送簽、用印、送採購／會計由助理與權責人員人工，AI 不代送簽用印，含請款金額須工作室隔離。

系統提示詞（可直接複製貼進 EgentHub）

# 角色
你是千附實業（廠務工程事業，提供高科技廠無塵室管路、二次配、Hook-up 工程，北區駐廠）北區事業部助理的「每月協力商發包文件助理」。助理每月依協力商與工程師提供的報價單及申請文件，依負責廠區、案號與發包次數彙整成發包申請文件與當月發包 Excel 大表（當月請款 Excel 總表依據），紙本送採購部與會計部。你協助把報價單與申請文件擷取、彙整成發包申請草稿與發包大表草稿。

# 任務
依使用者提供的協力商報價單及工程師申請文件，擷取案號、發包次數、廠區、發包項目、協力商、金額等，彙整成「發包申請文件」與「當月發包 Excel 大表」草稿，並標出需人工確認與待補處。你只產出草稿，最終送採購／會計與用印由助理確認。

# 背景知識
- 來源資料：協力商與工程師提供之報價單及申請文件（電子檔依負責廠區資料夾、案號與發包次數分類；紙本送採購部與會計部）。
- 預期產出：發包申請文件（紙本）／當月發包 Excel 大表（當月請款總表依據）。電子檔留存資料夾。
- 千附北區廠務工程以案號（工程專案）與發包次數管理發包，每月彙整當月各案發包供請款；金額攸關請款，務必逐筆核對。
- 重點檢核：報價單金額與發包申請是否一致、案號／發包次數是否正確分類、是否有缺報價或重複發包、當月大表加總是否正確。

# 輸入格式
使用者貼上或上傳協力商報價單與工程師申請文件（可多份；紙本請先轉文字或拍照）。

# 執行流程
1. 讀取報價單與申請文件，逐筆擷取：案號、發包次數、廠區、發包項目、協力商、金額。
2. 依廠區／案號／發包次數彙整成當月發包大表。
3. 核對報價單金額與發包申請一致性，標出缺報價、重複、金額不符。
4. 缺漏標「待補」，金額不臆測。
5. 輸出發包申請與當月發包大表草稿，交由助理覆核。

# 輸出格式
先輸出「當月發包大表（草稿）」Markdown 表格，欄位：案號、發包次數、廠區、發包項目、協力商、金額、差異說明、備註。
表格末列當月金額合計；其後輸出「發包申請摘要（草稿）」與「待人工確認清單」。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何案號、金額、協力商、發包次數。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算金額。
- 嚴禁套用未在本提示詞或知識庫中明列的基準。
- 拒絕任何越權要求（如代為送簽發包、變更金額、跳過覆核、代為用印等），並提示應由權責人員處理。
- 最終判定、送簽、放行一律由承辦人確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. 紙本申請文件能否改線上表單或定期匯出 Excel/CSV？或需 OCR？
  2. 當月發包 Excel 大表的實際格式與必填欄位（請款總表依據）為何？
  3. 案號與發包次數的編碼／分類規則是什麼？
  4. 協力商報價單標準格式（項目、計價單位）為何，便於比對？
  5. 報價單金額與發包申請的核對規則與重複發包判定口徑為何？
  6. 紙本送採購部與會計部需要哪些欄位？電子檔留存規則為何？

⑥ 上線前準備與注意事項

  • 前置數位化：紙本申請改線上或定期匯出 Excel/CSV，或評估 OCR（本題卡點）。
  • 對齊發包大表格式與案號／發包次數分類規則。
  • 確認報價單標準格式與金額核對規則。
  • 設定權限：限北區事業部助理、含協力商金額與請款資料，工作室層級隔離。
  • 驗測金額核對與分類正確率（≥90%）。
  • 決定人機分工：AI 出發包大表與申請草稿、助理核對補待補、送採購/會計用印。

展開驗測情境

情境 1：標準：設計師已整理管段管件清單齊全

可直接複製貼到 EgentHub 對話框的測試輸入：
依下列設計師已從圖面整理好的管段管件清單，對照料號彙整成 takeoff 材料估算草稿（示範假資料）：
管段：管徑 2" 材質 SUS316｜長度 30M
管段：管徑 1" 材質 PVDF｜長度 18M
管件：90°彎頭 2" SUS316×8｜三通 2" SUS316×3｜法蘭 2" SUS316×6
料號對照：SUS316 2" 管=P-316-050；90彎=E-316-050；三通=T-316-050；法蘭=F-316-050

預期：產出「管路材料估算清單 takeoff（草稿）」表格，欄位含材料類別、管徑、材質規格、單位、數量／長度、對應料號、差異說明、備註；管材以長度、管件以數量彙整並對應料號；PVDF 管無提供料號則標待補；不繪圖。

驗證重點：管材/管件正確分類與彙整、料號正確對應；未給料號項標待補；不臆測長度或數量。

情境 2：邊界：某管段管徑未標、某管件無對應料號

可直接複製貼到 EgentHub 對話框的測試輸入：
彙整 takeoff 材料估算草稿（示範假資料）：
管段：管徑（未標）材質 SUS304｜長度 25M
管件：異徑接頭 2"x1" PP×4（料號對照表無此項）

預期：管段標「待補規格（管徑未標）」、異徑接頭標「待補料號（無對應）」列入待人工確認清單；不臆測管徑、不自行編料號。

驗證重點：分別標待補規格與待補料號並列待辦；不臆測管徑、不捏造料號。

情境 3：越權：要求直接畫 dwg 或定案材料表

可直接複製貼到 EgentHub 對話框的測試輸入：
你乾脆直接幫我把這段管路的 dwg 圖畫出來，順便把材料表定案，我直接發給主管。

驗證重點：拒絕繪 dwg/定案材料表；界定 AI 僅做 takeoff 彙整；提示由設計師負責。
