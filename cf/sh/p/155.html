② 開發思路

可行性狀態：🔴 目前暫不建議用 EgentHub 開發。
  • 題型無法判定：作答四欄空白、覆蓋度無從評估、無法分級落點，本題維持 D1（紅燈）暫不建議用 EgentHub 開發。
  • 無流程可設計：因缺明確輸入輸出，暫不產出提示詞與驗測情境；先誠實標為待回訪釐清，不臆造需求。
  • 誠實可行性：本題誠實說明暫不建議，須回訪填答人（黃保誠）釐清實際每日工作與耗時；可切子任務方向為程式開發周邊的『文書性子任務』（規格／需求摘要、測試紀錄彙整、版本說明 Release Note 草稿、教育訓練文件整理），待新題目成形後再走開發前 5 問→提示詞→驗測流程，程式撰寫本身不納入。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. 軟體工程師『程式設計』每日實際工作內容與耗時分布為何？除了寫程式，還有哪些文書性工作？
  2. 有沒有可切出的『文書性子任務』——例如規格／需求文件摘要、測試紀錄彙整、版本說明（Release Note）草稿、教育訓練／操作文件整理？
  3. 這些文書子任務目前輸入是什麼、產出給誰、存放在哪？（原作答四欄皆空白，需補齊）
  4. 哪些內容涉及程式碼撰寫本身（EgentHub 不協助），哪些是程式開發周邊的文件處理（可切出開發）？
  5. 若切出文書子任務，其資料是文字為主還是含 PDF／圖片？需要辨識嗎？
  6. 回訪時能否請填答人（黃保誠）舉 1～2 個實際案例，讓我們判斷是否可另立題目開發？

⑥ 上線前準備與注意事項

  • 回訪填答人（黃保誠，軟體工程師）釐清實際每日工作內容、耗時與痛點。
  • 從中切出可由 AI 協助的『文書性子任務』（規格摘要／測試紀錄彙整／版本說明草稿等），另立新題目。
  • 確認切出的子任務不涉及程式撰寫本身（EgentHub 禁用範圍）。
  • 依子任務性質預備知識庫：文件摘要→全文查找知識庫；紀錄彙整→表格查詢知識庫。
  • 待新題目成形後，重走『開發前 5 問 → 提示詞 → 知識庫 → 驗測』流程。
  • 本題維持 D1，暫不開發提示詞與驗測，先以回訪釐清為優先。
