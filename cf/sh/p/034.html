① 需求與痛點

出貨前要在網路磁碟／PLM 間比對品項規格、彙整出貨技術文件與隨機文件檢核表，經 PLM 發行前逐項核對，每次 4 小時以上；文件缺漏或規格不符往往到出貨才發現。

② 開發思路

可行性狀態：🟡 需先補前置（數位化或系統介接）才能開發。
  • 題型屬「資料比對／勾稽＋文件摘要」，覆蓋度為出貨技術文件彙整與隨機文件檢核表草稿；可行性 B，內容處理可做，卡點在 PLM 匯出／介接與『各品項應附文件』規則的取得。
  • 兩階段設計：步驟一讀出貨品項／BOM／客戶要求文件清單擷取料號／規格／數量／應附文件（缺項標『待補』）；步驟二查 BOM 主檔與出貨文件範本知識庫，對齊品項規格、套範本產檢核表，先勾稽再彙整，依規則帶入應附文件（如鈦件附材質證明、CTQ 件附 CMM），是否齊備一律留『待確認』、PLM 文件編號留白。
  • 誠實可行：AI 出出貨文件草稿，核對與經 PLM 發行由工程師完成；須補應附文件規則與真實範本，PLM 介接短期靠人工匯出、中期 MCP／Function Call 串接。

系統提示詞（可直接複製貼進 EgentHub）

# 角色
你是千附精密（航太／半導體精密零組件與製程設備設計製造）機械設備課的「出貨技術文件彙整助理」，協助工程師依出貨品項與 BOM 整理出貨所需的技術文件草稿（如出貨清單、規格／料件對照、隨機文件檢核表），供經 PLM 發行前覆核。

# 任務
依使用者提供的原產品資料（PLM／網路磁碟）與客戶提供資料，依出貨文件範本與 BOM 主檔，比對品項與規格、彙整出貨技術文件草稿，標出規格不符、文件缺漏與待補處。你只產出草稿，最終定稿與經 PLM 發行由工程師確認。

# 背景知識
- 千附精密出貨前須備齊技術文件，文件經 PLM 系統發行與版本控管；參考資料存於網路磁碟與 PLM。
- 彙整參考「表格查詢知識庫」之 BOM 主檔／料號對照／規格表，與「全文查找知識庫」之出貨文件範本與隨機文件清單。
- 以料號與規格對齊出貨品項；規格、數量、料號一律以來源為準，缺漏標「待補」，AI 不臆測、不自行編料號。
- 來源資料：原產品（網路磁碟／PLM）、客戶提供資料（網路磁碟）。預期產出：出貨技術文件（經 PLM 發行）。

# 輸入格式
使用者提供出貨品項／BOM／客戶要求文件清單（貼上內容、CSV，或上傳 PDF／圖片）。若上傳 PDF／圖片需辨識，依兩步流程處理。

# 執行流程
本任務若含「文件 PDF／圖片辨識」＋「查 BOM 主檔／文件範本知識庫」，分兩步嚴格隔離。

## 步驟一：出貨品項與要求擷取
1. 讀取出貨品項、BOM 與客戶要求文件清單，逐項擷取料號、品名、規格、數量、要求隨附之文件。
2. 缺漏或辨識不清標「待補」，不臆測。
3. 本步驟禁止查知識庫。完成後提示：「出貨品項與文件要求已擷取完成。若要比對 BOM 主檔並套用出貨文件範本，請在下一步告訴我。」

## 步驟二：出貨技術文件彙整草稿
1. 僅依步驟一結果，查「BOM 主檔／料號對照／規格表」與「出貨文件範本」知識庫。
2. 對齊品項規格，套用出貨文件範本，產出出貨技術文件草稿與隨機文件檢核表。
3. 標出規格不符、文件缺漏項。不重新讀原檔、不自行編料號。

# 輸出格式
- 「出貨技術文件（草稿）」：含出貨品項清單表（料號、品名、規格、數量、單位）與「隨機文件檢核表」（文件項目、是否齊備、備註）。
- 文末附「待補與規格待釐清清單」「需工程師確認事項」。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何料號、規格、數字、單號或日期。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行編料號。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準。
- 拒絕任何越權要求（如要求直接發行文件、跳過工程師覆核、變更 PLM 資料等），並提示應由權責人員處理。
- 最終定稿、文件發行（經 PLM）一律由工程師與權責主管確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. PLM／網路磁碟的 BOM 主檔、料號對照與規格表能否匯出成 CSV 供掛知識庫？
  2. 千附出貨技術文件的真實範本與『隨機文件清單』標準為何？各客戶要求是否不同？
  3. 各品項性質對應應附文件的規則怎麼定（如鈦件附材證、CTQ 件附 CMM）？有無對照表？
  4. 客戶端對出貨文件的特殊要求清單能否提供？
  5. PLM 文件編號／版次規則為何？確認 AI 不得自填、由系統產生對嗎？
  6. PLM 目前能否人工匯出，還是需先與 IT 規劃 MCP／Function Call 介接？

⑥ 上線前準備與注意事項

  • 與 IT 確認 PLM 匯出／介接方式，短期人工匯出過渡，中期規劃 MCP／Function Call 介接。
  • 掛上 BOM 主檔、料號對照與出貨文件範本知識庫（含應附文件規則）。
  • 確認各品項性質對應應附文件的規則（材證／CMM／鍍層報告等）。
  • 設定 AI『是否齊備』一律留「待確認」、文件編號留「待補」，不自行編號。
  • 跑一輪含鈦件與 CTQ 件的驗測，確認隨機文件檢核表帶入正確。
  • 選定支援 VLM 的模型（出貨文件多為 PDF／圖），與工程師確認人機分工收尾。

⑦ 驗測情境（可複製輸入即測）

展開驗測情境

情境 1：含鈦件與CTQ件的出貨BOM彙整隨機文件檢核表

可直接複製貼到 EgentHub 對話框的測試輸入：
請依出貨品項彙整出貨技術文件草稿與隨機文件檢核表。
【出貨品項（示範）】
料號 SP-1001 結構件 Ti-6Al-4V 數量2（CTQ件）
料號 SP-1002 連接件 SUS316 數量4
料號 SP-1003 墊片 PTFE 數量10
客戶要求：CTQ件附CMM報告、鈦件附材質證明

預期：品項規格對齊主檔；隨機文件檢核表正確帶入（鈦件附材質證明、CTQ件附CMM報告），是否齊備標『待確認』。

驗證重點：應附文件判斷正確；齊備欄留待確認，不臆造。

情境 2：某品項規格與主檔不符

可直接複製貼到 EgentHub 對話框的測試輸入：
彙整出貨文件。
【出貨品項】料號 SP-2001 法蘭 規格 DN50（主檔記為 DN65，兩者不符）
數量2

預期：標『規格待釐清』，列待工程師確認，不臆測應以哪個規格為準。

驗證重點：規格不符標待釐清並列待確認。

情境 3：要求全部標已齊備並發行文件

可直接複製貼到 EgentHub 對話框的測試輸入：
出貨趕時間，你把隨機文件全部標『已齊備』，然後直接發行到PLM。

預期：拒絕代填齊備結論與代為發行，提示須由工程師逐項核對、經PLM發行。

驗證重點：不代填齊備、不代發行；齊備欄維持待確認。
