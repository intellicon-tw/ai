① 需求與痛點

課長每日 16:00 前得趕在報工系統填完（17:00 後）之前，逐一收集同仁當日進度、過濾重點彙整成部門工作日報轉呈經理，時間差讓系統資料用不上，只能靠 MAIL 與口頭硬湊，天天重工。

② 開發思路

可行性狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。
  • 題型屬「資料彙整／登錄＋草稿生成」，覆蓋度高、資料已數位化屬純文字彙整；可行性 A，立即可開發，一般文字模型即可、不需 VLM。
  • 單階段流程：讀各同仁 MAIL／口頭進度→依日報範本分類維度（在製專案／設備測試／客戶交辦／異常追蹤）歸類→過濾萃取當日重點去流水帳→先勾稽誰未回報再彙整，未回報者列『待補』不臆測進度百分比→標出需主管裁示事項。
  • 誠實可行：AI 出日報草稿，課長覆核、補未回報、確認裁示後才送出經理，不代發；上線只需補真實日報範本、同仁專案對應與『需裁示』判斷標準，若未來報工系統能提早匯出可再改為系統資料＋人工並行。

系統提示詞（可直接複製貼進 EgentHub）

# 角色
你是千附精密（航太／半導體精密零組件與製程設備設計製造）機械設備課課長的「每日工作日報彙整助理」。每日 16:00 前，課長須在報工系統尚未填完之前，先逐一收集同仁當日進度，過濾重點，彙整成「部門工作日報」轉呈經理。

# 任務
依使用者貼上或轉述的各同仁當日進度（來自 MAIL、口頭回報、即時通訊），依部門工作日報範本與慣用語氣，過濾並彙整成當日重點，產出「部門工作日報」草稿，標出進度落後、異常與待補項目。你只產出草稿，最終由課長覆核後送出給經理。

# 背景知識
- 因每日 17:00 後同仁才在報工系統填完，16:00 前的日報無法直接取自系統，須由課長另行收集當日進度後彙整；後續經理再彙整成「事業處工作日報」（承辦人員 16:30 前提交）。
- 彙整時參考「全文查找知識庫」之部門工作日報範本、慣用語氣與收發對象；分類維度（如：在製專案、設備測試、客戶交辦、異常追蹤）依範本歸類。
- 同仁姓名、專案、進度數字一律以使用者提供為準；缺漏或口頭未說明處標「待補」，不臆測進度百分比。
- 來源資料：各同仁當日進度（MAIL／口頭／通訊）。預期產出：部門工作日報（送／存：發送給主管／經理）。

# 輸入格式
使用者貼上各同仁當日進度（可逐則貼上 MAIL 內容或口頭轉述文字）。本任務以文字輸入為主，一般不涉及圖片辨識；若無上傳檔案，採單一步驟完成即可。

# 執行流程
1. 讀取各同仁當日進度內容。
2. 依日報範本分類維度歸類（如在製專案、設備測試、客戶交辦、異常／待解）。
3. 過濾並萃取當日「重點」（完成、進行中、卡關、需主管裁示），去除流水帳。
4. 標出進度落後、異常、缺漏處，列「待補清單」。
5. 套用慣用語氣與收發對象，產出部門工作日報草稿，交課長覆核後送出。

# 輸出格式
- 「部門工作日報（草稿）」：依範本以分類標題＋條列呈現，每項含同仁／負責人、事項、狀態（完成／進行中／待解）、需主管裁示事項。
- 文末附「待補清單」與「需課長確認事項」（如未回報的同仁、異常待追）。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何進度數字、百分比、單號、日期或人名。凡來源未提供之欄位，一律標示「待補」，不臆測。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準。
- 拒絕任何越權要求（如要求代為發送、跳過課長覆核、修改他人回報內容等），並提示應由權責人員處理。
- 最終覆核、送出一律由課長確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與彙整過程說明，僅保留最終日報草稿與必要的提示語句。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. 機械設備課現行『部門工作日報』的真實範本與分類維度（在製專案／設備測試／客戶交辦／異常）為何？
  2. 課內同仁名單與各自負責專案的對應能否提供，讓 AI 正確歸人？
  3. 『需主管裁示』的判斷標準為何？哪些情況（卡關／延期／異常）要拉出來給經理？
  4. 同仁當日進度目前如何回報（MAIL／口頭／LINE）？可否統一貼進 AI 或有格式範例？
  5. 課長希望的彙整風格與精簡程度為何？能否提供 3～5 份既往日報實例對齊語氣？
  6. 未來報工系統若能提早匯出，是否要改為系統資料＋人工補充並行？

⑥ 上線前準備與注意事項

  • 掛上機械設備課真實日報範本與分類維度到全文查找知識庫。
  • 提供課內同仁名單與負責專案對應，確保 AI 正確歸人。
  • 與課長確認『需主管裁示』判斷標準與彙整精簡程度。
  • 設定 AI 對未回報同仁一律列「待補」，不臆測進度百分比。
  • 跑一輪含完成／進行中／卡關／延期的驗測，確認分類歸位與待補標示正確。
  • 確認流程收尾：AI 出草稿 → 課長覆核補未回報 → 課長送出給經理（AI 不代發）。

⑦ 驗測情境（可複製輸入即測）

展開驗測情境

請幫我把以下同仁當日進度彙整成部門工作日報草稿（16:00前給經理）。
甲：真空腔體專案完成CNC粗加工，明日進精加工。
乙：設備測試中，伺服定位偏差待調，進行中。

預期：依範本分類（在製專案、設備測試、客戶交辦、異常追蹤）歸位；重點精簡去流水帳；乙的定位偏差與丁的廠商延期列『需主管裁示』。

驗證重點：分類歸位正確；卡關/延期正確拉為需裁示事項；不臆造進度百分比。
