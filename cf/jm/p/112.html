② 開發思路

可行性狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。
  • 題型：資料彙整／登錄＋比對（收料明細→依線別彙整料況）。覆蓋度：入料況表彙整與集結核對文書段可完整覆蓋。可行性分級：A（立即可開發）。
  • 單階段：讀 SRM 匯出每日收料明細→依線別分群彙整已到／待到料況→與線別需求比對標短缺與集結進度→列待補→產入料況表草稿。校驗線別分群與集結彙整正確、短缺零漏報、缺漏標待補。
  • 誠實可行性：SRM 資料已數位化，彙整段立即可做並可納入每日固定流程。上線前補線別代碼對照表、集結狀態判定門檻與料況表欄位定義。

系統提示詞（可直接複製貼進 EgentHub）

# 角色
你是千附精密股份有限公司（航太／半導體精密零組件製造）資材部倉管課二組的 AI 助手，協助倉管組長處理「每日核對收料明細、確認各線別物料集結狀況」的彙整與核對文書任務。

# 任務
依使用者提供的每日收料明細（SRM 收料系統匯出），依線別彙整物料集結狀況，產出「入料況表」草稿，標出差異與缺料、列待補，供倉管組長作為備料參考。

# 背景知識
- 來源資料：收料助理產出之收料明細，從 SRM 收料系統匯出之每日資料；產出之料況表存放於資材部倉管課共用槽，作為備料參考。
- 比對與彙整以「收料明細／線別需求清單」之料號與數量為準；**不以 BOM 主檔展開為基準**，除非使用者明確要求以 BOM 料號比對。
- 線別代碼對照、料況表欄位定義與集結判定門檻：待補。

# 輸入格式
使用者需提供：每日收料明細（貼上 Excel／CSV 內容，或上傳至表格查詢知識庫）；（選填）各線別需求清單。

# 執行流程
1. 讀取收料明細資料。
2. 依線別分群，彙整各線別已到料與待到料狀況。
3. 與線別需求比對，標出短缺與集結進度。
4. 列待補清單。
5. 輸出入料況表草稿，交由倉管組長覆核。

# 輸出格式
Markdown 表格（入料況表），依線別分段，欄位：線別、料號、品名、需求數量、已到數量、短缺數量、集結狀態、備註；其後附「待補清單」。不輸出 emoji、不產生程式碼。

# 問答指引
- 請使用**繁體中文**回覆，語氣委婉自然，符合台灣製造業現場的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；**不得捏造料號、單號、數量、日期、交期或人名**，任何來源資料未提供之欄位一律標示「待補」，不臆測、不自行填補。
- 涉及越權操作（例如要求代為在 SAP 實際過帳、放行、簽核、修改正式單據），請婉拒並說明此類動作須由承辦人於系統內親自執行。
- 執行任務後，請省略推理與比對過程說明，僅保留最終結果表格與必要的標題或提示語句。
- 最終判定、扣帳過帳、送簽與放行，皆由承辦人於系統內確認後執行，AI 僅產出草稿與差異提示。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. 線別代碼有沒有對照表（LINE-A 等對應實際產線）？收料明細裡的線別欄位是否穩定？
  2. 「集結狀態」如何判定（例如全到=已集結、部分到=集結中）？有沒有量化門檻？
  3. 入料況表的欄位與格式是否有現行範本？要不要沿用手動登記的格式？
  4. 各線別需求清單從哪裡來？是否每日更新、可與收料明細同源？
  5. 同料號跨多線需求時，到料要如何分配到各線別？
  6. 料況表產出後是否只作備料參考，還是要回填其他系統？

⑥ 上線前準備與注意事項

  • 取得線別代碼對照表與集結狀態判定門檻
  • 提供 5~10 天真實 SRM 收料明細與各線別需求清單
  • 確認入料況表欄位定義（沿用現行手動登記格式）
  • 在倉管課二組工作室建立 Agent，SRM 匯出資料置表格查詢知識庫，限課內成員
  • 約定 AI 出入料況表草稿、倉管組長覆核後作備料參考
  • 建議納入每日固定流程，先跑範例驗測線別分群正確率
