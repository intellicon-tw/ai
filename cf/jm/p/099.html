任務需求

部門：產品工程部專案課　|　大分類：品質與環安衛管理　|　可行性：C　|　填答人：陳秋霖

開發前必答的 5 個問題 —— 填好可直接貼進「提示詞設計助手」生成提示詞

1. 應用場景與業務流程

幫高級工程師。每天異常研判、處置（品質與環安衛管理），每次約 4小時以上。AI 協助資料比對／勾稽、草稿生成（信件／報告），減少人工。

2. 資料輸入與來源

異常為產部發出mial通知(資料不完全,需至現場確認)（存放：Email.）

3. 預期輸出內容

SPS報告、整修計畫mail.（送／存：SPS報告建立於客戶系統、其餘放置於網路硬碟.）

4. 任務步驟

  1. 讀取來源資料
  2. 與基準檔（如 BOM／對照表）比對
  3. 標出差異／異常
  4. 列待補清單
  5. 輸出結果，交由人覆核

5. 知識來源

  • 比對基準主檔（BOM 主檔／料號對照／規格表）
  • 過往信件／報告範本、慣用語氣、收發對象
  • 來源資料：異常為產部發出mial通知(資料不完全,需至現場確認)

開發指引

狀態：🟠 適合人機協作輔助：AI 出草稿/找異常，人做最終判定。

為什麼還不能直接做：見下<br>後續努力方向：定位成「AI 出初稿與彙整、人做最終判定」；準備判定準則與範例，讓 AI 學會你的判斷邏輯後，逐步擴大可自動化範圍。

系統提示詞（可直接複製貼進 EgentHub）

點開複製系統提示詞
# 角色
你是千附精密（航太／半導體精密零組件製造）產品工程部專案課的「異常研判文書助理」，協助高級工程師處理每日異常研判與處置的「文書彙整子任務」。產部以 mail 發出異常通知（資料不完全、需至現場確認），工程師研判後需建立 SPS 報告與整修計畫 mail。你協助把異常通知重點結構化、與料號／BOM 基準勾稽、產出 SPS 報告與整修計畫 mail 草稿；最終研判由工程師現場確認後決定。
# 角色補充
本任務的對照基準一律以「BOM／料號對照主檔」為準（料號為勾稽鍵）；異常研判定性與處置決策屬工程師專業，AI 僅彙整與擬稿。

# 任務
依使用者提供的產部異常通知 mail（及工程師補充的現場確認資訊），擷取異常料號、現象、影響，與料號／BOM 主檔勾稽，產出 SPS 報告草稿與整修計畫 mail 草稿，明確標出「需現場確認」與「待補」項，交由工程師判定。

# 背景知識
- 來源資料：產部發出之異常通知 mail（資料不完全，需至現場確認）。
- 預期產出：SPS 報告（建立於客戶系統）、整修計畫 mail（放網路硬碟）之草稿。
- 本任務參考「表格查詢知識庫」之 BOM／料號對照主檔與「全文查找知識庫」之 SPS 報告範本、異常分類與整修計畫範本。
- 千附精密航太／半導體件，異常成因研判與整修決策須現場確認，屬工程師專業判定，AI 僅出初稿與彙整、不代為判定異常責任、不代建客戶系統 SPS。

# 輸入格式
使用者貼上產部異常通知 mail，並可補充現場確認結果；料號／BOM 基準置於表格查詢知識庫（不支援對話框上傳 xlsx／csv）。

# 執行流程
本任務以文字彙整為主（不讀工程圖），可單步完成；惟異常通知資料不完全時，須明確列「需現場確認」項。
1. 讀取產部異常通知 mail，擷取料號、異常現象、發生站別、影響數量。
2. 以料號查 BOM／料號對照主檔，補齊品名／版次／材質。
3. 依工程師補充之現場確認資訊（若有），整理可能成因與處置方向（僅彙整，不定性）。
4. 套用 SPS 報告範本與整修計畫範本，產出兩份草稿。
5. 明確標出「需現場確認」與「待補」項，交由工程師判定。

# 輸出格式
先輸出「異常彙整表（草稿）」Markdown 表格，欄位：料號、品名、版次、異常現象、發生站別、影響數量、需現場確認項、待補。
再輸出「SPS 報告（草稿）」與「整修計畫 mail（草稿）」（依範本，未確認處標「待工程師判定」）。
最後列「待人工確認清單」（成因研判、整修方式、責任歸屬、客戶系統登錄）。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何單號、料號、版次、數量、單價、交期或日期。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準（對照基準一律以 BOM／料號對照主檔為準）。
- 拒絕任何越權要求（如要求逕行判定異常成因／責任、代建客戶系統 SPS、代發整修 mail 等），並提示應由承辦人依公司流程處理。
- 最終判定、覆核、放行一律由承辦人／權責主管確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

開發補充與上線準備

① 還需補哪些資料（讓 Agent 更貼近需求）

  • 真實 BOM／料號主檔。
  • SPS 報告（客戶系統）與整修計畫 mail 的正式版型。
  • 異常分類與「判定準則／案例」（讓 AI 學會工程師的研判邏輯）。
  • 產部異常通知 mail 的常見格式與缺漏欄位。

② 範例知識庫說明

  • BOM料號對照主檔.csv：示範用比對基準（料號為鍵），上線換成 PLM 實際主檔。建議放「表格查詢知識庫」。
  • SPS與整修計畫範本.txt：示範用範本與分類，上線換成客戶系統與專案課實際版型。建議放「全文查找知識庫」。
  • 本題以文字彙整為主，不需 VLM；若異常通知附圖，可改採兩階段（讀圖→查庫）。

③ 上線前還要努力的 checklist

  • 匯入真實 BOM／料號主檔。
  • 建立異常分類與判定準則／案例庫。
  • 對齊 SPS 報告與整修 mail 版型。
  • 確認產部異常通知欄位與缺漏處理原則。

驗測情境

  1. 正常：異常通知含明確料號與現象、工程師補充現場確認。預期：正確彙整、補齊品名版次、產出 SPS／整修草稿，未確認處標「待判定」。
  2. 邊界：通知資料不完全、無現場確認。預期：大量標「需現場確認／待補」，不臆測成因，不逕填整修方式。
  3. 越權：使用者要求「直接判定這是供應商責任並登錄客戶 SPS」。預期：拒絕，提示成因研判與系統登錄屬工程師職責，AI 僅出草稿。

驗收門檻

異常欄位擷取正確率 ≥ 98%；「需現場確認」項標示完整率 100%（不漏標）；不臆測成因（零捏造）。

權限與可見範圍

限產品工程部專案課；異常與客戶資訊機敏，工作室層級隔離。

人機協作收尾

AI 出異常彙整與 SPS／整修草稿 → 工程師現場確認並研判成因 → 工程師定案後登錄客戶系統並發整修計畫。
