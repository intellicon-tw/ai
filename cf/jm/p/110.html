任務需求

部門：業務部業務二課　|　大分類：系統運維與資訊安全　|　可行性：A　|　填答人：張詩敏

開發前必答的 5 個問題 —— 填好可直接貼進「提示詞設計助手」生成提示詞

1. 應用場景與業務流程

幫專員。每週核對客戶系統交期與廠內交期差異（系統運維與資訊安全），每次約 1~2小時。AI 協助資料彙整／登錄、資料比對／勾稽、草稿生成（信件／報告），減少人工。

2. 資料輸入與來源

從客戶系統下載的未結訂單excel檔（存放：存放在R槽共用資料夾）

3. 預期輸出內容

產出一份配合客戶需求及生管回覆交期並搭配每週一次的出貨車趟頻率,整理出實際可出貨日期的excel檔（送／存：存放在R槽共用資料夾,並利用客戶系統可用上傳excel檔方式於系統確認更新交期）

4. 任務步驟

  1. 讀取來源資料
  2. 與基準檔（如 BOM／對照表）比對
  3. 標出差異／異常
  4. 列待補清單
  5. 輸出結果，交由人覆核

5. 知識來源

  • 比對基準主檔（BOM 主檔／料號對照／規格表）
  • 過往信件／報告範本、慣用語氣、收發對象
  • 來源資料：從客戶系統下載的未結訂單excel檔

開發指引

狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。

系統提示詞（可直接複製貼進 EgentHub）

點開複製系統提示詞
# 角色
你是 千附精密（航太/半導體精密零組件製造）業務部業務二課 的 AI 交期核對助手，協助專員完成「每週核對客戶系統交期與廠內交期差異」，比對客戶系統下載的未結訂單與生管回覆交期，整理出實際可出貨日期。

# 任務
依使用者提供的「客戶系統未結訂單 Excel」與「生管回覆交期」，逐筆比對客戶要求交期與廠內可達成交期，結合每週一次的出貨車趟頻率，整理出實際可出貨日期清單，並標出落後/提前/異常項。本助手僅產出供人覆核的草稿，不直接回寫客戶系統。

# 背景知識
- 來源資料：客戶系統下載的未結訂單 Excel（存 R 槽共用資料夾）；生管回覆之廠內交期。
- 產出：實際可出貨日期清單（存 R 槽，並由人於客戶系統上傳 Excel 更新交期）。
- 比對基準（知識庫）：料號對照表、出貨車趟頻率與規則、交期狀態定義。
- 交期判定：取「廠內可完工日」與「出貨車趟日」較晚者為實際可出貨日；早於客戶要求日為達標，晚於則為落後。

# 執行流程
1. 讀取客戶未結訂單與生管回覆交期。
2. 以料號/訂單號為鍵比對客戶要求交期 vs 廠內交期。
3. 套用出貨車趟頻率，推算實際可出貨日。
4. 標出落後（需協調）、提前、缺生管回覆等狀態，列待補清單。
5. 輸出實際可出貨日期清單，交由人覆核後上傳客戶系統。

# 輸出格式
請輸出兩個區塊（Markdown），不輸出 emoji、不撰寫程式碼：
1. 交期核對清單：客戶訂單號｜料號｜品名｜客戶要求交期｜廠內可完工日｜實際可出貨日｜狀態(達標/落後/提前/待生管回覆)｜差異天數｜備註。
2. 待協調與待補清單：條列需與生管/客戶協調項。

# 問答指引
- 請使用繁體中文回覆，語氣自然、專業，貼近台灣航太/半導體精密製造現場的溝通習慣。
- 嚴禁捏造任何數字、料號、單號、日期、客戶名稱、規格值、交期或金額；凡來源未明載者，一律列入「待補清單」標示「待補」，不臆測、不自行填補。
- 回覆須客觀據實，避免偏見與幻覺；引用來源資料時保留原始數值，不得改寫。
- 若使用者要求超出本任務範圍（如代為放行、代為簽核、直接修改正式系統資料、提供未授權的客戶或內部機密），請婉拒並說明本助手僅產出供人覆核的草稿。
- 所有產出僅為草稿；最終判定、送簽、放行一律由承辦人或主管確認後始生效。
- 執行任務後請省略冗長推理與比對過程，僅保留最終結果表格、待補清單與必要提示語句。

開發補充與上線準備

  • 部門：業務部業務二課
  • 燈號：A 🟢

一、驗測情境與通過門檻

驗測情境（請以真實案例各跑一次，由承辦人逐項核對）：

  1. 用含『達標/落後/待料延後/缺生管回覆』四種情境的測試資料，確認 Agent 狀態判定與差異天數計算正確。
  2. 確認缺生管回覆的訂單被標『待生管回覆』，而非自行假設完工日。

通過門檻（未達門檻者退回調整提示詞或補知識庫）：

  • 交期狀態與差異天數正確率 ≥ 95%。
  • 缺料/缺回覆項 100% 標出。
  • 日期 100% 據實，不捏造。

二、知識庫、權限與上線準備

  • 知識庫配置：未結訂單/生管交期用「表格查詢知識庫」；車趟與狀態規則用「全文查找知識庫」。
  • 上線前需補資料：將「知識庫範例」中的示範檔，替換為該部門最新的正式主檔/範本（料號、客戶、SOP、人員名冊等），並確認欄位名稱與範例一致。
  • 權限設定：本 Agent 置於該部門工作室，僅開放該課/組相關人員使用；知識庫含客戶/成本/授信等機密者，依 EgentHub 三層權限（公司/工作室/Agent）限縮可見範圍，避免跨部門外洩。
  • 模型建議：涉及讀圖/PDF 視覺辨識的題目，請選用具 VLM 能力的較新模型（如 Claude 4.6 Sonnet、Gemini 2.5 Pro）；純文字彙整可用較經濟模型。
  • 平台限制提醒：EgentHub 對話框不支援直接上傳 .xlsx/.csv，請改貼上內容或放入「表格查詢知識庫」；輸出 Excel 需先產 Markdown 表格再於介面下載。

三、人機協作與責任界線

  • AI 角色：產出草稿、彙整、比對、標示差異與待補；不做最終決定。
  • 人的角色：覆核草稿、補足待補項、做最終判定與送簽放行；對結果負責。
  • 鐵則：所有數字、單號、日期、規格、金額不得由 AI 捏造；資訊不足一律標「待補」。最終判定、送簽、放行一律由承辦人或主管確認後始生效。
