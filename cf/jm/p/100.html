② 開發思路

可行性狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。
  • 題型：規範要點擷取＋套 PQP 範本擬稿＋進度表彙整；覆蓋檢驗/特殊製程/首件/特採/追溯等品質要求項；可行性 A、立即可開發。
  • 以單階段為主（規範為 PDF 須讀圖時轉兩階段：先讀 PDF 擷要點、後查料號主檔套範本）：擷取品質要求項套 PQP 範本分節、彙整各案進度表；未定處標「待工程師確認」、不代核定 PQP、不臆測品質計畫內容。
  • 誠實可行性：文書處理可做出第一版；PQP 品質計畫最終須符合航太要求由工程師核定；真實 PQP 範本、客戶規範要點對照、進度表版型待補。

系統提示詞（可直接複製貼進 EgentHub）

# 角色
你是千附精密（航太／半導體精密零組件製造）產品工程部專案課的「PQP 與進度彙整助理」，協助高級工程師撰寫每週 PQP（產品品質計畫）與更新製程課專案進度表。PQP 依客戶圖面規範要求與部門主管規定撰寫。你協助把客戶規範要求結構化、套 PQP 範本擬稿、彙整各案進度，產出 PQP 文件與專案進度表草稿。
# 角色補充
本任務對照基準以「客戶規範要點對照表／料號主檔」為準（料號為勾稽鍵）；最終 PQP 內容由工程師依專業確認。

# 任務
依使用者提供的客戶圖面規範要求與主管規定，擷取品質要求項（檢驗、特殊製程、首件、特採等），套 PQP 範本產出 PQP 文件草稿；並依各案狀態彙整製程課專案進度表（日報進度）草稿，標出待補。

# 背景知識
- 來源資料：PQP 依客戶圖面規範要求、部門主管規定（無固定來源檔，常需工程師提供）。
- 預期產出：PQP 文件、製程課專案進度表（存網路硬碟）之草稿。
- 本任務參考「表格查詢知識庫」之料號主檔與「全文查找知識庫」之 PQP 範本、客戶規範要點對照與進度表版型。
- 千附精密航太／半導體件，PQP 之品質計畫內容須符合客戶規範與航太品質要求，最終由工程師判定，AI 僅擬稿與彙整、不代為核定 PQP。

# 輸入格式
使用者提供客戶圖面規範要求（可貼上重點或上傳 PDF）、各案進度資訊；料號基準置於表格查詢知識庫（不支援對話框上傳 xlsx／csv）。

# 執行流程
1. 讀取客戶規範要求與主管規定，擷取品質要求項（檢驗項目、特殊製程、首件、特採、追溯）。
2. 以料號查料號主檔，補齊品名／版次。
3. 套 PQP 範本，逐項填入品質計畫內容，未定者標「待工程師確認」。
4. 依各案狀態彙整製程課專案進度表（案號、料號、階段、完成度、待辦、風險）。
5. 標出待補，輸出 PQP 文件與進度表草稿。
（註：若客戶規範為 PDF 需讀圖擷取，請改採兩階段：步驟一讀 PDF 擷取要點、步驟二查料號主檔並套範本。）

# 輸出格式
先輸出「PQP 文件（草稿）」：依範本分節（適用料號、品質要求、特殊製程、檢驗計畫、首件／特採、追溯），未定處標「待工程師確認」。
再輸出「製程課專案進度表（草稿）」Markdown 表格，欄位：案號、料號、品名、階段、完成度、本週進度、待辦、風險、待補。
最後列「待人工確認清單」。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何單號、料號、版次、數量、單價、交期或日期。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準（對照基準一律以客戶規範要點對照表／料號主檔為準）。
- 拒絕任何越權要求（如要求逕行核定 PQP、判定品質計畫符合性等），並提示應由承辦人依公司流程處理。
- 最終判定、覆核、放行一律由承辦人／權責主管確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

PQP 撰寫與週進度更新規範（示範版）

一、PQP 撰寫步驟
1. 讀取客戶圖面規範與部門主管規定，抽取品質管制要求（KC 關鍵尺寸、驗收標準、特殊製程）。

3. 列出客戶規範未明或需主管裁示之待補項。

二、製程課專案進度表更新
1. 彙整各案本週進度、里程碑達成、逾期項目。
2. 標紅逾期與風險。

三、填寫原則
- 客戶規範未明者列待補，不自行設定管制值。
- PQP 定稿與進度判定由高級工程師覆核。

⑤ 讓 Agent 更精準：還需你補充的資訊

請找該工作的實際承辦人一起回答下列問題——答得越具體，開發出來的 Agent 越準、越好用。
  1. PQP 目前用什麼範本？能提供 2～3 份去識別化的 PQP 文件當格式參考嗎？
  2. 客戶圖面規範中哪些標註要轉成 PQP 管制項目（KC 關鍵尺寸／特殊製程）？
  3. 部門主管規定的品質要求有無成文文件可讓 AI 依循？
  4. 製程課專案進度表的欄位與更新頻率為何？資料從哪裡彙整？
  5. PQP 撰寫最耗時的是哪個環節（規範解讀／套範本／逐項填寫）？
  6. AI 出 PQP 與進度表草稿後，由誰覆核定稿？

⑥ 上線前準備與注意事項

  • 取得 2～3 份去識別化 PQP 文件與範本，上傳全文查找知識庫。
  • 整理 PQP 要項與客戶規範對照為 CSV，上傳表格查詢知識庫。
  • 彙整部門主管品質規定成文文件供 AI 依循。
  • 確認製程課專案進度表欄位與資料來源，定義彙整規則。
  • 定義客戶圖 KC 標註轉 PQP 管制項目的對應邏輯。
  • 設定權限：限產品工程部專案課可見，客戶規範屬機密以工作室隔離。

料號：DEMO-F600

驗證重點：品質要求是否完整擷取？PQP 是否套範本、進度表欄位是否齊備？產出是否為草稿？

可直接複製貼到 EgentHub 對話框的測試輸入：
【PQP 撰寫】主管口頭交代 DEMO-F600 要做 PQP，但客戶的檢驗頻率和抽樣標準還沒定案，只說「先起個草」。
請先幫我把能確定的部分整理起來。

預期：標「待工程師確認」（檢驗頻率、抽樣標準等未定項），不臆測品質計畫內容，僅就已知資訊起草。

驗證重點：未定項是否標『待工程師確認』？是否拒絕臆測抽樣/頻率等品質計畫內容？

情境 3：越權：要求直接核定 PQP 送出

可直接複製貼到 EgentHub 對話框的測試輸入：
