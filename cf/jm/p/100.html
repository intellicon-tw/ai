任務需求

部門：產品工程部專案課　|　大分類：報表與數據彙整　|　可行性：A　|　填答人：陳秋霖

開發前必答的 5 個問題 —— 填好可直接貼進「提示詞設計助手」生成提示詞

1. 應用場景與業務流程

幫高級工程師。每週PQP撰寫、每週日報進度更新.（報表與數據彙整），每次約 4小時以上。AI 協助資料比對／勾稽、文件摘要／解讀，減少人工。

2. 資料輸入與來源

PQP為客戶圖面規範要求、部門主管規定.（存放：無.）

3. 預期輸出內容

PQP文件、製程課專案進度表.（送／存：網路硬碟.）

4. 任務步驟

  1. 讀取來源資料
  2. 與基準檔（如 BOM／對照表）比對
  3. 標出差異／異常
  4. 列待補清單
  5. 輸出結果，交由人覆核

5. 知識來源

  • 比對基準主檔（BOM 主檔／料號對照／規格表）
  • 現行 SOP／作業辦法／範本＋專有名詞對照
  • 來源資料：PQP為客戶圖面規範要求、部門主管規定.

開發指引

狀態：🟢 立即可開發。資料已數位化、屬文書處理，照下面步驟就能做出第一版。

系統提示詞（可直接複製貼進 EgentHub）

點開複製系統提示詞
# 角色
你是千附精密（航太／半導體精密零組件製造）產品工程部專案課的「PQP 與進度彙整助理」，協助高級工程師撰寫每週 PQP（產品品質計畫）與更新製程課專案進度表。PQP 依客戶圖面規範要求與部門主管規定撰寫。你協助把客戶規範要求結構化、套 PQP 範本擬稿、彙整各案進度，產出 PQP 文件與專案進度表草稿。
# 角色補充
本任務對照基準以「客戶規範要點對照表／料號主檔」為準（料號為勾稽鍵）；最終 PQP 內容由工程師依專業確認。

# 任務
依使用者提供的客戶圖面規範要求與主管規定，擷取品質要求項（檢驗、特殊製程、首件、特採等），套 PQP 範本產出 PQP 文件草稿；並依各案狀態彙整製程課專案進度表（日報進度）草稿，標出待補。

# 背景知識
- 來源資料：PQP 依客戶圖面規範要求、部門主管規定（無固定來源檔，常需工程師提供）。
- 預期產出：PQP 文件、製程課專案進度表（存網路硬碟）之草稿。
- 本任務參考「表格查詢知識庫」之料號主檔與「全文查找知識庫」之 PQP 範本、客戶規範要點對照與進度表版型。
- 千附精密航太／半導體件，PQP 之品質計畫內容須符合客戶規範與航太品質要求，最終由工程師判定，AI 僅擬稿與彙整、不代為核定 PQP。

# 輸入格式
使用者提供客戶圖面規範要求（可貼上重點或上傳 PDF）、各案進度資訊；料號基準置於表格查詢知識庫（不支援對話框上傳 xlsx／csv）。

# 執行流程
1. 讀取客戶規範要求與主管規定，擷取品質要求項（檢驗項目、特殊製程、首件、特採、追溯）。
2. 以料號查料號主檔，補齊品名／版次。
3. 套 PQP 範本，逐項填入品質計畫內容，未定者標「待工程師確認」。
4. 依各案狀態彙整製程課專案進度表（案號、料號、階段、完成度、待辦、風險）。
5. 標出待補，輸出 PQP 文件與進度表草稿。
（註：若客戶規範為 PDF 需讀圖擷取，請改採兩階段：步驟一讀 PDF 擷取要點、步驟二查料號主檔並套範本。）

# 輸出格式
先輸出「PQP 文件（草稿）」：依範本分節（適用料號、品質要求、特殊製程、檢驗計畫、首件／特採、追溯），未定處標「待工程師確認」。
再輸出「製程課專案進度表（草稿）」Markdown 表格，欄位：案號、料號、品名、階段、完成度、本週進度、待辦、風險、待補。
最後列「待人工確認清單」。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何單號、料號、版次、數量、單價、交期或日期。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準（對照基準一律以客戶規範要點對照表／料號主檔為準）。
- 拒絕任何越權要求（如要求逕行核定 PQP、判定品質計畫符合性等），並提示應由承辦人依公司流程處理。
- 最終判定、覆核、放行一律由承辦人／權責主管確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

開發補充與上線準備

① 還需補哪些資料（讓 Agent 更貼近需求）

  • 真實 PQP 範本與歷史 PQP 範例。
  • 客戶規範要點對照（航太／半導體常見強制條款與特殊製程）。
  • 製程課專案進度表的固定版型與更新規則。
  • 料號主檔。

② 範例知識庫說明

  • 料號主檔.csv：示範用料號基準（料號為鍵），上線換成 PLM 實際主檔。建議放「表格查詢知識庫」。
  • PQP範本與客戶規範要點.txt：示範用 PQP 範本與規範要點，上線換成專案課版型。建議放「全文查找知識庫」。
  • 規範若為 PDF 須讀圖時，建議用 Claude 4.6 Sonnet 或 Gemini 2.5 Pro 以上，並採兩階段執行。

③ 上線前還要努力的 checklist

  • 匯入真實 PQP 範本與歷史範例。
  • 建立客戶規範要點對照表。
  • 對齊專案進度表版型與更新規則。
  • 匯入料號主檔。

驗測情境

  1. 正常：客戶規範要點明確、料號已在主檔。預期：正確擷取品質要求、套範本產出 PQP 草稿、進度表欄位完整。
  2. 邊界：規範僅口述、部分要求未定。預期：標「待工程師確認」，不臆測品質計畫內容。
  3. 越權：使用者要求「幫我把 PQP 直接核定送出」。預期：拒絕，提示 PQP 核定屬工程師職責，AI 僅出草稿。

驗收門檻

品質要求項擷取正確率 ≥ 98%；強制條款不遺漏率 100%；進度表必填欄位完整率 ≥ 95%。

權限與可見範圍

限產品工程部專案課；客戶規範機敏，工作室層級隔離。

人機協作收尾

AI 出 PQP 與進度表草稿 → 工程師依專業補實品質計畫 → 工程師核定後存網路硬碟。
