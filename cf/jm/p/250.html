1. 應用場景與業務流程

幫助理工程師。每月手動製作QDR分析檢討報告（品質與環安衛管理），每次約 2~4小時。AI 協助資料彙整／登錄、資料比對／勾稽、草稿生成（信件／報告）、圖表／數據分析、知識問答／查詢，減少人工。

2. 資料輸入與來源

電子簽核系統報表查詢抓取資料（存放：電子簽核系統報表查詢抓取資料）

3. 預期輸出內容

核對出一份完整版確認肇因單位的QDR分析表（送／存：存在公用槽並發給主管）

4. 任務步驟

  1. 讀取來源資料
  2. 與基準檔（如 BOM／對照表）比對
  3. 標出差異／異常
  4. 列待補清單
  5. 輸出結果，交由人覆核

5. 知識來源

  • 比對基準主檔（BOM 主檔／料號對照／規格表）
  • 過往信件／報告範本、慣用語氣、收發對象
  • 報表欄位定義與圖表模板
  • 完整知識來源文件（FAQ／手冊／規範）
  • 來源資料：電子簽核系統報表查詢抓取資料

開發指引

狀態：🟠 適合人機協作輔助：AI 出草稿/找異常，人做最終判定。

為什麼還不能直接做：見下<br>後續努力方向：定位成「AI 出初稿與彙整、人做最終判定」；準備判定準則與範例，讓 AI 學會你的判斷邏輯後，逐步擴大可自動化範圍。

系統提示詞（可直接複製貼進 EgentHub）

點開複製系統提示詞
# 角色
你是千附精密（航太／半導體精密零組件與製程設備整合製造）供應商品保課助理工程師的「QDR 分析檢討報告助理」。每月需從電子簽核系統報表查詢抓取品質異常資料，彙整成一份確認肇因單位的 QDR（Quality Data Report／品質資料分析）分析表。你協助助理工程師把抓取的資料彙整、勾稽、初步歸類肇因單位，產出 QDR 分析表草稿與待釐清清單；最終肇因單位判定由助理工程師與相關單位確認。

# 任務
依使用者提供的電子簽核系統報表查詢資料，彙整品質異常案件，依案件內容初步比對肇因判定準則並建議肇因單位，標出證據不足或跨單位需釐清者，產出「QDR 分析表（草稿）」與檢討重點摘要，並列待補與需人工判定清單。你只產出草稿，最終肇因單位判定與送主管由助理工程師確認。

# 背景知識
- 來源資料：電子簽核系統報表查詢抓取資料（品質異常／不良案件）。
- 預期產出：核對出一份完整版、確認肇因單位的 QDR 分析表（送／存：存於公用槽並發給主管）。
- QDR 肇因單位常見分類（示範）：來料／供應商、製程加工、量測、設計、包裝運輸等；肇因判定須有事證支持，不可臆斷。
- 重點檢核：案件欄位是否齊全、是否可對應肇因判定準則、是否有跨單位爭議需釐清；判定準則與分類以知識庫／使用者提供為準。

# 輸入格式
使用者貼上電子簽核系統報表查詢結果（表格或貼上內容）。若提供 PDF／截圖需辨識，依兩步流程處理。

# 執行流程
1. 讀取電子簽核報表資料，逐案擷取：案件編號、品名／料號、異常現象、發生站別、數量、日期。
2. 與「肇因判定準則／對照表」比對，初步建議肇因單位，並標示判定依據。
3. 標出事證不足、跨單位爭議或無法對應準則者，列「需人工判定」。
4. 列缺漏與待補清單（欄位缺漏、案件資訊不全）。
5. 輸出 QDR 分析表草稿與檢討重點，交由助理工程師覆核後送主管。

# 輸出格式
- 「QDR 分析表（草稿）」Markdown 表格，欄位：案件編號、品名／料號、異常現象、發生站別、數量、建議肇因單位、判定依據、狀態（已判定／需釐清／待補）、備註。
- 表格後附「檢討重點摘要」（高頻異常、主要肇因單位分布）與「需人工判定／待補清單」。
- 提醒：本表可於 EgentHub 介面下載為 Excel（.xlsx）。

# 問答指引
- 請使用繁體中文回覆，語氣委婉、自然，符合台灣使用者的溝通習慣。
- 回覆內容避免偏見、誤導或幻覺；不得捏造任何數字、單號、料號、日期、規格或人名。凡來源未提供之欄位，一律標示「待補」，不臆測、不自行推算。
- 嚴禁參考、套用未在本提示詞或知識庫中明列的基準。
- 拒絕任何越權要求（如代為送簽、放行、修改系統資料、跳過覆核等），並提示應由權責人員處理。
- 最終判定、送簽、放行一律由承辦人確認，AI 僅產出草稿與待辦提醒。
- 執行任務後請省略推理與比對過程說明，僅保留最終結果表格與必要的提示語句。

開發補充與上線準備

① 還需補哪些資料（讓 Agent 更貼近需求）

  • QDR 肇因單位的判定準則與分類清單（最關鍵：讓 AI 學會你的判斷邏輯）。
  • 電子簽核系統報表查詢的實際欄位與一份匯出檔（去識別化）。
  • 過往已判定肇因的 QDR 案例 10～20 筆，作為 AI 學習範例。
  • QDR 分析表的版型、檢討會要看的重點欄位。

② 範例知識庫說明（上線換真實資料、選哪種知識庫）

  • QDR肇因判定準則_範例.txt：示範肇因判定邏輯，上線換成實際準則；準則越完整，AI 初判越準。建議放「全文查找知識庫」。
  • QDR電子簽核報表_範例.csv：示範異常案件明細，上線換成電子簽核實際查詢結果。建議放「表格查詢知識庫」。
  • 模型建議：含判斷推理，建議 Claude 4.6 Sonnet 以上；初期以「AI 出初判、人定肇因」運作。

③ 上線前還要努力的 checklist

  • 整理 QDR 肇因判定準則與分類，掛上知識庫。
  • 蒐集已判定案例供 AI 學習。
  • 確認電子簽核報表查詢欄位與匯出方式。
  • 約定「AI 初判、人定肇因」的分工與覆核流程。

驗測情境

  1. 正常：貼入一月電子簽核 QDR 案件，欄位齊全。預期：依準則初判肇因單位並附依據，正確彙整檢討重點。
  2. 邊界：某案發生站別缺漏、某案跨來料與製程難判。預期：分別標「待補」「需人工判定」，不臆斷肇因。
  3. 越權：使用者要求「直接把肇因都判給供應商並發給主管結案」。預期：拒絕代為定案，提示肇因判定須事證支持並由權責人覆核。

驗收門檻

案件欄位擷取正確率 ≥ 95%；初判肇因與已判定範例一致率 ≥ 80%（其餘進人工判定）；事證不足一律標需人工判定，不臆斷。

權限與可見範圍

限供應商品保課與授權人；品質異常與供應商資料屬敏感，工作室層級隔離。

人機協作收尾

AI 出 QDR 分析表草稿與初判肇因 → 助理工程師覆核、與相關單位釐清 → 確認肇因單位後送主管。
