照片整理與標籤助理

戰略：ROI（決策品質） ｜ 使用者：全公司 15 人 ｜ 工作室：知識整理與內容產出
由 AI 助理協助同仁完成「照片整理與標籤助理」相關的標準作業流程，提升效率並降低人工錯誤，從繁瑣作業中釋放時間。
⚠ 本 Agent 含示範資料：知識庫中標示為示範資料的檔案，需在上線前由斯創對應部門替換為真實資料，Agent 才能進行真正的業務驗測與上線。替換時請維持欄位結構與檔名一致。

專案目標

本 Agent 為斯創教育 AI 助理導入計畫的成員之一，目標是把日常重複性、規則明確的事務轉為 AI 協作，讓同仁專注於高價值工作。

對學員的引導：本說明書按 5 個 Tab 順序操作，預計 60-90 分鐘可完成單一 Agent 的部署與初步驗測。

技術配置建議

成本控管建議：

  • POC 階段先用首選模型確認品質，正式上線後再試低成本模型對照
  • 對話 history 不需保留過長（5-10 輪內），減少重複 token 計費
  • 知識庫採「精簡版」上傳，非必要欄位先剔除
  • 每月觀察「點數消耗／實際使用次數」比例，異常時即時調整

系統提示詞

下方為「照片整理與標籤助理」的完整系統提示詞，請整段複製並貼到 EgentHub Agent 設定頁的「Prompt」欄位。

SYSTEM PROMPT 
# 角色
您是「斯創教育照片整理與標籤助理」，協助同仁將活動現場拍攝的照片快速辨識內容、挑選代表照、加上標籤（活動類型、地點、主題、人物多寡），方便後續用於提案、報告、社群、年度集錦。

# 任務
當同仁上傳活動照片時，您需要：
1. 步驟一：純讀取每張照片，描述場景內容（地點特徵、人物動態、活動主題、構圖品質）。
2. 步驟二：依「照片標籤體系」為每張照片貼標，並建議「代表照」「備用」「不採用」三級分類。

# 背景知識
本 Agent 連結 Agent 向量知識庫「斯創照片標籤體系」，含活動類型標籤、場景標籤、主題標籤、人物多寡標籤、構圖品質標籤。

# 輸入格式
使用者上傳活動照片（一次最多 10 張）。可附加活動資訊（活動名稱、日期、地點）以提升標籤準確性。

# 執行流程
**重要提醒：本任務需嚴格分為兩個步驟完成**

---

## 步驟一：圖像理解

**您的任務：**
1. 接收使用者上傳的照片。
2. 逐張描述：場景（室內／戶外／森林／教室）、人物（兒童／青年／長者／群體大小）、動作（觀察／聆聽／操作教具／團體合影）、構圖品質（清楚／光線好／模糊／逆光）。
3. 以 Markdown 表格輸出。

**步驟一限制：**
- 嚴格禁止調用「照片標籤體系」或任何工具。
- 嚴格禁止猜測標籤或分類等級。

**結束提示：**
「我已完成照片內容辨識，若需建議標籤與分類，請在下一步告訴我。」

---

## 步驟二：標籤與分類

**您的任務：**
1. 接收使用者指示。
2. 僅依據步驟一已產出的場景描述，到「照片標籤體系」比對。
3. 為每張照片補上 2-4 個標籤與分類等級（代表照／備用／不採用）。
4. 推薦本批照片中的 3 張「代表照」用於社群與報告。

**步驟二限制：**
- 嚴格禁止要求使用者重新上傳照片。
- 僅使用步驟一已產出的描述。

資料清洗指南

為什麼一定要清洗
  • 知識庫資料若不一致（欄位定義模糊、編碼亂、空白工作表），會直接拉低 Agent 命中率
  • 個資或敏感商業資訊未先去識別化，會造成資安風險
  • 歷史資料若超過 6 個月未更新，將失去比對基準

結構化資料（CSV）清洗清單

  1. 檢查欄位名稱：與 Prompt 中假設的欄位完全一致（大小寫、用詞）
  2. 檢查資料筆數：本 POC 範例為 10-30 筆，正式上線建議至少 100-500 筆
  3. 檢查空值處理：缺漏欄位明確填「無」或「-」，不留空白
  4. 檢查日期格式：統一為 YYYY-MM-DD
  5. 檢查金額／數字：移除千分位逗號、貨幣符號，僅保留數字
  6. 檢查重複資料：依鍵欄位（如客戶 ID、訂單編號）去重

非結構化文件（TXT/PDF）清洗清單

  1. 章節結構化：明確的標題、子標題，避免一大段無斷句的文字
  2. 關鍵詞密度：每段應包含 2-3 個用戶可能問到的關鍵詞
  3. 同義詞涵蓋：客戶實際用語有差異時（「估價」「報價」「估算」），盡量在文件中各出現一次
  4. 定期更新：建議每季由內容負責人檢視一次

驗測情境（2 個）

下列情境涵蓋「正常路徑」「邊界提醒」「異常退回」三類典型結果。建議在上線前依此清單逐一驗測，目標：命中率 ≥ 80%、誤判率 ≤ 10%。

情境 #1：步驟一純辨識 邊界提醒

輸入描述：上傳 5 張照片

預期輸出：場景、人物、動作、構圖品質四欄描述

驗證重點：不出現標籤或分類等級

情境 #2：步驟二完成標籤 邊界提醒

輸入描述：依步驟一結果

預期輸出：2-4 個標籤 + 分類等級 + 推薦 3 張代表照

驗證重點：不可重讀圖片

驗測執行建議 每個情境至少準備 3 筆真實或仿造的測試資料，由貴單位資深人員擔任驗收者。將 Agent 輸出與專家判斷比對，若連續 2 次未通過某情境，請回頭調整 Prompt 或補強知識庫。

正式上線前的準備指南

本文件提供將本 Agent 從 POC 套件正式部署到 EgentHub 平台、並在斯創教育日常作業中穩定使用的逐步準備清單。建議由 IT/AI 種子團隊與業務負責人共同檢核。

下列六階段檢核清單，預估期程 4-8 週。

1
資料準備
  • 依「知識庫設定」Tab 的清洗指南整理本 Agent 正式資料
  • 確認資料負責人與更新節奏（每月／每季）
  • 機敏資料先去識別化或加註「僅限內部」標籤
2
知識庫建置
  • 於 EgentHub 後台建立本 Agent 所需的知識庫類型
  • 上傳已清洗的正式資料，並設定為「Agent 層級」或「工作室層級」
  • 確認索引建立完成（CSV 約 30 秒、PDF 視頁數 2-5 分鐘）
3
Agent 設定
  • 建立 Agent 並貼上「完整 Prompt」Tab 的完整 Prompt
  • 選擇推薦模型：Claude 4.6 Sonnet
  • 需在 Agent 設定啟用「兩階段隔離」（步驟一純讀取、步驟二純工具調用）
  • 啟用「知識庫設定」Tab 建議的知識庫與工具
  • 設定每月算力點數上限
4
測試驗證
  • 依「驗測情境」Tab 所列情境逐一驗測，記錄命中率與誤判率
  • 命中率未達 80%：先檢查知識庫資料完整性，再調整 Prompt
  • 連續 3 次發現相同錯誤：補強知識庫或新增規則
5
教育訓練
  • 對使用者 30 分鐘 onboarding：本 Agent 能做什麼、不能做什麼
  • 提供「常見問題與最佳輸入範例」3-5 例
  • 建立內部回饋機制（Email 或 LINE 群組），收集使用者問題
6
正式上線與監控
  • 正式發布並通知所有授權使用者
  • 前 2 週每日檢視使用量與異常率
  • 每月一次「Agent 健檢會議」，由負責窗口主持
關鍵風險提醒
  • AI 不取代人工最終決策：本 Agent 為協助工具，最終決策仍由相關人員承擔
  • 知識庫時效性：歷史資料若超過 6 個月未更新，將影響比對準確度
  • 規則完整性：建議持續補充規則與範例，提升命中率
