# 生產效率分析助手 - 正式上線前的準備指南

本文件提供將「生產效率分析助手」從 POC 套件正式部署到 EgentHub 平台、並在三達榮公司生產製造部與生管團隊日常作業中穩定使用的逐步準備清單。建議由 IT/AI 種子團隊與生產主管（廠長、生管主管）共同檢核。

---

## 1. 知識庫設定

本 Agent 使用兩個「表格查詢知識庫」(SQL)。請依下列步驟逐一建立。

### 1.1 2026 年 3 月生產日報表（表格查詢知識庫）

- **來源檔案**：`2_知識庫範例資料/2026年3月生產日報表.csv`
- **知識庫類型**：表格查詢知識庫（SQL）
- **建議名稱**：「生產日報表 (Daily Production Report)」
- **層級**：Agent 層級（僅本 Agent 使用，避免污染其他 Agent 的命名空間）
- **欄位確認**：生產日期、部門、訂單號、款號、款號名稱、顏色、訂單總數、製程、當日生產數量
- **編碼確認**：CSV 使用 `utf-8-sig`（UTF-8 with BOM），上傳前以 Excel 開啟確認中文不亂碼
- **正式資料替換**：上線後由生管窗口每日／每週上傳最新日報表。建議排程方式：
  - 短期：由生管人員手動匯出當週／當月日報表，覆蓋原檔
  - 中期：透過 EgentHub MCP 介接 ERP 或 MES，每日凌晨自動同步
- **資料品質要求**：
  - 確認「製程」欄位的值統一為「針車」「成型」「後整」或空白（外協），避免出現新值（如「针车」簡體）導致查詢遺漏
  - 確認「當日生產數量」欄位無負值或空字串
  - 配件雜項（訂單號=1、訂單總數=999999）保留不刪除，由 Agent 自行排除

### 1.2 生產組別與製程主檔（表格查詢知識庫）

- **來源檔案**：`2_知識庫範例資料/生產組別與製程主檔.csv`
- **知識庫類型**：表格查詢知識庫（SQL）
- **建議名稱**：「生產組別主檔 (Production Team Master)」
- **層級**：Agent 層級
- **欄位確認**：部門、製程類別、群組分類、備註
- **正式資料替換**：當廠內新增組別、調整製程歸屬時，由生管或廠務同仁更新此主檔。建議每季檢視一次。
- **資料品質要求**：「部門」欄位的命名必須與日報表完全一致（例如同樣使用「針十组」而非「針十組」），避免關聯失敗。

---

## 2. 模型選擇建議

- **建議預設模型**：Claude 4.6 Sonnet
- **理由**：本 Agent 需要做數字加總、百分比計算、平均推估、趨勢辨識等多步驟推理，且需依不同問題類型套用不同表格格式，對推理深度要求較高，Claude 4.6 Sonnet 的表現最穩定。
- **次選模型**：若點數預算吃緊，可改用 Gemini 3 Flash 處理單純查詢類問題（如情境 #1 日產量統計、情境 #4 跨製程對比），但情境 #2 訂單進度與情境 #3 趨勢分析建議保留 Claude 4.6 Sonnet。
- **避免使用**：Claude 3.5 Sonnet 以前、Gemini 1.5 Pro 以前等已停止維護的舊版模型。

---

## 3. 兩階段隔離設定

本 Agent 為單階段執行（純文字輸入 + 表格查詢），無須在 EgentHub 後台額外啟用兩階段隔離設定。

---

## 4. 權限與工作室設定

- **工作室歸屬**：建議掛在「工作室1_生產製造部」之下。
- **使用者範圍**：
  - 必開：廠長、生管主管、生管人員、生產各組組長
  - 選開：業務主管（查訂單進度時使用）、品管主管（觀察製程趨勢時使用）
  - 暫不開放：一般作業員（資料粒度與權限考量）
- **算力點數初期建議**：依預估每日 30-60 次查詢估算，第一個月配 1 萬點觀察消耗，視實際使用調整。
- **資料敏感度**：
  - 生產日報表包含客戶訂單號、款號（可推測客戶品牌如 Sam's、COSTCO、Victorinox），屬於營業機密
  - 不建議將本 Agent 開放給外部供應商或代工廠
  - 建議啟用 EgentHub 的「日誌追蹤」功能，所有查詢可追溯稽核
  - 若未來考慮對東莞工廠開放，建議走「混合雲架構」：知識庫保留在台灣總部地端，僅 Prompt 上雲運算

---

## 5. 上線前驗測 Checklist

執行 `3_驗測情境.txt` 中的全部 6 個情境，並確認以下為 pass/fail gate（必須通過才能正式上線）：

- [ ] **情境 #1（日產量統計，標準路徑）**：必須通過。多筆同日紀錄需正確加總、配件雜項需排除。
- [ ] **情境 #2（訂單進度，含預估完工日）**：必須通過。需正確識別同訂單多顏色變體、累積數量計算無誤、預估完工日合理。
- [ ] **情境 #4（跨製程對比）**：必須通過。外協（空白製程）需獨立計算、佔比加總須為 100%。
- [ ] **情境 #6（超出範圍的查詢，誠實揭露）**：必須通過。不可捏造 OEE 或不良率數字，需具體說明缺哪些資料。

情境 #3（趨勢分析）與情境 #5（異常識別）為「建議通過」，若初版回覆品質尚未穩定，可在上線後持續優化。

---

## 6. 上線後監控指標（首 2 週）

請生管主管或種子團隊成員每週檢視一次以下指標：

- **每日使用量**：使用次數、使用者人數、人均查詢次數（觀察是否被實際採用）
- **常見問句聚類**：實際使用者最常問哪幾類問題？是否與 POC 設計的 5 大分析面向吻合？若有大量問題落在「資料範圍提醒」之外，代表使用者需求超出資料涵蓋範圍，需評估是否補資料。
- **異常觸發次數**：使用者主動回報「答錯」「答不出來」的次數，若每週超過 5 次，需檢視提示詞與資料品質。
- **回應時間**：平均回應應在 10 秒內，若超過 30 秒需檢查模型負載或知識庫大小。
- **算力點數消耗速率**：依此推估月成本，做後續預算規劃。

## 7. 異常處理與回退方案

### 常見錯誤型態與排查步驟

- **回覆「找不到資料」但實際應有資料**：先確認生產日報表是否更新到當期；再確認查詢條件中的組別名稱、訂單號是否與資料完全一致（注意簡繁體差異，例如「針十组」用簡體「组」字）。
- **數字加總明顯錯誤**：可能是模型在處理大量列時推理失準，建議讓使用者改用更具體的條件（例如先指定單一日期或單一訂單號），縮小查詢範圍。
- **預估完工日離譜（過去日期或數年後）**：通常是該訂單近 5 日平均日產為 0 或極低所致，請查該訂單是否已停產，並提醒使用者該數字僅供參考。
- **回覆中夾雜推理過程或泛泛說明**：違反問答指引「省略推理、僅保留結果表格」，應在後台微調提示詞或重新訓練種子團隊。

### 何時應暫時下架 Agent 並回到人工流程

- 連續 3 天以上發現重大數字錯誤（如訂單進度差超過 10%）
- 知識庫資料來源遺失或污染（例如錯誤匯入了其他月份的資料導致跨期混淆）
- 使用者大量反映回覆品質下降

### Agent 維運窗口

- **業務負責人**：建議由生管主管擔任，負責決定查詢需求變更、是否擴展功能
- **技術維運**：由 IT 種子團隊成員擔任，負責資料更新、提示詞微調、點數監控
- **變更管理**：每次提示詞修改，至少需重跑情境 #1、#2、#4、#6 四個必過 gate，全部通過後才上線

## 8. 種子團隊接手提示

### 本 Agent 最常見的提示詞調整方向

- **新增分析類型**：若使用者開始問「哪幾天加班最多」「哪些訂單最常被延遲」等新問題，可在「執行流程」與「輸出格式」段落新增對應的處理邏輯與表格範本。
- **資料涵蓋範圍擴大**：若日報表新增「不良數」「工時」「機台號」等欄位，需同步更新「背景知識」段落的欄位說明，並讓 Agent 知道可計算 OEE、不良率等新指標。
- **跨月份比較**：當資料累積到 6 個月以上時，可在執行流程中新增「跨月份對比」處理邏輯。

### 客戶若要新增類似 Agent，可從本 Agent 為起點 copy

- 「生產排程驗證助手」（同部門姊妹 Agent，已列為旗艦級 POC）：與本 Agent 共用相同的生產日報表知識庫，但需另外加入「訂單交期表」「產能標準」「模具占用表」三份資料。建議從本 Agent 的提示詞架構為起點調整。
- 「品質檢驗趨勢分析助手」（品管部）：如未來補入品檢資料，可參考本 Agent 的「趨勢分析」「異常識別」邏輯重複利用。

### 與其他 Agent 的資料依賴關係

- **上游資料**：生產日報表來自生管人員手動匯出或 ERP/MES 同步。資料品質直接決定本 Agent 的回答品質。
- **下游應用**：本 Agent 的訂單進度查詢結果，可進一步餵給「業務客戶訂單追蹤整合助手」（業務部 ROI 應用），讓業務窗口直接看到訂單狀況。
- **互補 Agent**：「生產排程驗證助手」是本 Agent 的姊妹題目，兩者結合可形成完整的生產可視化與排程決策閉環。
