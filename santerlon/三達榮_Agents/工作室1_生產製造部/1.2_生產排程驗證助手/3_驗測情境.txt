# 生產排程驗證助手 - 驗測情境

本檔案分為兩部分：
- **Part A — 測試資料集**：可直接貼到 EgentHub 對話框做手動驗測的自然語言輸入。
- **Part B — 驗測情境**：每個情境的預期輸出與驗證重點，作為手動或半自動回歸驗測依據。

---

## Part A — 測試資料集

### 測試 #1：全庫範圍掃描

```
幫我掃一遍所有排程，列出有問題的批次。
```

### 測試 #2：指定單一組別

```
檢查 SL03 這組的排程狀況。
```

### 測試 #3：指定時間區間（含勞動節連假）

```
看看 5/1~5/10 這段時間的排程，有沒有排到例假日的？
```

### 測試 #4：指定單一訂單

```
VS2512044 的排程合理嗎？
```

### 測試 #5：組合條件查詢

```
SL01 和 SL02 整個 5 月有沒有逾期風險？
```

### 測試 #6：模糊指令（觸發範圍確認）

```
幫我檢查一下排程。
```

### 測試 #7：外發廠掃描

```
海陽外發廠這個月的排程有沒有問題？
```

### 測試 #8：產能超載驗證

```
SL01 在 4 月份有沒有產能超載的情況？
```

## Part B — 驗測情境

### 情境 #1：全庫掃描（標準路徑）

**輸入描述**

> 使用者要求掃描全庫所有排程，找出所有違反 8 條規則的批次。

**預期輸出**

> 採用 Markdown 表格格式，列出所有命中規則的批次，依「廠別 → 組別 → 訂單號」排序。應涵蓋三達榮、响越、百越三個自有廠的 schedule_gantt 共 960 筆，以及 7 個外發廠的 schedule_outsource 共 280 筆。
> 表格後附統計區，分類列出每條規則命中數量。

**驗證重點**

> 1. 是否完整掃描兩張排程表（schedule_gantt + schedule_outsource），不應遺漏外發廠。
> 2. 排序是否依「廠別 → 組別 → 訂單號」三層次正確。
> 3. 同一筆若命中多條規則，應拆成多列（每列一個問題類型），而非合併。
> 4. 統計區的各類筆數加總應等於表格總列數。
> 5. 不應出現捏造的訂單號或廠別。

### 情境 #2：指定組別查詢（範圍解析）

**輸入描述**

> 使用者要求檢查 SL03 組的排程。

**預期輸出**

> 僅呈現 SL03 組的問題清單。若 SL03 排程完全正常，回覆「本次檢查範圍內未發現排程異常，排程狀態正常。」否則以標準表格列出問題並附統計。
> 統計區應明確標示「本次檢查範圍：SL03 組別」。

**驗證重點**

> 1. 是否正確過濾廠別=三達榮、組別=SL03 的紀錄，並排除其他組別。
> 2. 是否避免將「SL03X」或類似編碼誤抓進來。
> 3. 範圍標示是否與使用者指令一致。

### 情境 #3：時間區間 + 例假日驗證（R6）

**輸入描述**

> 使用者要求檢查 2026/5/1~5/10 區間的排程，特別關注例假日。

**預期輸出**

> 列出計畫起訖區間（計畫開始日期 ~ 計畫結束日期）覆蓋到以下日期的批次：
> - 2026-04-30（解放日&統一日）
> - 2026-05-01（勞動節）
> - 2026-05-02（補假）
>
> 問題類型標示為「R6 排在例假日」，問題描述應指出具體被覆蓋的假期名稱與日期。

**驗證重點**

> 1. 是否正確查 holidays 表，三個連假日期皆覆蓋。
> 2. 「計畫起訖日區間包含假期」的判斷邏輯：包含端點（計畫開始日 ≤ 假期 ≤ 計畫結束日）。
> 3. 是否將 4/30 也納入（雖然使用者指令是 5/1~5/10，但若有批次計畫從 4/29 跨到 5/3，4/30 假期應被命中）。
> 4. 問題描述應包含假期名稱（例如「批次計畫覆蓋勞動節 5/1」），不只列日期。

### 情境 #4：單一訂單驗證

**輸入描述**

> 使用者詢問 VS2512044 訂單的排程合理性。

**預期輸出**

> 列出 VS2512044 在所有廠／組／外發廠中的所有批次紀錄。對每筆批次套用 8 條規則。若全部正常，回覆排程狀態正常；否則列出問題清單並附統計。

**驗證重點**

> 1. 是否同時查 schedule_gantt 與 schedule_outsource，避免遺漏該訂單在外發廠的批次。
> 2. 若該訂單有多個顏色變體（同訂單號多筆款號），是否全部列出。
> 3. 統計區是否明確標示「本次檢查範圍：訂單 VS2512044」。

### 情境 #5：模糊指令（防呆 - 範圍確認）

**輸入描述**

> 使用者僅說「幫我檢查一下排程」，未指定任何條件。

**預期輸出**

> Agent 應主動回覆：
> 「請問您想檢查哪個範圍？可指定以下條件之一或多項：廠別、組別、訂單號／款號、日期區間。若不指定任何條件，預設會掃描全庫。」
> 不應直接掃全庫並丟出大表格。

**驗證重點**

> 1. 是否觸發範圍確認對話而非直接執行。
> 2. 語氣是否委婉自然，符合台灣使用者風格。
> 3. 是否明確列出四類可指定條件供使用者選擇。

### 情境 #6：外發廠掃描

**輸入描述**

> 使用者要求掃描海陽外發廠這個月的排程。

**預期輸出**

> 從 schedule_outsource 過濾外發廠=海陽、且計畫日期落在當月（依 Current Time 工具取得當前日期，推算當月起訖）的批次。套用 8 條規則。

**驗證重點**

> 1. 是否正確呼叫 Current Time 工具取得當前日期。
> 2. 「這個月」的解讀是否為當月 1 日至月底。
> 3. 是否僅查 schedule_outsource，不應誤把自有廠資料混入。
> 4. 對 schedule_outsource 中「款號名稱」與「顏色」欄位疑似錯位的批次，是否仍能正確識別訂單並驗證規則（驗證依「款號」欄位代碼進行）。

### 情境 #7：產能超載驗證（R5）

**輸入描述**

> 使用者要求檢查 SL01 在 4 月份是否有產能超載。

**預期輸出**

> 對 schedule_gantt 中廠=三達榮、組=SL01、計畫起訖日落在 2026-04-01 ~ 2026-04-30 的批次，依「計畫開始 ~ 計畫結束」展開到每日，加總同日生產數量，與該批次標準日產能比對。
> 若同日加總超過標準日產能，列為 R5 產能超載問題。

**驗證重點**

> 1. 「展開到每日」邏輯是否正確（一筆批次計畫從 4/3 ~ 4/7，需展開為 4/3、4/4、4/5、4/6、4/7 各一份）。
> 2. 同日多筆任務並存時，是否正確加總所有批次的生產數量。
> 3. 標準日產能比對基準是否正確（每筆批次的標準日產能應一致或取最大者，需於回覆中說明採用的判斷）。
> 4. 問題描述應指出具體日期與超載數量（例如「4/8 同日生產加總 1800 件，超過標準日產能 1500 件」）。

### 情境 #8：無問題回覆（負向測試）

**輸入描述**

> 使用者指定一個經人工確認排程完全正常的範圍（例如某個小組別的單一週）。

**預期輸出**

> 僅回覆：「本次檢查範圍內未發現排程異常，排程狀態正常。」不應額外輸出表格或推理過程。

**驗證重點**

> 1. 是否完全省略表格，僅輸出單一句子。
> 2. 是否避免畫蛇添足列出「無問題」之類的空表格。
> 3. 語氣是否一致（與其他情境的回覆風格相符）。
