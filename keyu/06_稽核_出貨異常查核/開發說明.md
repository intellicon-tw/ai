# Agent 06 — 出貨異常查核助手（稽核部 / 林育秀）

## 一、可行性
**可做但有重要限制 — 強烈建議搭配程式驗證。** 純 LLM 做四表交叉比對在高頻、高精度需求下容易漏判。本 Prompt 設計為「單一預收單 / 小批次」使用，**不建議直接拿來做大批量稽核**。

## 二、模型建議
- 預設：`claude-sonnet-4-6`（結構化比對能力佳）
- 若精度更重要：`claude-opus-4-6`

## 三、知識庫設定

| 知識庫名稱 | 來源檔案 | 狀態 |

| TP0003-45 預收表 | TP0003-45_預收表.csv | ✅ 已備齊 |

| TP0003-45 出貨紀錄 | TP0003-45_出貨紀錄.csv | ✅ 已備齊 |

## 四、結構性限制與替代方案

### 限制 1：LLM 在大量結構化比對的可靠性
原任務每月 300 次比對，純 LLM 容易：
- 漏看部分欄位
- 在欄位多（30+）的表上產生幻覺
- 同一筆查兩次可能給不同結果

**建議三段式做法：**

| 階段 | 工具 | 工作內容 |
|------|------|---------|
| 1 | Python 程式（最可靠） | 用 pandas 做四表 join，產出原始異常清單 CSV |
| 2 | EgentHub Agent | 把異常清單上傳，請 Agent 寫成人類可讀的稽核摘要 |
| 3 | 稽核同仁 | 對「⚠️ 待人工確認」項目做最終判斷 |

如果客戶不接受外部程式，可以先用本 Agent 處理「單張預收單細查」這種低量場景，大批比對再說。

### 限制 2：30+ 欄位的 EIP 全紀錄
EIP 全紀錄有 30 多個欄位，可能超出 Token 限制。**建議**：在上傳前先精簡 EIP 表，只保留比對需要的欄位（客戶編號、機號、品名、日期、出貨單號、發票號碼、數量、金額），其他欄位另存備查。

### 限制 3：預收表標題行格式
預收表第一行是「TP0003-45」「師大科教中心404室」這種標題列，不是欄位名。在做成表格查詢知識庫前，需把標題行移除、以第 2 行的「預收單號、日期、品名…」作為欄位名。

## 五、待釐清事項
1. 服務卡格式為何？欄位有哪些？
2. 是只稽核 TP0003-45 一個客戶，還是會擴展到所有客戶？若是後者，建議改用 Python pipeline。
3. 異常認定的「容差」是多少？例如金額差幾元算異常？品名相似度多少算一致？

## 六、驗測情境建議
1. 「比對 KE20190700448 這張預收單」→ 預期：Agent 在四表查找對應，產出明細。
2. 「比對 2019 年 10 月所有預收單」→ 預期：Agent 給出比對結果，若超過 50 筆建議分批。
3. 「找出本月所有金額不一致的紀錄」→ 預期：產出「⚠️ 金額不一致」清單。
4. 故意修改一筆預收表的金額 → 預期：Agent 標示為金額不一致。
