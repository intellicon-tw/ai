# Agent 05 — 差旅費審核助手（財務部 / 張雅棉）

## 一、可行性
**部分可做。** PDF 抽取 + 公式計算可以做。卡關在「Google Map 即時里程計算」EgentHub 平台目前無原生工具，本 Prompt 採用「歷史平均里程 ± 容差」作為替代方案。

## 二、模型建議
- 預設：`claude-sonnet-4-6`（PDF 視覺理解＋表格輸出最穩）
- 若要更高品質：`claude-opus-4-6`

## 三、兩階段執行機制
此任務同時包含「讀取 PDF」+「調用知識庫」，已設計嚴格的兩階段：
- 步驟一：只讀 PDF，產出對照表
- 步驟二：基於對照表查知識庫、計算金額

## 四、知識庫設定

| 知識庫名稱 | 類型 | 狀態 |
|-----------|------|------|
| 差旅費計算公式表 | 全文查找 | **待張雅棉補**（附範本） |
| 歷史平均里程基準 | 表格查詢 | **待張雅棉補**（附範本） |

## 五、結構性限制與替代方案

### 限制 1：無原生 Google Map 距離 API
原始任務需求是「以 Google Map 核算合理里程」。三個替代方案：

| 方案 | 做法 | 適用情境 | 開發成本 |
|------|------|---------|---------|
| A. 歷史平均（本 Prompt 採用） | 建「歷史平均里程基準」表，比對 ±10 KM | 業務拜訪路線重複性高 | 低，立即可上線 |
| B. MCP 串 Google Maps API | 請 IT 開發 MCP，呼叫 Google Maps Distance Matrix | 想要精確 | 中高，需 IT 介入 |
| C. 人工確認 | Agent 只做表格抽取與公式計算，里程合理性由會計肉眼判斷 | 案件量少 | 0 |

### 限制 2：任務頻率與 ROI
此任務每月 1 次、2 人需求，總節省工時約 96 小時/月（2 人 × 48 小時）。優先順序可放在 Agent 03、04 之後。

## 六、待釐清事項
1. 公司差旅費補助公式具體為何？（每公里多少元？汽機車各別？停車費上限？）
2. 業務同仁的拜訪路線是否有「常用路線清單」？有的話可直接做成歷史基準。
3. 油資補助申報單與業務工作日誌是否為固定格式 PDF？若不是，PDF 抽取的穩定性會受影響。

## 七、驗測情境建議
1. 上傳一份「正常的申報單 + 日誌」 → 預期：無異常標示，補助金額計算正確。
2. 上傳一份「故意把申報里程改 +30 KM」 → 預期：標示為異常。
3. 路線是新拜訪客戶（無歷史基準）→ 預期：標示為「無基準，待人工確認」而非「異常」。
