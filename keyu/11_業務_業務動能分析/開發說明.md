# Agent 11 — 業務動能分析（業務部 / 黃子濬）

## 一、可行性
**部分可做。** 對照表整理、異常判讀、嚴重度分級、主管報告產出皆可以做。卡關有三：
1. EIP 服務卡無 API（同 Agent 01、08），需手動匯出或人工貼入。
2. 點與點之間的車程合理性，無原生 Google Map 工具（同 Agent 05），改以歷史平均車程 ± 容差替代。
3. 「發送訊息給主管」EgentHub 平台無原生 LINE／Email MCP，需另外串接或改為主管自行進入 Agent 檢視。

## 二、模型建議
- 預設：`claude-sonnet-4-6`（兩階段執行 + 多表格交叉比對最穩）
- 若要更高品質：`claude-opus-4-6`

## 三、知識庫設定

| 知識庫名稱 | 類型 | 狀態 |
|-----------|------|------|
| 預排行程表 | 表格查詢 | **待黃子濬補**（建議從 EIP 預排行程匯出 CSV） |
| EIP 服務卡 | 表格查詢 | **待黃子濬補**（每日由業務同仁匯出或定期同步） |
| 業務工作日誌 | 表格查詢 | **待黃子濬補**（建議統一日誌欄位格式） |
| 歷史平均車程基準 | 表格查詢 | **與 Agent 05 共用**（若 Agent 05 已建好，此處共用即可） |
| 異常判讀規則 | 全文查找 | **附範本**（`知識庫/_範本_異常判讀規則.txt`），請黃子濬與業務團隊共同調整閾值 |

## 四、結構性限制與替代方案

### 限制 1：EIP 服務卡無 API
原始任務需求是「AI 自動抓取上傳 EIP 服務卡」。三個替代方案：

| 方案 | 做法 | 適用情境 | 開發成本 |
|------|------|---------|---------|
| A. 手動匯出貼入（本 Prompt 預設） | 業務同仁或主管每日將 EIP 服務卡內容貼入對話框 | 案件量少、初期驗證 | 0 |
| B. 定期同步知識庫 | 設定每日定時把 EIP 服務卡匯出 CSV 上傳至「EIP 服務卡」表格知識庫 | 案件量穩定 | 低 |
| C. MCP 串 EIP API | 請 IT 評估 EIP 是否可開放 API，由 MCP 即時抓取 | 案件量大、要即時 | 中高，需 IT 介入 |

### 限制 2：無原生 Google Map 車程 API
原始邏輯需「核對點與點之間的車程時間」。三個替代方案：

| 方案 | 做法 | 適用情境 | 開發成本 |
|------|------|---------|---------|
| A. 歷史平均（本 Prompt 採用） | 建「歷史平均車程基準」表（與 Agent 05 共用），比對 ± 50% 區間 | 業務拜訪路線重複性高 | 低 |
| B. MCP 串 Google Maps Distance Matrix | 請 IT 開發 MCP，即時取得車程 | 想要精準 | 中高 |
| C. 人工確認 | Agent 只標示「車程偏離歷史基準」，由主管自行用 Google Map 核對 | 案件量少 | 0 |

### 限制 3：發送訊息給主管
原始邏輯需「發送訊息給主管」。三個替代方案：

| 方案 | 做法 | 適用情境 | 開發成本 |
|------|------|---------|---------|
| A. 主管自行檢視（本 Prompt 預設） | 主管每日進入 Agent 查詢，由 Agent 產出報告 | 主管習慣主動查詢 | 0 |
| B. LINE Notify MCP | 開發 LINE Notify MCP，異常達高嚴重度時自動推播給主管 | 主管想被動接收 | 中 |
| C. Email MCP | 開發 Email MCP，每日定時寄送異常報告給主管 | 主管習慣 Email | 中 |

## 五、異常判讀規則建議

系統預設應偵測下列異常（詳見 `知識庫/_範本_異常判讀規則.txt`）：

| 異常類型 | 判讀條件 | 預設嚴重度 |
|---------|---------|----------|
| 停留時間過長 | 實際停留 > 預估停留 × 1.5 | 中 |
| 車程時間過短 | 實際車程 < 預估車程 × 0.5 | 高（疑似未實際前往） |
| 車程時間過長 | 實際車程 > 預估車程 × 1.5 | 中（可能塞車或繞路） |
| 空白時段過久 | 連續 > 90 分鐘無服務卡與日誌紀錄 | 中 |
| 服務卡 vs 日誌不一致 | 同一時段地點／客戶不符 | 高 |
| 無預排臨時拜訪 | 服務卡有紀錄但預排行程無 | 低（中性，可能為合理臨時加單） |
| 超過合理工時 | 日誌或服務卡涵蓋時段 > 12 小時 | 低（提醒關懷） |

閾值（1.5 倍、0.5 倍、90 分鐘等）皆為預設值，建議由黃子濬與業務團隊試運行 1–2 週後共同調整。

## 六、敏感性提醒

此 Agent 屬「監督性 Agent」，業務同仁知道後可能造成抗拒或士氣影響。導入前建議：

1. **先與業務團隊溝通定位**：明確告知此工具是「協助主管掌握團隊動能、優化資源分配」，**不是抓人或考核工具**。
2. **報告統一標示「待人工確認」**：所有異常項目皆不直接定性為失職，由主管實際與業務同仁對話釐清。
3. **嚴重度分級謹慎使用**：高嚴重度不等於違規，僅代表「值得主管優先了解」。
4. **避免擴大用途**：不建議將報告直接連動到績效考核、薪酬計算，以免引發信任問題。
5. **試行期透明**：建議第一個月由黃子濬主動向業務同仁說明試行範圍與用途。

## 七、待釐清事項

1. EIP 服務卡的匯出格式為何？是否含「實際到達／離開時間」欄位？若只有當日彙整時段，車程合理性將難以判讀。
2. 業務工作日誌的填寫粒度為何？（每小時一筆？每客戶一筆？自由填寫？）
3. 歷史平均車程基準是否要與 Agent 05 共用一份表？若共用，需協調張雅棉與黃子濬的欄位定義。
4. 主管是否需要每日自動收到報告？若需要，需評估 LINE Notify 或 Email MCP 的開發優先序。

## 八、驗測情境建議

1. **正常一日** — 上傳一份完整、無異常的服務卡 + 日誌 → 預期：無異常標示，行程對照表完整。
2. **車程過短** — 故意把兩個遠距客戶的車程改為 10 分鐘 → 預期：標示為「車程時間過短／高嚴重度／待人工確認」。
3. **空白時段** — 服務卡缺一段下午時段、日誌也未填 → 預期：標示為「空白時段過久／中嚴重度」。
4. **服務卡 vs 日誌不一致** — 同一時段地點不同 → 預期：標示為「不一致／高嚴重度」。
5. **歷史無基準路線** — 新拜訪客戶、車程基準表中沒有資料 → 預期：標示為「無歷史車程基準，待人工確認」而非異常。
