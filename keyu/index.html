二、建議上線順序

階段 1（本週可上線）

  • Agent 03 — 印表機耗材詢答：資料齊全、客戶端不需要再補資料，可作為首發示範。

階段 2（補完知識庫後 1–2 週可上線）

  • Agent 04 — 電商文案 SEO：待王鈺婷補品牌語氣指南、3–5 篇成功文案範例。
  • Agent 02 — 行銷數據分析師：待蘇郁婷確認數據欄位、上傳當月銷售流量資料。

階段 3（需 IT 或客戶協作）

  • Agent 01 — 設備排查客服助手：需 IT 匯出保固清單、客服部彙整故障代碼對照表、上傳維修手冊。
  • Agent 05 — 差旅費審核：採「歷史平均里程」替代方案可上線；若要精確 Google Map 距離需開發 MCP。
  • Agent 06 — 出貨異常查核：建議先用 Python 做大批量比對，Agent 只做「單張預收單細查」+「人類可讀稽核摘要」。
  • Agent 07 — 招募流程自動化：建議拆解後做履歷篩選＋信件草稿＋測驗分析三件事，104 邀請維持人工。

三、共通限制（EgentHub 平台層級）

以下限制對所有 Agent 都適用，與單一 Agent 設計無關：

  1. 無法直接產出 .pptx／.xlsx 檔案：可輸出 Markdown 表格，使用者在 EgentHub 介面下載 .xlsx。
  2. 對話框不支援上傳 .xlsx／.csv：替代方案為改用「表格查詢知識庫」、貼上文字、或上傳 PDF/TXT。
  3. 單次上下文約 20K tokens：資料過大時需先摘要或分批處理。
  4. 無原生 Google Map／日曆／104 整合：需透過 MCP 或外部工具串接。

四、檔案結構說明

每個 Agent 子資料夾包含： - prompt.md — 可直接複製到 EgentHub 後台的系統提示詞（已符合 Prompt Designer 規範） - 開發說明.md — 該 Agent 的可行性、知識庫設定、待補項、驗測情境 - 知識庫/ — 該 Agent 需要的知識庫檔案（已實際備齊的為 CSV；待補的為 _範本_ 開頭的範本檔，由客戶覆蓋）

五、後續行動建議

  1. 與睿朱對齊：把這份盤點結果與睿朱、客戶各提案部門代表確認可行性。
  2. 資料蒐集：請王鈺婷／蘇郁婷／張雅棉／林育秀／黃靖涵／客服部各自補齊知識庫資料。
  3. 先上線 Agent 03：作為示範樣板，讓客戶感受到 EgentHub 的速度與品質。
  4. 每週迭代：每個 Agent 上線後，蒐集 5–10 個實際對話樣本，依測試結果微調 Prompt。

四、結構性限制與替代方案

限制 1：EIP 系統無 API

原始任務希望「自動查詢機台保固狀態」，但 EIP（KYOCERA 5520）系統未開放 API。 - 短期方案（建議）：由 IT 每週／每月匯出保固清單 CSV，上傳至 EgentHub 表格查詢知識庫。 - 長期方案：請 IT 開發 EIP 查詢 API，再透過 EgentHub MCP 串接。

限制 2：維修手冊 PDF 體積大

若維修手冊單檔超過 30 頁，建議拆分為「常見故障速查表」（10–20 頁濃縮版）與完整版分開存放，常用查詢以濃縮版為主，提升回應速度。

五、待釐清事項

  1. 「過保先報價」的報價邏輯 — 是固定價格？依機種？依故障類別？
  2. 哪些故障是「現場可自助處理」？哪些是「一定派工」？需要 SOP 邊界。
  3. 客服同仁工作流程：是把客戶訊息直接貼進 Agent，還是有結構化欄位？

六、驗測情境建議

  1. 客戶提供「機台序號 A7PU011020147AM，印出來都是黑線」→ 預期：查保固→若已過保給報價，若保固中給排查步驟。
  2. 客戶情緒激動「印不出來我明天要交件！」→ 預期：先安撫，再請補機台序號。
  3. 客服輸入不完整（沒給序號）→ 預期：禮貌請補。

四、結構性限制

限制 1：104 自動發送邀請（無解）

104 人力銀行不開放第三方 API發送邀請。三個方案： - A. 接受限制（建議）：保留人工發送 104 邀請，AI 只做履歷收進來後的「篩選」與「後續流程加速」。 - B. 瀏覽器自動化（不建議）：用 Selenium/Playwright 模擬登入發送 — 違反 104 服務條款，且 EgentHub 不在範圍內。 - C. 切換管道：若量大，考慮直接刊 LinkedIn 或自架徵才頁，較有 API。

限制 2：手繪四象限測驗無法 AI 評分

上傳的測驗圖（image2.png）是手繪填寫的紙本表單。Claude 對手寫四象限： - 任務導向 C、人際導向 D 軸線 - 慢步調 A、快步調 B 軸線 - 四個分數寫在手寫稿上（26／31／22／21）

手寫 OCR 容易出錯，建議： - 改用 Google Form 計分（測驗連結已存在）：自動算分後分數直接給 AI 分析。或 - 改用結構化表格：應徵者填表後人工 keyin 四個分數，再交給 AI。

本 Prompt 設計為「已數位化分數」版本，不負責讀手寫稿。

限制 3：面試預約需要日曆 API

原任務希望「自動預約面試時間」。EgentHub 目前無原生日曆整合，建議： - 用「Calendly 共享連結」加在面試確認信中，候選人自己選時段。 - 之後再接 Google Calendar MCP（IT 開發）。
