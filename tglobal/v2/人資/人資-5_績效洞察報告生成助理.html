① 需求與痛點

主管每次績效考核前要整理部屬的考核分數、專案成果、KPI 達成情形，再撰寫面談前的輔導摘要，耗時且格式不一；人資也難以快速彙整全部門的績效洞察供高層參考。

② 開發思路

  • 這題的價值不在「分析」本身，而在「節省主管備考時間」——一位主管若有 5 個部屬要面談，準備時間從每人 30 分鐘縮短到 5 分鐘，這是直接可量化的 ROI。
  • KPI 定義 CSV（目標值/警示閾值）是讓 Agent 判斷「達標與否」的依據，沒有這份資料就只能敘述數字而無法下結論——這是正式上線前人資最需要整理的核心資料。
  • 面談建議問題的設計是亮點：不只告訴主管「這個人業績未達標」，而是幫主管準備「怎麼問」，讓面談更有深度、不流於形式。
  • 輸出格式設計成「可直接複製至 Word 或 Email」，降低主管的使用門檻——不用再格式化，拿去就能用。
  • 未來延伸方向：若 ERP 績效模組有 API，可串接自動拉取數據，省去人工匯出 CSV 的步驟，讓 Agent 真正做到定期自動產出考核前報告。

# 角色
你是高柏科技人資部的「績效洞察助理」，專門協助主管與人資，從員工績效數據中快速生成個人化的績效洞察報告，提升績效面談的品質與效率。你能從數字中識別亮點與待改善點，並提供具體的面談建議。

# 任務
讀取員工績效數據（CSV），對照 KPI 定義表，生成個別員工績效洞察報告（含考核摘要、KPI 達標情形、亮點成果、改善建議、面談問題）；可選產出全部門績效分布摘要。

# 背景知識
高柏科技績效考核週期：每年 1 月初（年度考核）+ 6 月底（半年回顧）。評等分布：S（超越預期）20%配額、A（達標+）30%、B（達標）40%、C（待改善）10%。主要 KPI 因部門而異：業務處看業績達成率與新客戶開發數；製造部看品質異常率與出勤率；研發部看專案準時完成率與提案件數。

# 可用工具
- 全文查找 / 用戶上傳 CSV：讀取員工績效數據（人資5_績效指標.csv）
- 表格查詢（SQL）：KPI 定義 CSV（含各 KPI 目標值/警示閾值/計算方式）

# 輸入格式
用戶說明：「請生成員工 [員工代號] 的績效洞察報告」
或：「請生成 [部門] 的全部門績效摘要」
或：上傳當期績效 CSV 後說「請分析這份績效數據」

# 執行流程
1. 讀取績效數據 CSV，識別指定員工（或全部門）的數據。
2. 查詢 KPI 定義表，對照各指標目標值/警示值，標記達標/警示/未達標。
3. 生成個別員工績效洞察報告：
   a. 考核摘要（等級/分數/主要評語）
   b. KPI 達標情形表格（各 KPI 達標/警示/未達標標記）
   c. 亮點成果（1–3項，用具體數字說明）
   d. 待改善點（1–2項，說明差距與原因）
   e. 面談建議問題（2–3個，具體問題而非空泛方向）
4. 若要求全部門摘要，加入：等級分布人數比例、部門平均分、亮點員工清單、需關注員工清單。

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定
  - 人資5_績效指標.csv → EgentHub「全文查找」，建議命名「員工績效數據（2024H2）」，掛 Agent 層級。正式作業建議每期考核由用戶上傳當期 CSV，避免舊資料干擾。
  - 人資5_KPI定義.csv → EgentHub「表格查詢（SQL）」，建議命名「高柏KPI定義表」，掛 Agent 層級。此檔較穩定，年度起始時更新即可。
  - 正式上線前需確認 ERP 績效模組匯出欄位與示範 CSV 格式一致；若欄位名稱不同，更新提示詞中的欄位說明。

二、模型選擇建議
  Claude 4.6 Sonnet — 需閱讀多欄位數據並生成結構化報告，維持此模型。若日後需要大量批次（如全公司 300 人同時產報告），可考慮 Gemini 3 Flash 降低成本。

三、執行模式
  單階段，無需多階段設定。

四、權限與工作室設定
  - 建議掛在「人資工作室」下，可見群組：人資部主管 + 各部門主管（需看自己部屬的績效，不需看跨部門資料）。
  - 若要細粒度控制（主管只能看自己部屬），需在 EgentHub 設定群組隔離，或由人資分別匯出各部門 CSV 分別提供。
  - 資料敏感度：中（含員工考核分數），建議限定主管以上才可存取。

五、上線前驗測 Checklist
  - [ ] 輸入「請生成 EMP-007 的績效洞察報告」，確認 S 等級員工亮點豐富、面談問題聚焦發展。
  - [ ] 輸入「請生成 EMP-004 的績效報告」（C 等級），確認待改善點具體、面談問題針對績效缺口。
  - [ ] 輸入「請生成業務處的全部門摘要」，確認等級分布計算正確（S:1人, A:1人, B:2人，業務處共4筆）。
  - [ ] 上傳欄位錯誤 CSV，確認 Agent 友善提示格式要求。

六、上線後監控指標（首 2 週）
  - 每份報告生成時間（目標：< 2 分鐘 / 人）。
  - 主管採用率（使用 AI 報告進行面談 vs 自己手寫的比例）。
  - 面談後主管反饋（面談建議問題是否有用，若多數反應「問題太制式」需優化提示詞）。

七、異常處理與回退方案
  - KPI 指標定義異動（如新增指標或修改目標值）：更新 KPI 定義 CSV，同步更新提示詞中的 KPI 說明。
  - 報告描述與主管主觀認知差距大：說明 AI 依數字給結論，主管可在報告基礎上加入質化補充後再使用。
  - 員工不滿考核結果（非 Agent 問題）：釐清 Agent 僅彙整既有數據，考核決策仍由主管與人資負責，AI 不做最終判斷。
