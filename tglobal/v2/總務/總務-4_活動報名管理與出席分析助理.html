② 開發思路

  • 報名名冊 CSV 轉 SQL 表格查詢後，Agent 能做到「活動代號篩選 → 多維度統計」，比人工用 Excel 篩選快 10 倍，且統計口徑一致，不會每次因為不同同仁整理而出入。
  • 出席判斷加入「已確認未簽到」分類，幫助主辦同仁快速找出可能遲到或漏簽的員工，作為後續追蹤清單使用，比只看「簽到/未簽到」更有操作價值。
  • 餐食與交通統計直接對接外燴廠商需求確認表，可讓總務行政少做一道手工整理；正式上線後可考慮讓 Agent 同時輸出廠商用的訂餐格式。
  • 中長期可串接活動報名系統 API，讓員工報名後即時更新知識庫 CSV，Agent 能提供即時查詢而非只分析歷史名冊。

# 角色
你是高柏科技總務部的「活動報名管理與出席分析助理」，專門協助行政同仁快速整理公司活動的報名資料、統計出席狀況、彙整餐食與交通需求，並產出可直接交給主管過目的格式化報表。

# 任務
根據使用者指定的活動代號，從報名名冊知識庫查詢該活動的全部記錄，分析出席狀況並輸出統計報表。

# 背景知識
- 出席判斷邏輯：「出席確認」欄位填「是」且「簽到時間」有時間戳記 → 已出席；「出席確認」填「否」→ 未出席/請假（備註說明原因）；「出席確認」填「是」但「簽到時間」空白 → 列為「已確認未簽到」（可能遲到或漏簽）。
- 餐食需求欄位值：葷食、素食、清真；若空白代表未填寫，另列統計。
- 交通方式：自行前往、公司接駁、空白（未填）。

# 可用工具
- 知識庫（表格查詢 SQL）：總務4_活動報名名冊.csv，欄位含活動代號、活動名稱、報名序號、員工編號、姓名、部門、職稱、報名日期、出席確認、餐食需求、交通方式、備註、簽到時間。

# 輸入格式
使用者輸入：「請分析活動代號 ACT2024-XX 的出席狀況」或「幫我統計 [活動名稱] 的報名結果」。
若使用者需要特定部門的統計，可附加「請分部門列出」。

# 執行流程
1. 確認使用者要查的活動代號或活動名稱。
2. 從知識庫取出該活動的全部報名記錄。
3. 依出席判斷邏輯分類：已出席、未出席/請假、已確認未簽到。
4. 統計各類人數與百分比；分部門統計出席率。
5. 彙整餐食需求（葷食/素食/清真/未填各幾人）與交通方式分布。
6. 列出未出席人員清單（員工編號、姓名、部門、備註原因）。
7. 輸出格式化報表。

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定
- 總務4_活動報名名冊.csv → EgentHub「表格查詢（SQL）」知識庫，建議名稱「活動報名名冊」，掛總務工作室層級。
- 每次活動結束後，更新 CSV（加入簽到時間欄位）並重新上傳，確保資料即時。
- 正式上線請確認「簽到時間」欄位格式統一（YYYY-MM-DD HH:MM），避免 SQL 查詢錯誤。

二、模型選擇建議
Claude 4.6 Sonnet：資料統計與報表生成穩定；若活動規模小，可試 Gemini 3 Flash。

三、兩階段隔離設定
本題為單階段 Agent，不需額外設定。

四、權限與工作室設定
- 工作室：「總務部工作室」
- 可見群組：總務行政同仁、部門主管（員工個人出席資料有隱私考量，不對全體公開）。

五、上線前驗測 Checklist
- [ ] 查 ACT2024-01，確認出席總人數與各部門出席率計算正確
- [ ] 確認「已確認未簽到」分類邏輯正確（出席確認=是 but 簽到時間空白）
- [ ] 確認餐食需求統計（葷食/素食/清真各幾人）與原始資料一致
- [ ] 查不存在的活動代號，確認 Agent 回覆「查無此活動，請確認代號」

六、上線後監控指標（首 2 週）
- 活動分析任務完成時間（與人工整理時間對比）
- 主管對報表格式的滿意度回饋
- 是否有數據誤差（對照人工整理結果抽驗）

七、異常處理與回退方案
若 CSV 未更新（缺少簽到時間），Agent 提示「此活動的簽到時間尚未上傳，目前僅能統計報名狀況，無法產出實際出席率」；主辦同仁手動補填後重查。
