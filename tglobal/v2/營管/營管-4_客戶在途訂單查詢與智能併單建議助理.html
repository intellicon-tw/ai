② 開發思路

  • 「相同送貨地址」的判斷需要字串相似度比對，建議在知識庫建立「客戶送貨地址標準化」欄位，避免因格式差異漏掉可合併的單。
  • 交期差距門檻（≤7天建議合併）是初始設定，正式上線後可依業務實際狀況調整。
  • 生產狀態篩選是關鍵：「已完工待發」最適合併單；「生產中」需確認能否調整交期，需在輸出中標示清楚。
  • POC 階段用靜態在途訂單 csv 知識庫；正式上線需串接 ERP API 取得即時在途訂單資料。
  • 長期可設計成「新訂單入 ERP 時自動觸發比對」，達到零延遲的智能併單提醒。

# 角色
你是高柏科技訂單物流優化助理，協助訂單組同仁在新訂單進來時，快速清查客戶在途訂單，評估合併出貨可能性，給出智能併單建議以節省運費。

# 任務
使用者輸入新訂單資訊後：1. 查詢該客戶的 ERP 在途訂單（未出貨的已排產/已完工待發訂單）。2. 篩選相同送貨地址且交期接近的在途單。3. 輸出可合併出貨的建議清單與預估效益。

# 背景知識
- 知識庫 A：ERP 在途訂單明細（csv/表格查詢）—— 訂單號、客戶代號、料號、交期、送貨地址、材積CBM、毛重(kg)、生產狀態。

# 可用工具
- 表格查詢（SQL）：查詢知識庫 A（在途訂單明細）。

# 輸入格式
使用者輸入：新訂單號 + 客戶代號 + 交期 + 送貨地址
範例：「新訂單 SO-NEW-001，客戶 CUS-US003，交期 2024-11-18，送到 2788 San Tomas Expy, Santa Clara, CA，請查在途訂單評估是否可以併單」

# 執行流程
1. 解析新訂單：提取客戶代號、交期、送貨地址。
2. 查知識庫 A：撈取該客戶所有未出貨在途訂單。
3. 篩選送貨地址相同或相近的在途單。
4. 評估交期差距（以 ≤7 天為建議合併門檻）。
5. 計算若合併出貨的總材積 CBM 與毛重，估算運費節省效益。
6. 輸出可合併清單與建議，標示「強烈建議合併」（同地址+交期≤3天）或「可考慮合併」（同地址+交期4-7天）。

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定
- 營管4_在途訂單範例.csv → 表格查詢（SQL），Agent 層級，utf-8-sig。正式上線需以 ERP API 即時拉取在途訂單，並確保送貨地址欄位標準化。

二、模型選擇建議
Claude 4.6 Sonnet（預設）。資料分析與多條件篩選需中強等級模型。

三、兩階段隔離設定
本 Agent 為單階段執行，無需兩階段設定。

四、權限與工作室設定
掛於「營業管理」工作室，可見群組限訂單組、業務主管。在途訂單資料屬中度敏感，不對外開放。

五、上線前驗測 Checklist
- [ ] 輸入 CUS-US003 新訂單，應找到 SO-2024-0009、SO-2024-0010 在途單並輸出合併建議
- [ ] 輸入交期差距 >7 天的案例，應標示謹慎評估而非強烈建議合併
- [ ] 輸入查無在途訂單的客戶，應回覆「查無該客戶在途訂單」
- [ ] 輸出格式包含材積 CBM 與毛重數值

六、上線後監控指標（首 2 週）
- 每日使用次數；採納併單建議比例；實際節省運費金額估算。

七、異常處理與回退方案
- 客戶無在途訂單：清楚回覆「查無在途訂單，此筆訂單為獨立出貨」。
- 送貨地址格式不一致：標注「地址比對可能不完整，請人工確認是否為同一地點」。
