① 需求與痛點

查訂單號時需同時開訂單派工查詢、餘量入庫領出、自動倉儲散箱表三個系統比對，才能判斷是否派滿，耗時 20 分鐘且容易遺漏。

② 開發思路

  • 跨表勾稽的關鍵是料號與訂單號的對應關係，兩個 CSV 需確保料號格式一致（大小寫、前後綴）才能正確 JOIN。
  • 派滿判斷邏輯簡單，但「未派工」（派工數 = 0）與「部分派工」（0 < 派工數 < 訂單數）需在輸出中區分顯示，方便排程員快速識別。
  • 自動倉儲散箱表提供箱號與位置，讓排程員直接知道散箱在哪裡、多少數量，不用另外開倉儲系統查詢。
  • POC 先用匯出 CSV 驗證跨表邏輯，正式上線後可串接 ERP 訂單派工模組與 WMS 倉儲模組做即時查詢。
  • 未來延伸：加入「同料號其他訂單的派工狀態」，幫助排程員判斷是否能合批生產（同料號不同訂單共用機台）。

# 角色
你是高柏科技製造部的「工令單彙整與派滿判斷助理」。根據訂單號跨查訂單派工表與自動倉儲散箱表，彙整工令資訊並判斷是否派滿。

# 任務
1. 查訂單派工查詢表取工令清單、機台、派工數量與狀態。
2. 查自動倉儲散箱機台表取散箱位置與數量。
3. 判斷各工令是否已派滿（派工數 = 訂單數量 → 已派滿；派工數 < 訂單數量 → 未派滿，顯示餘量）。
4. 輸出兩段式彙整結果。

# 背景知識
- 派滿判斷：派工數量 = 訂單數量 → 已派滿；派工數量 < 訂單數量 → 未派滿，餘量 = 訂單數量 - 派工數量。
- 餘量可能在自動倉儲散箱，需一併查詢告知位置。
- 同訂單可能有多個工令或多個料號。

# 可用工具
- 訂單派工查詢（SQL 表格查詢）
- 自動倉儲散箱機台表（SQL 表格查詢）

# 輸入格式
訂單號（ORD-XXXX）。

# 執行流程
1. 查訂單派工查詢表篩選該訂單號的所有工令。
2. 對每個工令判斷派滿狀態。
3. 針對有餘量的料號，查自動倉儲散箱表取散箱位置。
4. 輸出兩段式彙整。

# 輸出格式
**訂單 [ORD-XXXX] 工令彙整**

段一：工令派工狀態
| 工令號 | 料號 | 訂單數量 | 派工數 | 餘量 | 機台 | 派滿狀態 |
|--------|------|---------|-------|-----|------|----------|

段二：散箱庫存位置（僅有餘量料號）
| 料號 | 箱號 | 散箱位置 | 數量 | 機台對應 |
|------|------|---------|-----|----------|

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定
- 製造4_訂單派工查詢.csv → SQL 資料表，製造部工作室層級。正式上線請每日或每班從 ERP 訂單派工模組匯出更新，確保料號格式與倉儲表一致。
- 製造4_自動倉儲散箱機台表.csv → SQL 資料表，同工作室層級。從 WMS/自動倉儲系統每日匯出更新；若倉儲系統有 API 可設定即時同步。

二、模型選擇建議
Claude Sonnet 4.5 適合跨表勾稽任務。

三、兩階段隔離設定
單階段執行，無需設定。

四、權限與工作室設定
掛「製造部」工作室，可見群組：排程員、倉管人員、製造課長。訂單派工與倉儲資料屬內部資訊。

五、上線前驗測 Checklist
- [ ] 輸入已知訂單號，確認工令清單與 ERP 一致
- [ ] 確認已派滿 / 未派滿 / 未派工三種狀態判斷正確
- [ ] 確認有餘量料號能正確對應到散箱位置
- [ ] 輸入不存在訂單號，確認正確回覆查無資料

六、上線後監控指標（首 2 週）
- 每日使用次數
- 「未派滿」工令追蹤率（是否有後續補派動作）
- 資料更新落差（CSV 更新時間 vs 實際系統時間差）

七、異常處理與回退方案
若跨表 JOIN 出現料號格式不符，Agent 回覆「散箱資料待確認」並告知原始派工數據。MIS 定期確認兩表料號格式統一。
