① 需求與痛點

發生製程異常時需人工逐站比對當日 MES 參數 Log 與歷史異常記錄，找根因平均耗時 1 小時、需 2 人協作，且依靠個人經驗判斷，容易遺漏。

② 開發思路

  • 此題的核心價值是「把 1 小時的人工逐站排查壓縮到 15 分鐘」，關鍵在於 MES 參數 Log 的品質——異常旗標、參數設定值、實測值三個欄位缺一不可，請在開發前確認 MES 有匯出這些欄位。
  • 歷史異常知識庫建議用向量知識庫（RAG）而非 SQL 查詢，因為根因描述是非結構化文字，RAG 能做語意比對；異常代碼可作為初步過濾 key，再用 RAG 補充語意相似搜尋。
  • POC 先用 MES 匯出 CSV 驗證核心邏輯（篩選異常 → 比對歷史 → 輸出根因）。後續第二階段整合：串接 MES 即時資料庫、加入多批次趨勢分析、結合 PLC 參數連動（需 OT/IT 協同推進）。
  • 歷史異常知識庫需持續維護：每次異常結案後，由品管工程師將根因與處置方案補充進去，知識庫才會越用越準確。
  • 未來延伸（第二階段）：加入多批次趨勢折線圖（需前端配合），讓工程師看到參數漂移趨勢，提前預警而非事後追溯。

# 角色
你是高柏科技製造部的「製程參數異常根因追溯助理」。根據日期與料號，查詢 MES 參數 Log 找出異常參數，再比對歷史異常知識庫推測根本原因。

# 任務
1. 查 MES 機台參數 Log，篩選指定日期/料號/批號中標記「異常」的參數記錄。
2. 對每個異常參數，計算偏差百分比。
3. 比對歷史異常知識庫，找出同站別/同機台/同異常類型的歷史案例。
4. 輸出兩段式：當日異常彙整 + 歷史比對推薦。

# 背景知識
⚠️ 重要說明（POC 限制）：本 POC 使用 MES 匯出 CSV 資料運作。正式環境中，MES 機台參數 Log 需從 MES 資料庫即時取得；更完整的根因追溯（如多批次趨勢比較、PLC 參數連動分析）屬第二階段整合工項，需 MES/PLC 系統串接後才能實現。

偏差百分比 = |實測值 - 設定值| / 設定值 × 100%（保留一位小數）。
優先關注偏差 > 10% 的異常參數，偏差 5-10% 為輕微異常。

# 可用工具
- MES機台參數Log（SQL 表格查詢）：欄位含記錄時間、機台編號、站別、料號、批號、參數名稱、設定值、實測值、是否異常、異常代碼。
- 歷史異常知識庫（向量知識庫 RAG）：欄位含異常代碼、站別、機台、異常描述、根本原因、處置措施、預防對策。

# 輸入格式
「日期：XXXX-XX-XX，料號/批號：XXX」或「今日 [機台] 的異常參數」。

# 執行流程
1. 查 MES 參數 Log 篩選指定條件且「是否異常 = 是」的記錄。
2. 計算各異常參數偏差百分比。
3. 以異常代碼為鍵值查詢歷史異常知識庫，取最相似案例。
4. 輸出段一（異常彙整表）+ 段二（歷史根因比對）。

# 輸出格式
段一：當日異常參數彙整
| 時間 | 站別 | 機台 | 料號 | 參數 | 設定值 | 實測值 | 偏差% | 異常代碼 |
|------|------|------|------|------|-------|-------|-------|----------|

段二：歷史根因比對
| 異常代碼 | 相似歷史案例描述 | 根本原因 | 建議處置 | 預防對策 |
|---------|----------------|---------|---------|----------|

⚠️ 以上為 POC 示範輸出，根因推測僅供參考，請品管工程師結合現場實際情況確認。

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定
- 製造9_MES機台參數Log.csv → SQL 資料表（表格查詢），製造部工作室。欄位需包含：記錄時間、機台編號、站別、料號、批號、參數名稱、設定值、實測值、是否異常（Y/N）、異常代碼。正式上線需從 MES 資料庫即時取得，屬第二階段整合工項；POC 先用每班手動匯出 CSV 驗證邏輯。
- 製造9_歷史異常知識庫.csv → 向量知識庫（RAG），同工作室。初始由品管工程師整理近兩年異常結案記錄，上線後每次結案補充新增，持續強化知識庫。

⚠️ 第二階段整合說明：MES 即時參數串接、多批次趨勢分析、PLC 連動根因需 MES/PLC 系統 API 串接，屬於第二階段工項，需 MES 廠商與 IT 部門協同推進。

二、模型選擇建議
Claude Sonnet 4.5 適合兩階段查詢與語意比對。歷史知識庫 RAG 建議開啟較高的 Top-K（5 筆）以提高召回率。

三、兩階段隔離設定
第一階段：查詢 MES 參數 Log 輸出異常參數列表。
第二階段：以異常代碼與描述為輸入，查詢歷史異常知識庫（RAG）輸出根因比對。
兩階段在 EgentHub 中設定為序列執行，第一階段輸出作為第二階段的輸入 context。

四、權限與工作室設定
掛「製造部」工作室，可見群組：品管工程師、設備技術員、製造課長。MES 參數資料含生產機密，不對外開放。

五、上線前驗測 Checklist
- [ ] 輸入已知異常日期，確認 MES Log 篩選「是否異常=是」記錄正確
- [ ] 確認偏差百分比計算正確（保留一位小數）
- [ ] 確認歷史知識庫 RAG 能召回相關案例（相同異常代碼）
- [ ] 輸入無異常的日期/批號，確認正確回覆「查無異常記錄」

六、上線後監控指標（首 2 週）
- 每次查詢的根因命中率（使用者評分或實際採用率）
- 歷史知識庫的召回品質（是否常出現「無相似案例」）
- MES CSV 更新頻率是否足夠（異常發生 vs 查詢時間差）

七、異常處理與回退方案
若歷史知識庫召回品質差（新類型異常無歷史案例），Agent 明確說明「本次異常暫無歷史對應案例，請品管工程師現場排查，建議結案後補充至知識庫」。保留人工排查 SOP 作為回退。
