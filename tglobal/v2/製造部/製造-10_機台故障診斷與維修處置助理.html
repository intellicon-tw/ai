① 需求與痛點

機台停機發警報時，維修人員需翻找紙本 PLC 手冊查報警碼、再查歷史維修工單判斷處置方式，平均 2 小時，且資深師傅不在時易延誤復機。

② 開發思路

  • PLC 報警碼對照表是快速查詢入口（結構化），維修手冊提供詳細步驟（全文），兩者組合讓維修人員既快（30 秒內知道要做什麼）又完整（有詳細 SOP）。
  • 維修手冊建議以全文查找（Full-Text）而非向量 RAG，因為維修 SOP 是精確步驟說明，不適合語意模糊比對；全文查找以章節標題定位更精準。
  • POC 先用報警碼 CSV + 手冊 TXT 驗證查詢與輸出格式。後續第二階段：串接 PLC → MES 自動推播，讓 Agent 在機台警報的同時主動推送診斷報告到維修人員手機或工作站。
  • 維修手冊需先做「數位化整理」——原廠 PDF 手冊轉成結構化 TXT（按機台系列、章節組織），是 POC 落地的前置準備工作，建議列為知識庫建置任務。
  • 未來延伸（第二階段）：串接歷史維修工單（維修時間、零件更換記錄），讓 AI 能比較「同報警碼歷史平均修復時間」，給出更精準的預計復機時間預測。

# 角色
你是高柏科技製造部的「機台故障診斷與維修處置助理」。根據機台編號與報警碼，查詢 PLC 報警碼對照表與維修手冊，輸出結構化故障診斷報告。

# 任務
1. 查 PLC 報警碼對照表取故障說明、嚴重等級、停機類型、第一處置步驟、聯絡對象、預計修復時間。
2. 查維修手冊（全文查找），取對應機台與故障類型的相關章節內容。
3. 組合輸出結構化診斷報告。

# 背景知識
⚠️ 重要說明（POC 限制）：本 POC 使用 PLC 報警碼 CSV 與維修手冊 TXT 運作。正式環境中，PLC 報警碼應由 PLC 直接推送至 MES，再由 MES 觸發 AI 診斷（自動 → 推播通知）；此「PLC → MES 自動觸發」屬第二階段整合工項，需 OT/IT 系統串接後才能實現。POC 先由維修人員手動輸入報警碼查詢。

嚴重等級：緊急（立即停機）→ 警告（限速/減速繼續）→ 一般（計畫停機）。

# 可用工具
- PLC報警碼對照（SQL 表格查詢）：報警碼/說明/嚴重等級/影響設備/停機類型/第一處置/聯絡對象/預計修復時間。
- 維修手冊摘錄（全文查找）：各機台故障排除步驟、SOP、保養週期等完整文件。

# 輸入格式
機台編號（WC-XXX）+ 報警碼（ALM-XXX）。或描述症狀（如「WC-101 主軸異音，轉速降低」）。

# 執行流程
1. 解析機台編號與報警碼。
2. 查 PLC 報警碼對照表取故障基本資訊。
3. 查維修手冊全文，找對應章節（依機台系列 + 故障類型）。
4. 組合輸出診斷報告。
5. 若輸入為症狀描述而非報警碼，根據症狀推測可能的報警碼並說明。

2. ...

⚠️ 本報告為輔助診斷參考，實際維修以設備廠商原廠手冊為準，請維修人員現場確認安全後再操作。

⑤ 建置流程與上線重點

正式上線前的準備指南
一、知識庫設定
- 製造10_PLC報警碼對照.csv → SQL 資料表（表格查詢），製造部工作室。正式上線請整合所有機台機型的 PLC 報警碼（依設備廠商手冊補充），確認「第一處置」欄位描述具體可執行（非「聯絡廠商」等模糊指引）。建議每次新增機台或設備改版時更新。
- 製造10_維修手冊摘錄.txt → 全文查找（Full-Text），同工作室。正式上線前需完成「維修手冊數位化整理」：將原廠 PDF 手冊轉為結構化 TXT，按機台系列（WC-101/102 切割機、WC-103/104 壓合機等）與故障類型（主軸、液壓、加熱、進給等）分章節整理，確保全文搜尋能精準定位章節。

⚠️ 第二階段整合說明：PLC 直接推送報警碼至 MES、再觸發 AI 自動診斷並推播給維修人員，需 PLC → MES → EgentHub API 整合，屬第二階段工項，需 OT/IT 協同推進。POC 先由維修人員手動輸入報警碼查詢。

二、模型選擇建議
Claude Sonnet 4.5 適合此全文查找與結構化輸出任務。緊急停機場景要求回應速度快，建議設定 timeout < 30 秒。

三、兩階段隔離設定
第一階段：查 PLC 報警碼對照表，輸出故障基本資訊（說明、嚴重等級、第一處置）。
第二階段：以機台系列與故障類型為條件，查維修手冊全文，取對應章節摘錄。
兩階段設定為序列執行，第一階段輸出確定故障類型後，第二階段再針對性查手冊。

四、權限與工作室設定
掛「製造部」工作室，可見群組：維修人員、設備工程師、機電組長、製造課長。維修手冊含設備機密，不對外開放。

五、上線前驗測 Checklist
- [ ] 輸入已知報警碼（如 ALM-001），確認說明、嚴重等級、第一處置輸出正確
- [ ] 確認維修手冊全文查找能定位到對應機台章節
- [ ] 輸入不存在報警碼，確認正確回覆「查無此報警碼」
- [ ] 確認緊急等級的輸出格式醒目（⛔ 標示清楚）

六、上線後監控指標（首 2 週）
- 每次使用與停機時間記錄，評估維修時間縮短效果
- 使用者反饋診斷報告的實際幫助程度
- 「查無報警碼」出現次數（可能是報警碼表不完整）

七、異常處理與回退方案
若 AI 診斷報告與實際故障不符，維修人員以原廠設備手冊為準。每次維修結案後，建議將實際根因與處置方案補充到歷史異常知識庫（第 9 題）與報警碼對照表中，形成持續改善的知識閉環。

Part 2 驗測情境
【緊急案例】
Input：機台：WC-101，報警碼：ALM-001
Expected：說明「主軸過電流」，⛔ 緊急，立即停機，第一處置：①切斷電源確認過載原因②量測電流③判斷刀具/軸承/進給問題。維修手冊摘錄第一章主軸系統相關段落（含正常電流範圍8-12A、過電流原因清單）。聯絡機電維修，預計30分鐘。
驗證點：報警碼說明正確，手冊章節定位到切割機系列，⛔標示醒目。

【警告等級可繼續案例】
Input：機台：WC-103，報警碼：ALM-002
Expected：說明「液壓壓力不足」，⚠️ 警告，限速繼續（非立即停機）。第一處置：確認液壓油位與油管。維修手冊第二章液壓系統相關段落。預計20分鐘。
驗證點：停機類型「限速繼續」非強制停機，正確區分緊急/警告。

【查無報警碼案例】
Input：機台：WC-101，報警碼：ALM-999
Expected：「PLC 報警碼對照表中查無 ALM-999，請確認報警碼是否正確，或此警報可能為新增設備代碼，建議聯絡設備廠商確認。」
驗證點：查無時不捏造，提供具體建議。

【症狀輸入案例】
Input：WC-101 主軸有異音，轉速下降
Expected：根據症狀推測可能為 ALM-001 主軸過電流或 ALM-012 軸承振動超標，建議先查 ALM-012（振動量測），並提供兩個報警碼的對比說明。
驗證點：症狀 → 報警碼推測邏輯合理，提供多個可能性而非唯一答案。
