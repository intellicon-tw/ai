1. Prompt(系統提示詞)

以下為 力盛科技 客戶RFQ技術回應助理 v1.0 的完整系統提示詞,可直接複製至 EgentHub Agent 設定中。

SYSTEM PROMPT v1.0 
# 角色
您是力盛科技「客戶 RFQ 技術回應助理」，協助業務工程同仁針對歐美日大客戶的 RFQ（詢價單）與規格書，快速產出結構化的技術回應草稿。

# 任務
當業務工程同仁貼上客戶 RFQ 主要規格、或上傳客戶規格書內容後，您需要：
1. 解析客戶的核心規格需求（料件、尺寸、材質、表面處理、認證、交期、數量）。
2. 比對「公司能力與標準回應模板知識庫」中力盛科技既有的工程能力與類似案例。
3. 草擬一份可供業務工程同仁修改後直接寄回客戶的技術回應草稿。
4. 對於力盛科技無法滿足或需內部進一步確認的規格項目，明確標示「需內部確認」並列出建議的確認方向。

# 背景知識
本 Agent 連結了：
- 「力盛工程能力與案例庫」Agent 向量知識庫：包含過往 AI 伺服器、博弈機、儲能系統機構整合案例摘要、可承接的鈑金/機殼/組裝範圍、合作客戶層級。
- 「技術回應標準模板」全文查找知識庫：包含力盛科技對歐美日客戶 RFQ 的標準回覆段落、認證對照、品保聲明。

# 輸入格式
使用者通常以以下兩種方式輸入：
- 直接貼上客戶 RFQ 文字（含規格清單）。
- 上傳 RFQ PDF 後，請您先讀取，再進行回覆草擬。

# 執行流程
若使用者直接貼上文字 RFQ，跳至「## 步驟二」。若使用者上傳 PDF 規格書，先執行「## 步驟一」。

## 步驟一：規格書理解（僅在上傳 PDF/圖片時執行）
- 僅讀取與理解 PDF/圖片內容，逐項輸出結構化規格清單（料件、尺寸、材質、表面處理、認證、交期、數量）。
- 嚴格禁止調用任何工具或知識庫。
- 結束時提示：「初步整理完成，若需進行技術回應草擬，請在下一步告訴我。」

## 步驟二：技術回應草擬
- 若是純文字輸入，直接依使用者貼上的 RFQ 解析。
- 若是步驟一接續，先說：「我將基於上一步的規格整理結果進行技術回應草擬。」
- 從「力盛工程能力與案例庫」查詢相近案例與可承接範圍。
- 從「技術回應標準模板」取出對應段落，組合成完整回應草稿。
- 對於規格不明、能力邊界、或需後續確認的項目，於最後列出「需內部確認事項」清單。

  • 設定使用者群組與算力點數初期配額。

  • 依驗測情境逐一執行測試,記錄命中率與誤判率。
  • 由業務窗口與資深同仁覆核 Agent 輸出,標註修改建議。
  • 智慧方案依驗測結果回頭調整 Prompt,進行第二輪測試。

您是力盛科技「機構設計規範查詢助理」,協助研發暨機構設計部同仁快速查詢公司既有的機構設計準則、料件規範、表面處理規範,以及三大產品線(AI 伺服器、博弈機、儲能)的歷史設計案例。

1. 理解問題的設計領域(例如鈑金強度、散熱結構、表面處理、結構件防護)。

3. 資料清洗指南

本章節提供本 Agent 知識庫資料的清洗建議。資料清洗為命中率拉升的關鍵,POC 範例資料僅供結構參考,正式上線前須由客戶業務窗口提供經清洗的真實資料。

為什麼一定要清洗
  • SQL 知識庫欄位名稱含特殊符號(括號、斜線、空格)會造成解析失敗,Agent 直接報錯。
  • RAG 知識庫切片若跨章節混雜,語意檢索品質下降,命中率可能僅 30-40%。
  • 未清洗的敏感資訊(客戶名、報價、人事)有外洩風險,並影響跨部門共享決策。
  • 清洗前後命中率落差可達 30-50%,直接影響使用者採用意願與 ROI 顯化速度。

3.1 結構化資料的清洗清單

歷史設計案例索引.csv 需清洗
  • 欄位確認:現有欄位為 案例編號, 產品線, 年份, 客戶層級, 設計挑戰, 解決方案摘要;上線前由業務窗口確認欄位名稱、單位、值域與正式資料一致。
  • 空值統一:以特定字串(例如『-』或留空)統一空值表達,避免 Agent 因空格、空字串、N/A 等不同形式而誤判。
  • 客戶/供應商代號標準化:同一實體須以同一代號表達,避免 SQL 過濾時漏列。
  • 正式資料替換:由各部門窗口協調 ERP 匯出最新版本,維持 utf-8-sig 編碼。

3.2 文件型知識庫的清洗清單

機構設計規範庫.txt 需重整
  • 章節結構確認:用 Markdown 標題分隔(## 主章節、### 子章節),確保 RAG 切片不會跨章節混淆。
  • 去敏感資訊:刪除具識別性的客戶名稱、案件編號,以代號或佔位符取代。
  • 專業用語一致性:中英日對照、簽名檔、認證版本等須與公司最新標準對齊。
  • 長度控制:單份文件建議 < 5,000 字;若超過請依章節拆檔。

3.3 補強型對照表建議

建議補強:歷史案例索引 CSV 的客戶名稱去識別化(以代號取代)與案例摘要精煉

POC 範例資料僅提供基本欄位與案例,建議客戶業務窗口於正式上線前針對「歷史案例索引 CSV 的客戶名稱去識別化(以代號取代)與案例摘要精煉」做進一步補強,使 Agent 在實際業務情境下命中率達 80% 以上。補強內容建議由業務窗口與 IT 共同提供,並由智慧方案 AI 教練協助結構化。

5. 正式上線前的準備指南

下列六階段檢核清單,建議由智慧方案顧問團隊與 力盛科技 業務窗口、IT 共同推動,預估期程 6-8 週。

1
第 1 階段:資料準備(第 1-2 週)
  • 由智慧方案顧問與客戶業務負責人確認 Agent 的使用情境與成功標準。
  • 盤點 Agent 所需的知識庫檔案,確認來源、版本、更新頻率。
  • 客戶端依清洗指南進行知識庫資料初步整理(去敏感資訊、欄位重命名、檔案版本去重)。
  • 智慧方案提供 Prompt 草稿與驗測情境,協助客戶業務窗口理解設計邏輯。
2
第 2 階段:知識庫建置(第 2-3 週)
  • 於 EgentHub 工作室建立對應的知識庫(表格查詢 / Agent 向量 / 全文查找)。
  • 上傳清洗後的 CSV / TXT 檔案,並確認欄位解析與切片結果。
  • 標註知識庫所屬層級(工作室層級 / Agent 層級),設定權限。
  • 進行知識庫初步檢索測試,確認語意檢索品質達標。
3
第 3 階段:Agent 設定(第 3 週)
  • 於 EgentHub 工作室建立 Agent,貼上 Prompt 草稿。
  • 設定模型選擇(首選 / 對照測試組)。
  • 啟用兩階段隔離(若 Agent 為條件式或必要兩階段)。
  • 設定使用者群組與算力點數初期配額。
4
第 4 階段:測試驗證(第 4-5 週)
  • 依驗測情境逐一執行測試,記錄命中率與誤判率。
  • 由業務窗口與資深同仁覆核 Agent 輸出,標註修改建議。
  • 智慧方案依驗測結果回頭調整 Prompt,進行第二輪測試。
  • 達成命中率 ≥ 80% 與誤判率 ≤ 10% 後,啟動上線準備。
5
第 5 階段:教育訓練(第 5-6 週)
  • 對使用者群組舉辦 1 場 1-2 小時的入門訓練,涵蓋 Agent 操作與限制。
  • 提供使用範例與常見問答清單,降低使用門檻。
  • 由業務窗口擔任『AI 教練』,協助同仁進行第一週使用支援。
  • 建立內部回饋管道(LINE 群組 / 表單),收集調整建議。
6
第 6 階段:正式上線(第 6-8 週)
  • 正式開放使用者群組存取 Agent。
  • 前 2 週啟動高頻監控(每日查使用量、異常觸發率、點數消耗)。
  • 智慧方案 AI 教練每週與業務窗口同步,進行 Prompt 迭代與知識庫補強。
  • 1 個月後召開回顧會議,評估是否擴大使用者群組或啟動新 Agent。
關鍵風險提醒
  • AI 不取代人工最終決策:Agent 輸出僅供參考,最終決策責任仍由相關人員承擔。
  • 知識庫資料時效性:報價、評估、案例等時效性資料若超過 3-6 個月未更新,將顯著影響命中率。
  • 新情境盲區:對於完全沒有歷史先例的新情境,Agent 能力有限,應由資深員工接手。
  • 規則語料完整性:建議持續補充規則對照表、案例庫,提升命中率。種子團隊每月例行回顧。

3. 資料清洗指南

本章節提供本 Agent 知識庫資料的清洗建議。資料清洗為命中率拉升的關鍵,POC 範例資料僅供結構參考,正式上線前須由客戶業務窗口提供經清洗的真實資料。

為什麼一定要清洗
  • SQL 知識庫欄位名稱含特殊符號(括號、斜線、空格)會造成解析失敗,Agent 直接報錯。
  • RAG 知識庫切片若跨章節混雜,語意檢索品質下降,命中率可能僅 30-40%。
  • 未清洗的敏感資訊(客戶名、報價、人事)有外洩風險,並影響跨部門共享決策。
  • 清洗前後命中率落差可達 30-50%,直接影響使用者採用意願與 ROI 顯化速度。

3.1 結構化資料的清洗清單

客訴明細表.csv 需清洗
  • 欄位確認:現有欄位為 客訴編號, 客戶代號, 發生日期, 產品線, 現象分類, 嚴重度, 處理狀態, 結案日期;上線前由業務窗口確認欄位名稱、單位、值域與正式資料一致。
  • 空值統一:以特定字串(例如『-』或留空)統一空值表達,避免 Agent 因空格、空字串、N/A 等不同形式而誤判。
  • 客戶/供應商代號標準化:同一實體須以同一代號表達,避免 SQL 過濾時漏列。
  • 正式資料替換:由各部門窗口協調 ERP 匯出最新版本,維持 utf-8-sig 編碼。

3.2 文件型知識庫的清洗清單

客訴歷史案例庫.txt 需重整
  • 章節結構確認:用 Markdown 標題分隔(## 主章節、### 子章節),確保 RAG 切片不會跨章節混淆。
  • 去敏感資訊:刪除具識別性的客戶名稱、案件編號,以代號或佔位符取代。
  • 專業用語一致性:中英日對照、簽名檔、認證版本等須與公司最新標準對齊。
  • 長度控制:單份文件建議 < 5,000 字;若超過請依章節拆檔。

3.3 補強型對照表建議

建議補強:客訴明細表客戶代號標準化(同一客戶不同案件採同代號)、現象分類欄位字典化

  • 智慧方案依驗測結果回頭調整 Prompt,進行第二輪測試。

  • 前 2 週啟動高頻監控(每日查使用量、異常觸發率、點數消耗)。

以下為 力盛科技 零組件規格查詢助理 v1.0 的完整系統提示詞,可直接複製至 EgentHub Agent 設定中。

SYSTEM PROMPT v1.0 
# 角色
您是力盛科技「零組件規格查詢助理」,協助採購暨供應鏈管理部同仁以自然語言快速查詢鈑金、機殼、連接器、散熱、精密零組件的料件規格、替代品建議與供應商歷史交易資料。

# 任務
當採購同仁以自然語言查詢料件時,您需要:
1. 解析查詢意圖(查詢規格 / 查詢供應商 / 查詢替代品 / 查詢歷史交易)。
2. 從「料件主檔」查詢規格、替代料件編號。
3. 從「供應商歷史交易表」查詢該料件的主要供應商、最近報價、最近交期表現。
4. 整合輸出料件規格表、供應商概況、建議行動。

# 背景知識
本 Agent 連結了:
- 「料件主檔」表格查詢知識庫:料號、品名、規格、單位、替代料號、最後維護日。
- 「供應商歷史交易表」表格查詢知識庫:料號、供應商代號、最近報價、最近交期表現(準時/延遲)、品質紀錄。

# 輸入格式
採購同仁以自然語言查詢,例如:『SUS304 1.0mm 鈑金最近報價?』、『M3 防拆螺絲替代料件?』、『XX 連接器近半年交期表現?』

# 執行流程
1. 解析查詢意圖與關鍵料件詞。
2. 由「料件主檔」查詢規格與替代料號。
3. 由「供應商歷史交易表」查詢供應商概況。
4. 整合為料件規格表 + 供應商概況 + 建議行動。
5. 若料件主檔中找不到對應料件,明確回覆『料件主檔尚未收錄此料件,建議透過 ERP 進一步查詢或與資深採購確認』,不得自行推論料件規格。

3. 資料清洗指南

本章節提供本 Agent 知識庫資料的清洗建議。資料清洗為命中率拉升的關鍵,POC 範例資料僅供結構參考,正式上線前須由客戶業務窗口提供經清洗的真實資料。

為什麼一定要清洗
  • SQL 知識庫欄位名稱含特殊符號(括號、斜線、空格)會造成解析失敗,Agent 直接報錯。
  • RAG 知識庫切片若跨章節混雜,語意檢索品質下降,命中率可能僅 30-40%。
  • 未清洗的敏感資訊(客戶名、報價、人事)有外洩風險,並影響跨部門共享決策。
  • 清洗前後命中率落差可達 30-50%,直接影響使用者採用意願與 ROI 顯化速度。

下列六階段檢核清單,建議由智慧方案顧問團隊與 力盛科技 業務窗口、IT 共同推動,預估期程 6-8 週。

1
第 1 階段:資料準備(第 1-2 週)
  • 由智慧方案顧問與客戶業務負責人確認 Agent 的使用情境與成功標準。
  • 盤點 Agent 所需的知識庫檔案,確認來源、版本、更新頻率。
  • 客戶端依清洗指南進行知識庫資料初步整理(去敏感資訊、欄位重命名、檔案版本去重)。
  • 智慧方案提供 Prompt 草稿與驗測情境,協助客戶業務窗口理解設計邏輯。
2
第 2 階段:知識庫建置(第 2-3 週)
  • 於 EgentHub 工作室建立對應的知識庫(表格查詢 / Agent 向量 / 全文查找)。
  • 上傳清洗後的 CSV / TXT 檔案,並確認欄位解析與切片結果。
  • 標註知識庫所屬層級(工作室層級 / Agent 層級),設定權限。
  • 進行知識庫初步檢索測試,確認語意檢索品質達標。
3
第 3 階段:Agent 設定(第 3 週)
  • 於 EgentHub 工作室建立 Agent,貼上 Prompt 草稿。
  • 設定模型選擇(首選 / 對照測試組)。
  • 啟用兩階段隔離(若 Agent 為條件式或必要兩階段)。
  • 設定使用者群組與算力點數初期配額。
4
第 4 階段:測試驗證(第 4-5 週)
  • 依驗測情境逐一執行測試,記錄命中率與誤判率。
  • 由業務窗口與資深同仁覆核 Agent 輸出,標註修改建議。
  • 智慧方案依驗測結果回頭調整 Prompt,進行第二輪測試。
  • 達成命中率 ≥ 80% 與誤判率 ≤ 10% 後,啟動上線準備。
5
第 5 階段:教育訓練(第 5-6 週)
  • 對使用者群組舉辦 1 場 1-2 小時的入門訓練,涵蓋 Agent 操作與限制。
  • 提供使用範例與常見問答清單,降低使用門檻。
  • 由業務窗口擔任『AI 教練』,協助同仁進行第一週使用支援。
  • 建立內部回饋管道(LINE 群組 / 表單),收集調整建議。
6
第 6 階段:正式上線(第 6-8 週)
  • 正式開放使用者群組存取 Agent。
  • 前 2 週啟動高頻監控(每日查使用量、異常觸發率、點數消耗)。
  • 智慧方案 AI 教練每週與業務窗口同步,進行 Prompt 迭代與知識庫補強。
  • 1 個月後召開回顧會議,評估是否擴大使用者群組或啟動新 Agent。
關鍵風險提醒
  • AI 不取代人工最終決策:Agent 輸出僅供參考,最終決策責任仍由相關人員承擔。
  • 知識庫資料時效性:報價、評估、案例等時效性資料若超過 3-6 個月未更新,將顯著影響命中率。
  • 新情境盲區:對於完全沒有歷史先例的新情境,Agent 能力有限,應由資深員工接手。
  • 規則語料完整性:建議持續補充規則對照表、案例庫,提升命中率。種子團隊每月例行回顧。
